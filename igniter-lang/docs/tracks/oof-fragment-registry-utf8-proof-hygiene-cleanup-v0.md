## Goal

Remove locale-dependent UTF-8 handling from OOF/Fragment Registry proof scripts
and summaries so the R103/R107/R111 proof chain passes without `RUBYOPT` or an
external UTF-8 locale.

---

## Problem

The R103 proof (`oof_fragment_registry_implementation_boundary_proof.rb`) wrote
UTF-8 checkmark characters (✓, U+2713, encoded as `\xE2\x9C\x93`) into check
detail strings such as `"valid=true, inactive_rows=0 ✓"`. These strings were
serialized into the R103 summary JSON. The R107 proof then read that summary
with a plain `File.read` call (no encoding specified). In environments where the
default external encoding is `US-ASCII` (i.e. `LANG=""` / `LC_ALL=""` / `C`
locale), this produced:

```
Encoding::InvalidByteSequenceError: "\xE2" on US-ASCII
```

The R111 proof also used plain `File.read` for fixture and library files, and
all three proofs had no defensive encoding on file reads.

## Changes Made

### 1. R103 proof — replace ✓ glyphs with ASCII `[ok]`

All 27 check detail strings that contained ` ✓` were changed to ` [ok]`.
This fixes the problem at the source: the regenerated R103 summary JSON is
now pure ASCII (verified: 0 non-ASCII bytes).

Also added `encoding: "utf-8"` to all `File.read` calls in R103 (load_fixture,
CS1 lib check, CS5 R92 JSON reads) as belt-and-suspenders.

### 2. R107 proof — explicit UTF-8 encoding on File.read

All `File.read` calls in the R107 proof now specify `encoding: "utf-8"`:
- `File.read(FIXTURE_PATH, encoding: "utf-8")`
- `File.read(R103_SUMMARY_PATH, encoding: "utf-8")`
- `File.read(ROOT / "lib/igniter_lang.rb", encoding: "utf-8")`
- `File.read(path, encoding: "utf-8")` (compiler file checks)

### 3. R111 proof — explicit UTF-8 encoding on File.read

All `File.read` calls in the R111 proof now specify `encoding: "utf-8"`:
- `File.read(FIXTURE_PATH, encoding: "utf-8")`
- `File.read(main_lib, encoding: "utf-8")`
- `File.read(path, encoding: "utf-8")` (compiler file checks)
