## Decision

Recommendation:

```text
Do not create `lib/igniter_lang/oof_fragment_registry_data.rb`.
Reject static proof-derived sample constants as internal library data for now.
Keep proof data fixture-only.
Future non-proof registry data should be caller/loader-supplied behind a
separate source-authority and loading-boundary decision.
```

Short form:

```text
static internal data: reject for current route
fixture-only proof data: keep
loader/caller-supplied data: future design route, not opened here
```

Reason:

- R103 proved the validator works for supplied registry hashes.
- Static internal constants would look like default registry content even if
  labeled non-canon.
- No accepted source-authority exists for a complete live registry catalog.
- `lib/igniter_lang.rb` intentionally does not require the validator; adding a
  data file now would create pressure to require or discover it.
- Proof fixtures already provide the forward R98/R101 shape without implying
  compiler authority.

Verdict:

```text
Use fixture-only data for proofs.
Use caller/loader-supplied data for future live validation design.
Do not add static internal library data now.
```

## Why Static Proof-Derived Constants Are Rejected

Rejected shape:

```ruby
module IgniterLang
  class OOFFragmentRegistryData
    SAMPLE_REGISTRY = { ... }.freeze
  end
end
```

Even with `SAMPLE_` or `NON_CANON_` naming, a Ruby library data file under
`lib/igniter_lang/` would create four bad pressures:

- callers may treat it as a default registry;
- later compiler code may reach for it instead of requiring a source-authority
  decision;
- row drift from proof fixtures becomes harder to see;
- a data file invites `require "igniter_lang/oof_fragment_registry_data"` from
  public or compiler paths.

Therefore:

```text
No static proof-derived constants in `lib/igniter_lang/` for now.
```

## Preferred Proof-Data Posture

Keep proof data under:

```text
experiments/oof_fragment_registry_implementation_boundary_proof/fixtures/
```

Allowed proof fixture posture:

- JSON fixture files remain proof-owned;
- fixtures carry `source_authority.canon_status: "non_canon"`;
- fixtures keep `migration_policy: "non_migration_historical_artifact"` for R92;
- fixture rows may cite source files and tracks as evidence, not authority;
- proof output may summarize PASS/FAIL and closed-surface assertions;
- no fixture is loaded by compiler code or public API.

If generated fixture data is later desired, keep generation inside the
experiment path, for example:

```text
experiments/oof_fragment_registry_implementation_boundary_proof/generate_fixture_data.rb
```

That generator would remain proof-only and must not write under `lib/`.

## Future Loader/Caller-Supplied Route

The validator already accepts supplied hashes:

```ruby
IgniterLang::OOFFragmentRegistry.new.validate(registry_hash, installed_boundaries: ...)
```

Future non-proof usage should preserve this direction:

```text
caller/loader supplies registry data
validator validates supplied data
compiler behavior stays unchanged unless separately authorized
```

This route is not opened here because loader/report, CompatibilityReport,
public API/CLI, and compiler integration remain closed.

If opened later, the design must answer:

- where registry data is loaded from;
- whether the source is manifest, profile contract, pack descriptor, or another
  authority-bearing artifact;
- how row source authority is represented;
- whether invalid data blocks loading, records internal diagnostics, or remains
  report-only;
- whether loader/report or CompatibilityReport may surface validation evidence.

## If Static Data Is Reopened Later

This track recommends against static internal data now. If Architect reopens it
later, minimum requirements before implementation authorization:

| Requirement | Minimum bar |
| --- | --- |
| File name | `lib/igniter_lang/oof_fragment_registry_data.rb` only if explicitly authorized. |
| Naming | Avoid `DEFAULT`, `CANONICAL`, `CURRENT`, or `REGISTRY`; prefer `NON_CANON_SAMPLE_REGISTRY` if unavoidable. |
| Labeling | Every exported object must include `canon_status: "non_canon_sample"` and `source_authority.authority_kind: "proof_sample"`. |
| Require policy | Must not be required from `lib/igniter_lang.rb` or compiler pass files. |
| Compiler integration | Must stay closed; validator may be invoked only by proof-local harness unless another gate opens integration. |
| Public surface | No public API/CLI, reports, `CompilerResult`, `.igapp`, loader/report, or CompatibilityReport fields. |
| Source rows | Must be a sample subset, not a complete registry claim. |
| R92 policy | Must retain R92 non-migration and forward support-marker separation. |
| Drift policy | Must define how fixture data and static data stay in sync or why they intentionally differ. |

Even if reopened, the safer design is:

```text
proof fixture -> validator
not
static lib data -> compiler default
```
