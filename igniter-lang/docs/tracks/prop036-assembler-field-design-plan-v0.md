## Design Boundary

[D] Future assembler support should be split into at least two implementation
surfaces:

1. `assembler-compiler-profile-id-field-v0`
2. `artifact-hash-profile-id-golden-migration-v0`

[D] The field-emission card may place `compiler_profile_id` in the future
manifest construction path, but the golden/hash migration card owns intentional
hash churn and fixture updates.

[D] No current `.igapp` artifact should be silently regenerated by this design
card.

## Future Assembler Field Plan

### Field Placement

`compiler_profile_id` belongs at top-level in `.igapp/manifest.json`:

```json
{
  "kind": "igapp_manifest",
  "format_version": "0.1.0",
  "compiler_profile_id": "compiler_profile_unified/sha256:2944e573270aa56fca51cea3",
  "artifact_hash": "sha256:..."
}
```

[D] The manifest field is an artifact identity claim about compiler
understanding. It is not runtime approval, TBackend authorization, Gate 3
authority, or compiler dispatch selection.

### Artifact Hash Ordering

Required future ordering:

```text
finalize CompilerProfile
derive compiler_profile_id
assemble manifest material with compiler_profile_id present
compute artifact_hash over profiled artifact material
attach artifact_hash to manifest/report surfaces according to existing policy
```

[D] `compiler_profile_id` must be present in the material that contributes to
`artifact_hash`.

[D] Post-hash or post-sign annotation is invalid:

```text
assemble -> hash/sign -> add compiler_profile_id
```

### Signature Payload Dependency

If a future signing card is authorized, its payload must cover both:

```text
artifact_hash
compiler_profile_id
```

[D] Signing remains out of scope here. The S3-R37-C5-P proof used synthetic
signatures only to demonstrate payload dependency.

### Rollout Policy

Initial rollout remains:

```text
legacy_optional
```

[D] Missing `compiler_profile_id` remains inspectable as `absent_legacy` until a
later migration card and Architect decision authorize `profile_required`.

[D] `present_verified` remains separate from runtime readiness.

### Migration And Golden Boundaries

[D] Existing `.igapp` fixtures and artifact hash goldens are not changed by this
track.

[D] Any future golden migration must be explicit, mechanical, and scoped to the
fixtures named by that card. It must show before/after hash effects caused by
adding `compiler_profile_id`.

[D] Loader/report status implementation remains separate from assembler field
emission. Receipt links remain separate again.

## Proposed Future Slice Order

[R] Recommended implementation sequencing, once explicitly authorized:

1. `assembler-compiler-profile-id-field-v0`
   - assembler-only;
   - introduces field construction behind the accepted PROP-036 shape;
   - may use a proof-local or test-only manifest fixture first;
   - does not migrate all goldens unless explicitly assigned.
2. `artifact-hash-profile-id-golden-migration-v0`
   - golden-migration-only;
   - regenerates named `.igapp` fixture hashes intentionally;
   - proves hash deltas are caused by `compiler_profile_id`.
3. `loader-compiler-profile-status-report-v0`
   - loader/report-only;
   - implements the five status values under `legacy_optional`;
   - preserves `present_verified != runtime ready`.
4. `compilation-receipt-manifest-link-v0`
   - receipt-link-only;
   - may require `receipt.compiler_profile_id == manifest.compiler_profile_id`
     after manifest ordering is stable.

## Blockers Before Any Implementation Card

[R] Before any PROP-036 implementation card, the card must:

1. cite PROP-036, the S3-R35-C3-A acceptance decision, the S3-R36-C5-P loader
   proof, and the S3-R37-C5-P hash-ordering proof;
2. receive separate Architect/supervisor implementation authorization;
3. name exactly one implementation surface;
4. state its surface type: assembler-only, loader-only, report-only,
   golden-migration-only, receipt-link-only, or signing-only;
5. preserve `compiler_profile.present_verified != runtime ready`;
6. preserve `legacy_optional` unless a later Architect decision changes rollout;
7. avoid `profile_required` rollout without migration evidence;
8. avoid real `.igapp` or `.ilk` mutation unless explicitly authorized;
9. for assembler work, prove `compiler_profile_id` enters artifact hash
   material before any golden update is accepted;
10. for golden migration, list exact fixtures and expected hash churn;
11. for loader/report work, test `absent_legacy`, `present_verified`,
    `mismatch`, `malformed`, and `missing_required`;
12. keep compiler dispatch migration out of scope unless separately authorized;
13. keep RuntimeMachine binding and runtime execution authority out of scope
    unless separately authorized;
14. keep receipt links and production signing out of scope unless the card names
    that exact surface.
