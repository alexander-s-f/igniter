## Snapshot

Ruflo presents itself as a full multi-agent orchestration platform for Claude
Code and adjacent agent workflows.

Core public promise:

```text
init once, then keep working while agents coordinate, remember, route, and learn
in the background.
```

Visible capability clusters:

- CLI and MCP server;
- plugin marketplace;
- many specialized agents;
- swarm coordination;
- persistent memory/RAG;
- background workers;
- web UI;
- goal planner;
- federation/trust boundary;
- security and compliance features.

## Why It Appears To Be Growing

Interpretation:

- It has a strong first-run promise: install/init and continue using Claude Code.
- It bundles many things users currently want: agents, memory, tools, swarms,
  UI, federation, security.
- It provides both lite and full paths, reducing commitment anxiety.
- It has visible product surfaces, not only docs.
- It uses familiar integration surfaces: npm, CLI, MCP, plugins.
- It frames agents as collaborators with memory and coordination, not isolated
  one-shot prompts.

The growth signal is likely not only technical. It is narrative and packaging:

```text
agents do not just run; they collaborate.
```

## Our Coordinate Match

Ruflo and Igniter orchestration are adjacent but not equivalent.

Ruflo coordinate:

```text
runtime platform
agent tooling
memory infrastructure
plugin ecosystem
product onboarding
federated execution
```

Igniter orchestration coordinate:

```text
authority discipline
bounded cards
gate decisions
pressure review
proof matrices
closed-surface preservation
portfolio supervision
cross-lane letters/reports
```

Short comparison:

```text
Ruflo asks: how do many agents run, coordinate, remember, and use tools?
Igniter asks: how do many agents preserve meaning, authority, proof, and human control?
```

This means Ruflo is more advanced as a packaged runtime/product surface, while
our current strength is governance, semantic containment, and proof-backed
decision flow.

## Useful Pressure

Questions Ruflo creates for us:

1. Is our first-run experience too dependent on the main Igniter culture?
2. Do we need clearer lite/full modes for orchestration adoption?
3. Should Portfolio letters become a first-class cross-supervisor protocol?
4. Are Current Map, Operating Guide, and Verification Log distinct enough?
5. Can we create a stronger goal-to-card-to-report UX without copying runtime
   scope?
6. Are we over-indexing on authority before new users feel flow?

The strongest pressure:

```text
Make the experience feel alive before asking users to understand the whole
governance system.
```

---

## What To Adopt

Adopt as patterns, not implementation:

- lite/full onboarding split;
- stronger first-run promise;
- visible mode distinction between demo, local workspace, and portfolio;
- Current Map / Operating Guide / Verification Log triad;
- cross-supervisor letters as a lightweight federation analogue;
- goal-to-plan language for dispatch, when appropriate;
- explicit "what you get" section in future onboarding docs.

Possible adapted wording:

```text
After init, you keep leading the work.
The orchestra keeps roles, context, decisions, and boundaries coherent.
```

---

## What To Avoid

Avoid:

- copying runtime/MCP/swarm scope into Igniter without gates;
- treating large agent count as maturity;
- importing "self-learning" claims without our own proof/audit model;
- broad plugin-marketplace thinking before we have stable portable method;
- assuming Agent Orchestra DNA becomes portable by adding more docs;
- blurring product/runtime authority with governance authority.

The risk for us would be imitation of surface area instead of sharpening our
actual advantage.

## Agent Orchestra DNA Relevance

This scan reinforces the current conclusion:

```text
Agent Orchestra DNA is not yet a proven portable full-system package.
```

Ruflo succeeds as a packaged runtime/product. Our DNA repo attempted to package
a governance culture. The pilots show that governance culture did not transfer
as a whole into Spark CRM and other projects. Agents drifted especially around
authority when two or more agents were active.

Current interpretation:

```text
The portable unit is not the whole orchestra.
The portable units are smaller patterns:
cards, current map, Fast Lane, letters, bounded reports, authority/pressure split.
```

Spark CRM current lesson:

```text
Fast Lane works better than full Agent Orchestra DNA for application delivery.
```
