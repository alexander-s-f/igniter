## Snapshot

elizaOS presents itself as a practical agent runtime/framework rather than a
governance method.

Visible capability clusters:

- TypeScript runtime and project scaffolding;
- CLI path such as project creation and dev server;
- agent projects and plugin templates;
- actions, providers, services, routes, and events;
- runtime dashboard / GUI path;
- many examples;
- benchmark suites;
- plugin ecosystem.

The first-run shape is product-oriented:

```text
create project -> run dev -> see an agent alive -> learn plugins/runtime
```

---

## What It Is

Best category:

```text
agent runtime / application framework
```

Secondary categories:

```text
plugin framework
agent project scaffold
autonomous agent application platform
benchmark/example ecosystem
```

elizaOS is not primarily a proof/gate/authority system. It focuses on making
agent applications concrete, runnable, extensible, and deployable.

---

## Why It Is Growing

Interpretation:

- It gives users an immediate runnable path.
- It has a clear project/plugin mental model.
- It uses familiar developer scaffolding patterns.
- It has examples and benchmarks that make the ecosystem feel alive.
- It turns "agent framework" into something a developer can install, run, and
  extend without first accepting a large governance theory.
- It has language that matches the current ecosystem appetite for autonomous
  agents, plugins, tools, and runtime surfaces.

Strongest growth lesson:

```text
users feel the system before they understand the system
```

## Our Coordinate Match

elizaOS and Igniter should not be viewed as direct substitutes.

Coordinate mapping:

```text
elizaOS asks: how do I build and run an agent with tools, plugins, memory, and UI?
Igniter asks: what may this system believe, decide, execute, refuse, explain, and audit?
```

This makes elizaOS useful pressure for Igniter's future runtime/app ecosystem,
not for immediate compiler core mutation.

The useful overlap is vocabulary and packaging:

```text
action/provider/service/plugin
```

These map naturally to future Igniter pressure vocabulary:

```text
Observation Provider
Effect Adapter
Evidence Source
Capability Tool
```

Igniter's responsibility would be to govern such capabilities, not to become a
free-running autonomous agent runtime.

## Useful Pressure

Questions elizaOS creates for Igniter:

1. Can Igniter eventually offer a small project template instead of only proof
   artifacts?
2. What is Igniter's equivalent of "create project -> run dev -> see it alive"?
3. Can examples and benchmarks become a public trust surface without weakening
   gate discipline?
4. Should capability adapters borrow action/provider/service vocabulary, or do
   we need stricter names to preserve authority?
5. How do we show a living system while keeping implementation surfaces closed
   until they are ready?
6. Can a future Igniter demo ladder be:

```text
contract specimen -> compile -> proof/report -> runtime decision ->
accepted/refused action -> receipt
```

Strongest pressure:

```text
Igniter needs a runnable feeling, not only a correct architecture.
```

---

## What To Adopt

Adopt as patterns:

- project/template mental model;
- demo ladder from tiny runnable path to advanced system;
- examples and benchmarks as trust surface;
- plugin/action/provider/service distinction as external vocabulary pressure;
- dev loop that shows liveness early;
- standalone-runtime path separate from full platform path.

Possible future Igniter packaging:

```text
demo -> project -> capability adapter -> proof/benchmark -> deploy boundary
```

---

## What To Avoid

Avoid:

- importing autonomous-agent rhetoric into Igniter core;
- treating "agent can act" as "agent is authorized to act";
- opening runtime/plugin execution before language/compiler/profile foundations
  are stable;
- plugin-marketplace thinking before capability/authority boundaries exist;
- weakening proof/gate discipline to chase product surface;
- conflating app framework UX with language semantics.
