# S3-R228 - IVM AOT Bytecode File Loading Authorization

Status: Planned
Date: 2026-06-01

## IDD Classification

Mode:

```text
standard / playground-only AOT bytecode file-loading authorization before proof
```

Why:

- R227 accepted playground-only native acceleration research evidence;
- C/Fiddle native runner proves parity with Ruby IVM for Add, GT, lazy branch,
  unsupported selected fail-close, unsupported non-selected silence, and
  malformed input fail-close;
- R227 also identified Fiddle transition and per-run serialization overhead as
  the next runtime research bottleneck;
- the next smallest executable-runtime question is whether a native runner can
  load a proof-local bytecode file directly without creating mainline runtime
  authority.

Smallest useful artifact:

```text
authorization review -> playground AOT file-loading proof -> pressure -> acceptance -> status
```

## Round Intent

Decide whether a bounded playground-only AOT bytecode file-loading proof may
begin, and if so prove direct file-backed native execution with Ruby IVM parity.

This round should answer:

```text
Can a proof-local .igbin or equivalent bytecode file be emitted under playground out/?
Can the native runner load that file directly and execute it?
Does file-backed native execution preserve Add, GT, branch/lazy, and unsupported behavior?
Can malformed file handling fail closed?
Can timing be captured as local research only, not public performance claims?
Does file-backed execution remain delegated experimental evidence only?
```

This round does not implement Reference Runtime, public runtime support,
RuntimeSmoke productization, `igc run`, package exposure, reusable helper
extraction, Runtime Specification implementation, Spark integration, or release
work.
