# S3-R158 - Fractal Supervisor Seed Round

Status: open
Date: 2026-05-24

Dispatch: R158 = [C1-P1, C2-P1, C3-P1, C4-P1] -> C5-A -> C6-S

Pattern:
- Run `C1-P1`, `C2-P1`, `C3-P1`, and `C4-P1` in parallel.
- Each card is a supervisor seed card, not a fully expanded executor round.
- Each supervisor should create its own internal mini-round from the seed,
  using local cards/tracks/reports in its own operating surface.
- Each supervisor should also place its compact micro-round packet under:
  `igniter-lang/docs/cards/S3/R158/`.
- Each supervisor returns one compact packet to Portfolio:
  summary, evidence links, risks, next requested boundary.
- Run `C5-A` only after all available supervisor packets land.
- Run `C6-S` after the Portfolio decision.
- This is a fractal dispatch experiment: top-level cards define direction,
  authority boundaries, and return shape; child supervisors own their local
  decomposition.

Global constraints:
- Do not authorize production deployment, public release, public demo claims,
  or cross-project integration unless explicitly and separately stated.
- Keep Spark code/data access, Igniter-Lang implementation, Ruby release
  execution, and Org process changes inside their local supervisor authority.
- Prefer small proof/review/design packets over broad implementation.
- Compiler release movement remains the main priority.

Fractal packet convention:
- `igniter-lang/docs/cards/S3/S3-R158.md` remains the top-level dispatch file.
- `igniter-lang/docs/cards/S3/R158/` is the local packet directory for this
  round's supervisor micro-rounds.
- Supervisors may still write detailed tracks/reports in their native operating
  surfaces, but the packet directory must contain the compact return artifact
  that Portfolio can read first.
- Recommended packet names:
  - `lang-supervisor-packet.md`
  - `ruby-framework-supervisor-packet.md`
  - `spark-app-supervisor-packet.md`
  - `org-architect-supervisor-packet.md`
