# S3-R196 - if_expr Runtime Evaluator Design

Status: planned
Date: 2026-05-27

Dispatch: R196 = C1-D -> C2-X -> C3-A -> C4-S

Pattern:
- C1-D designs runtime/evaluator semantics for accepted expression-level
  `if_expr` v0.
- C2-X pressure-checks lazy evaluation, dependency/cache implications,
  diagnostics, and closed-surface boundaries.
- C3-A accepts, conditionally accepts, holds, or redirects the design and
  decides whether any later implementation-authorization review may open.
- C4-S curates the outcome and exact next Main Line boundary.

- Move from accepted compiler-only `if_expr` support to runtime/evaluator design.
- Decide the intended lazy branch execution model before any runtime code opens.
- Preserve release/public/Spark/API/CLI and implementation boundaries.

  - runtime/evaluator support status and placement;
  - lazy branch execution semantics:
    - evaluate condition first;
    - evaluate only selected branch;
    - do not evaluate non-selected branch side effects/failures;
  - condition evaluation order and Bool requirement expectations at runtime;
  - branch selection and value propagation;
  - behavior for malformed/unsupported SemanticIR `if_expr` nodes;
  - runtime diagnostics / error vocabulary, if any;
  - failure propagation policy:
    - condition failure;
    - selected branch failure;
    - non-selected branch failure must not fire;
  - dependency/cache invalidation implications:
    - static deps already union at compiler level;
    - runtime may dynamically touch only selected branch;
    - cache invalidation must remain conservative unless later designed;
  - interaction with existing runtime/evaluator architecture;
  - proof matrix required before implementation;
  - implementation write-scope candidates for a future authorization review;
  - excluded surfaces and non-claims.
- Explicitly answer:
  - whether v0 runtime semantics should be lazy or eager;
  - whether non-selected branch evaluation is forbidden;
  - whether static dependency union remains accepted for compiler/runtime
    boundary;
  - whether dynamic selected-branch dependency tracking is required now,
    deferred, or out of scope;
  - whether runtime diagnostics need new `OOF-RT-*` vocabulary or can use
    existing runtime error surfaces;
  - whether malformed SemanticIR `if_expr` should fail closed or remain
    unreachable under compiled input assumptions;
  - whether implementation authorization review may open next or must wait;
  - whether release lane remains paused;
  - whether public demo/stable/production/all-grammar claims remain closed;
  - whether Spark/API/CLI remain closed.

- Do not implement.

  - accept design and authorize later implementation-authorization review;
  - accept design but keep implementation held;
  - conditional accept with blockers;
  - hold pending more architecture survey;
  - redirect.
