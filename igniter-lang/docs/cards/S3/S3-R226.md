# S3-R226 - IVM Adapter Branch Coverage Hardening

Status: Planned
Date: 2026-06-01

## IDD Classification

Mode:

```text
standard / playground-only adapter hardening authorization before proof
```

Why:

- R225 accepted adapter-fit evidence:
  `compiler-emitted semantic_ir_program.json -> IVM AST -> 4-opcode bytecode ->
  IVM execution -> 42`;
- R225 lazy branch evidence is accepted as supplemental playground evidence,
  not yet a fresh compiler-emitted `semantic_ir_program.json` branch proof;
- `stdlib.integer.gt` and `field_access` are still explicit adapter gaps;
- before FFI/C acceleration or reusable helper extraction, the adapter needs
  source-backed branch/comparison coverage and clearer digest fields.

Smallest useful artifact:

```text
authorization review -> playground hardening proof -> pressure -> acceptance -> status
```

## Round Intent

Decide whether a bounded playground-only adapter hardening proof may begin for
compiler-emitted branch/comparison coverage.

This round should answer:

```text
Can a compiler-emitted semantic_ir_program.json branch fixture feed IVM?
Can IVM adapter prove selected branch execution and non-selected branch silence
from source-backed compiler output?
What is the exact stdlib.integer.gt stance: mapped, held, or explicit gap?
Can digest fields distinguish semantic_ir_program digest from .igapp bundle
digest?
Does IVM remain delegated experimental evidence only?
```

This round does not implement runtime productization, FFI/C acceleration,
helper extraction, package exposure, `igc run`, RuntimeSmoke changes,
Reference Runtime, public runtime support, Spark integration, or release work.
