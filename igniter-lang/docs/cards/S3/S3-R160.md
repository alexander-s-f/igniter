# S3-R160 - Compiler Release Acceptance Harness Design

Status: open
Date: 2026-05-24

Dispatch: R160 = C1-D -> C2-X -> C3-A -> C4-S

Pattern:
- Use the classic unified round-file mode.
- Run `C1-D` first.
- Run `C2-X` after `C1-D` lands.
- Run `C3-A` after `C2-X` lands.
- Run `C4-S` after the Portfolio decision.
- This round is design-only unless `C3-A` explicitly narrows a later route.

Round intent:
- Define the compiler release acceptance harness before any RC evidence
  gathering or harness implementation.
- Convert the accepted release-readiness map into a concrete acceptance design:
  inputs, normalization, corpus requirements, command matrix, result packet,
  non-claims, and closed-surface scans.
- Keep analyzer/tracer/visualizer as design vocabulary only.

Global constraints:
- Do not implement code.
- Do not run release execution.
- Do not authorize public release/demo claims.
- Do not authorize analyzer/tracer/visualizer implementation.
- Do not authorize public API/CLI widening, loader/report,
  CompatibilityReport, RuntimeMachine/Gate 3 widening, Spark integration,
  Ruby release/doc sync, production runtime, Ledger/TBackend production
  binding, BiHistory, stream/OLAP, cache, signing, or deployment.
