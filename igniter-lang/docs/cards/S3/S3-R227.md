# S3-R227 - IVM Bytecode Acceleration Authorization

Status: Planned
Date: 2026-06-01

## IDD Classification

Mode:

```text
standard / playground-only native acceleration authorization before proof
```

Why:

- R226 accepted source-backed branch/comparison adapter-hardening evidence;
- IVM now proves Add, `if_expr`, `stdlib.integer.gt -> OP_GT`, selected branch
  execution, non-selected branch silence, and unsupported-node fail-close;
- the next runtime-productization pressure is whether IVM bytecode can escape
  the Ruby execution loop without creating mainline runtime authority;
- FFI/C/Rust acceleration must remain playground-only, non-canonical, and
  evidence-only until a later gate changes authority.

Smallest useful artifact:

```text
authorization review -> playground native proof -> pressure -> acceptance -> status
```

## Round Intent

Decide whether a bounded playground-only IVM bytecode acceleration research
proof may begin, and if so prove a native execution path against Ruby IVM
parity without widening public/runtime/package surfaces.

This round should answer:

```text
Can a native runner execute a narrow IVM bytecode subset with parity to Ruby IVM?
Can branch/lazy semantics survive native execution?
Can OP_GT and Add parity be proven?
Can unsupported selected/non-selected behavior remain honest?
Can benchmark evidence be captured without becoming performance claims?
Does the native path remain delegated experimental evidence only?
```

This round does not implement Reference Runtime, public runtime support,
RuntimeSmoke productization, `igc run`, package exposure, reusable helper
extraction, Runtime Specification implementation, Spark integration, or release
work.
