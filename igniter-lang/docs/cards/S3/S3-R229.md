# S3-R229 - Experimental Runtime Implementations and Portability Boundary

Status: Planned
Date: 2026-06-01

## IDD Classification

Mode:

```text
standard / design-only runtime implementation arena and portability boundary
```

Why:

- R223-R228 accepted delegated experimental runtime evidence without opening
  public runtime or Reference Runtime authority;
- R228 exposed the file-per-execution I/O bottleneck and pointed toward
  load-once / execute-many architecture;
- `playgrounds/igniter-runtime` now contains additional resident supervisor,
  C temporal backend, and ESP32/mesh research;
- `playgrounds/igniter-tbackend` now contains a Rust TBackend playground;
- the implementation status model now frames playgrounds as an implementation
  arena, not as a replacement for the official line.

Smallest useful artifact:

```text
design -> facts packet -> pressure -> decision -> status
```

## Round Intent

Design the boundary for experimental runtime implementations and future
portability before authorizing any implementation.

This round should answer:

```text
How do we name the implementation hierarchy in day-to-day routing?
What counts as Igniter Specification vs Official Reference Implementation vs
Delegated Experimental Runtime vs future Alternative Certified Implementation?
How should experimental executable use name the runtime it uses?
How should resident supervisor, C temporal backend, and Rust TBackend be
intaken without becoming authority?
What minimal artifact passport fields will later be needed for portability?
Can an `igc run` route be discussed next, and if so only as design or
authorization review?
```

This round does not implement runtime, CLI, package, report, API, RuntimeSmoke,
Reference Runtime, public support, stable API, Spark integration, release work,
or alternative implementation certification.

## Binding Context

Accepted status model:

```text
Igniter Specification
  -> Official Reference Implementation
  -> Delegated Experimental Runtimes
  -> Alternative Certified Implementations later
```

Current implementation arena:

```text
R225 adapter-fit evidence accepted
R226 branch/comparison hardening evidence accepted
R227 native FFI acceleration evidence accepted
R228 AOT .igbin file-loading evidence accepted
resident supervisor: sandbox candidate / needs intake
C temporal backend: sandbox candidate / needs intake
ESP32/mesh research: speculative sandbox research / needs separate framing
Rust TBackend: delegated storage/backend candidate / needs intake
```

Allowed framing:

```text
implementation arena
delegated experimental runtime
runtime candidate
storage/backend candidate
proof-local
non-canonical
pre-v1
no stable API guarantee
future portability
future certification
design-only
```
