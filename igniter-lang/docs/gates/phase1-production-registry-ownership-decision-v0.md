## Decision

Decision: **approve the Phase 1 production registry source-of-truth model as
gate document store + generated content-addressed registry index**.

This decision answers the open registry ownership questions from
`production-registry-ownership-options-v0` and closes the ownership ambiguity
for the design-only `phase1-production-durable-audit-v0` track.

This decision does **not** implement registry storage, authorize production
signing/key-management execution, bind to Ledger, bind to RuntimeMachine, or
open Phase 2.

Safe status phrase:

```text
Phase 1 production registry ownership is decided for design purposes:
the gate document store is the source of truth, and generated
content-addressed registry indexes are query artifacts. Package/runtime code may
consume the index as a read-only cache/validator only. Registry implementation,
production signing execution, RuntimeMachine binding, Ledger, and Phase 2 remain
closed until later Architect decisions.
```

## Answers To C3-P Q1-Q6

### Q1. Source Of Truth

Decision: the **gate document store** is the Phase 1 production registry source
of truth.

The source of truth is the set of signed gate documents, decision addenda,
revocation records, and supersession records under `igniter-lang/docs/gates/`,
anchored by content-addressed references.

Generated registry indexes are query artifacts derived from that source. They
are not independent authority sources.

### Q2. Freshness SLA

Decision: Phase 1 freshness is **release-time by default, startup-time when a
deployed artifact includes an updated generated registry index**.

Allowed Phase 1 freshness modes:

- `release_time`: registry index generated and bundled with a release artifact;
- `startup_time`: runtime/process reads the bundled/generated index at startup.

Not approved by this decision:

- per-invocation online lookup;
- external live revocation service;
- package-local authority mutation;
- runtime self-refresh from arbitrary branches or local checkouts.

If lower revocation latency becomes required, an external authority service may
be proposed later, but must serve full receipts and content-addressed decision
refs.

### Q3. Index Generation Owner

Decision: the registry index should be **generated by CI/release tooling from
signed gate docs**, with Architect/Meta Expert review over the signed docs and
index-generation rules.

Manual index curation is not the authority path. Humans review and sign gate
documents; tooling derives the index.

The index generation process must fail closed when:

- a referenced gate document is unsigned or has an invalid status;
- `content_sha256` does not match the document bytes;
- an authority transition receipt is missing or malformed;
- a direct `active -> superseded` transition appears without an approved
  paired transition model;
- duplicate active entries exist for the same `authority_ref`.

### Q4. Immutable Anchor

Decision: production should prefer **both release artifact digest and git commit
SHA** when available.

Minimum acceptable anchor for Phase 1 production design:

```text
document_path
content_sha256
git_commit or release_artifact_digest
status
signed_on
authority_ref
```

Preferred anchor:

```text
document_path
content_sha256
git_commit
release_artifact_digest
status
signed_on
authority_ref
```

`workspace-current` is not acceptable for production registry entries.

### Q5. External Service Receipt Exposure

Decision: any future external authority service must expose **full transition
receipts and content-addressed decision refs** sufficient for offline audit.

An external service may serve the registry index later, but it must not become a
black box. At minimum it must expose:

- registry storage identity;
- queried `authority_ref`;
- status as of requested time;
- issuance/revocation/supersession receipt chain;
- content-addressed decision refs;
- immutable anchors;
- service response timestamp;
- refusal/error reason when lookup fails.

### Q6. Package Authority Prohibition

Decision: package/runtime-owned registry stores are **prohibited from being the
authority source** for Phase 1.

Package/runtime code may only consume generated registry indexes as:

- read-only validator;
- read-only cache;
- schema checker;
- local query accelerator.

Package/runtime code must not:

- mutate authority state;
- self-issue `gate3_authorized: true`;
- treat local package data as canonical authority when it conflicts with the
  gate document source of truth;
- own production signing keys by default;
- silently refresh from arbitrary branches, worktrees, or unverified paths.

## Coupling To Durable Audit Design

Decision: `phase1-production-durable-audit-v0` may proceed **coupled to this
registry ownership decision** for design purposes.

The durable audit design may assume:

- gate docs are the source of truth;
- generated content-addressed registry index is the query artifact;
- package/runtime consumers are read-only cache/validator only;
- registry ownership implementation remains unbuilt;
- registry signing/key-management execution remains unapproved.

The durable audit design must not require production registry implementation to
exist before it can specify audit record schema, signing recommendation,
restart rebuild, format-version rules, audit traversal, storage identity, audit
reader role, and compliance language.

---

## Non-Authorization

This decision does not authorize:

- registry storage implementation;
- generated index implementation;
- production signing execution;
- production key management;
- external authority service;
- package-owned authority;
- RuntimeMachine binding;
- Ledger binding;
- Phase 2 Ledger adapter;
- BiHistory / transaction-time reads;
- stream / OLAP production executors;
- production cache;
- writes, replay, compact, subscribe;
- broadening `gate3_authorized: true`.

The signed Gate 3 Phase 1 live-read addendum remains unchanged.

---

## Blockers Before Registry Implementation Authorization

Registry implementation authorization requires a later Architect decision and
at least:

1. generated registry index schema;
2. CI/release generation proof plan;
3. content-address validation proof;
4. effective-time status lookup proof;
5. receipt-chain verification proof;
6. freshness-mode statement for the target deployment;
7. immutable anchor policy proof with real git commit and/or release artifact
   digest;
8. package/runtime cache-only enforcement proof;
9. pressure review confirming no runtime/package self-authorization;
10. explicit non-Ledger / non-Phase-2 statement preserved.

Until those blockers are closed, the only approved registry work is design.
