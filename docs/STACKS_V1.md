# Igniter::Stack v1

`Igniter::Stack` is the root coordinator for the standard Igniter app layout.

It treats `apps/` as the primary unit of composition:

- one repository
- one root coordinator
- one or more leaf apps under `apps/<name>`
- shared code under `lib/<project>/shared`

Historically this often looked like ‚Äúone app, one deployable process‚Äù.

That is no longer the preferred direction.

`Igniter::App` remains the leaf runtime for an app-shaped code boundary.
`Igniter::Stack` is moving toward service-oriented runtime grouping. See
[STACKS_NEXT.md](./STACKS_NEXT.md).

Generated stacks include `sqlite3` by default. That keeps `Igniter::Stack`
practical out of the box for local app data and execution persistence, while the
core `igniter` gem itself still avoids mandatory production dependencies.

Rules:

- `apps/<name>` is a real unit with its own code and specs.
- shared code lives in `lib/<project>/shared`.
- `config/topology.yml` is for runtime services, deployment roles, and multi-app wiring.
- `config/deploy/*` is for operational artifacts such as Docker and Compose.
- root `spec/` is for shared code, integration specs, and stack boot/config behavior.
- `apps/<name>/spec` is for that app's contracts, executors, tools, agents, and boot behavior.
- one app must not `require` code from another app directly.
- the root stack may start any registered app or service, but app logic still lives inside leaf `Igniter::App` subclasses.

## Stack DSL

```ruby
require "igniter/stack"
require_relative "apps/main/app"
require_relative "apps/inference/app"

module MySystem
  class Stack < Igniter::Stack
    root_dir __dir__
    shared_lib_path "lib"

    app :main,      path: "apps/main",      klass: MySystem::MainApp, default: true
    app :inference, path: "apps/inference", klass: MySystem::InferenceApp
  end
end
```

Generated root file:

```ruby
# stack.rb
require "igniter/stack"
```

Available methods:

- `root_dir(path)` ‚Äî base directory for shared paths.
- `shared_lib_path(path)` ‚Äî add a stack-level support directory to `$LOAD_PATH`.
- `app(name, path:, klass:, default: false)` ‚Äî register a leaf app.
- `app(name = nil)` ‚Äî return the registered leaf `Igniter::App` subclass.
- `app_class(name = nil)` ‚Äî explicit alias for fetching the registered leaf app class.
- `start(name = nil)` ‚Äî start a named app.
- `rack_app(name = nil)` ‚Äî return a Rack app for a named app.
- `start_service(name = nil)` ‚Äî start a named runtime service.
- `rack_service(name = nil)` ‚Äî return a Rack app for a named runtime service.

`stack.yml` is the place for stack-level metadata and shared defaults such as
data-store settings. Unlike `apps/<name>/app.yml`, it is not the leaf app config;
service/runtime grouping should increasingly live in topology.

`config/topology.yml` is the place for deployment-time wiring:

- which service plays which operational role
- which apps live inside which service
- replica counts
- cross-app dependencies
- public vs internal services

Keep this separate from `apps/<name>/app.yml`, which should stay focused on
app-local runtime defaults rather than deployment ports.

## Leaf App DSL

Leaf apps are still plain `Igniter::App` subclasses:

```ruby
require "igniter/app"
require "igniter/core"

module MySystem
  class MainApp < Igniter::App
    root_dir __dir__
    config_file "app.yml"

    tools_path     "app/tools"
    skills_path    "app/skills"
    executors_path "app/executors"
    contracts_path "app/contracts"
    agents_path    "app/agents"

    on_boot do
      register "OrderContract", MySystem::OrderContract
    end
  end
end
```

The important difference is `root_dir __dir__`: it makes relative paths resolve from
`apps/<name>/`, not from the repo root.

## Why `apps/` Is The Default

This layout is the standard because it matches Igniter's strengths better than a flat Rails-style `app/`:

- adding a new bounded context is cheap: create `apps/<name>`
- experiments are easy to isolate and delete
- one repo can host multiple runtime roles without mixing their code
- extraction into a separate deployable unit is straightforward later
- shared code remains explicit instead of leaking across feature folders

In practice, this makes Igniter projects feel more like Lego:

- add an app
- group it into a service when runtime isolation is needed
- keep it if it works
- delete it cleanly if it does not

## Generated Commands

```bash
igniter-stack new my_app
cd my_app
bundle install
bundle exec rspec   # stack + app-local specs
bin/start         # starts apps/main
bin/start --service main    # explicit service selection
ruby bin/demo
```

`config.ru` now typically defaults to the stack's default service unless
`IGNITER_SERVICE` is set.

`stack.rb` and generated `bin/start` also understand deployment-aware launch options:

```bash
bin/start --service main
bin/start --role api
bin/start --role admin
bin/start --env production --role api
bin/start --profile local --role api
bin/start --print-compose
bin/start --write-compose
```

For local multi-app development there is also a stack-native dev mode:

```bash
bin/dev
bin/dev --env production
ruby stack.rb --print-procfile-dev
ruby stack.rb --write-procfile-dev
```

`bin/dev` starts every registered service locally with prefixed logs, without Docker.
`config/deploy/Procfile.dev` is the generated compatibility artifact for `foreman`,
`overmind`, or similar process managers when you want them.

For owner-routed debugging and projection recovery patterns at the stack level,
see [Cluster Debug v1](./CLUSTER_DEBUG_V1.md).

Resolution rules:

- explicit service name wins when using `--service`
- otherwise `--role` resolves the first matching service from `config/topology.yml`
- otherwise the stack default service is used
- legacy app-shaped topology continues to work
- `--env` loads `config/environments/<name>.yml`
- `--profile` validates against `topology.profile`
- `--print-compose` renders a Docker Compose config from `config/topology.yml`
- `--write-compose` writes the rendered Compose config to `config/deploy/compose.yml`
