# Контрактная модель Igniter как основа нового языка — Исследовательский репорт

Дата: 2026-04-26.
Перспектива: экспериментальные исследования в области языков программирования; теория информационной плотности.
Тема: Осуществимость, набросок грамматики, полнота по Тьюрингу и гипотеза об информационной плотности для контрактно-нативного языка на основе ядра Igniter.

*Статус: экспериментальное исследование — не feature proposal, не трек.*

*Каноничная версия: [igniter-lang.md](igniter-lang.md)*

---

## 1. Вопрос

Ruby DSL Igniter выразителен. Контрактная модель — валидированный, compile-time граф зависимостей с ленивым разрешением и структурным кешированием — сильная абстракция. Но DSL всё равно *hosted* в Ruby, и Ruby сам начинает генерировать синтаксический шум: service-классы, наследование boilerplate, файловую структуру, обёртки `def call(kwargs:)`. Каждый узел `compute` в итоге делегирует в Ruby-класс, который существует исключительно для того, чтобы держать одно выражение.

Исследовательский вопрос:

> Может ли контрактная модель Igniter стать семантическим основанием нового
> языка программирования — такого, где топология графа и вычисления внутри
> каждого узла говорят на одном языке, устраняя налог языка-хозяина?

---

## 2. Возможно ли это?

**Да, и основания для этого unusually сильные.**

DSL Igniter уже определяет полный набор примитивов уровня языка:

| Ключевое слово DSL | Концепция уровня языка |
|--------------------|------------------------|
| `input` | типизированный параметр / свободная переменная |
| `compute` | именованное выражение (let-binding) |
| `branch` | pattern matching / условный dispatch |
| `compose` | применение функции / вызов контракта |
| `collection` | map / functor над коллекцией |
| `guard` | assertion / предусловие |
| `await` | алгебраический эффект / async-приостановка |
| `effect` | ограниченный side effect |
| `aggregate` | fold / reduce |
| `const` | константное связывание |
| `project` / `map` | проекция поля / трансформация значения |

Топологически отсортированный `resolution_order`, производимый `GraphCompiler`, уже является планом исполнения. `CompiledGraph` уже является AST. Чего не хватает — не семантической модели, а *конкретного синтаксиса* и возможности писать вычислительные выражения inline в узле, без выхода в Ruby-класс.

## 3. Гипотеза об информационной плотности

### 3.1 Формулировка проблемы

Текущая кодировка программы-контракта:

```
Программа = Contract DSL  (топология, ~1 строка/узел)
          + Ruby-классы   (вычисления, ~5–10 строк/узел)
```

Ruby — язык общего назначения; его синтаксис оптимизирован под все программы, а не конкретно под контрактные графы. Результат — систематическое *несоответствие плотности*: контракт описывает *что* и *в каком порядке* с высокой плотностью; Ruby-класс описывает *как* с низкой плотностью, обёрнутый в языковую механику, не имеющую отношения к вычислению.

```ruby
# Контракт: одна строка несёт четыре семантических утверждения
compute :vendor, depends_on: :vendor_id, call: FetchVendor, cache_ttl: 60

# Реализация: пять строк ради одного выражения
class FetchVendor < Executor
  def call(vendor_id:)
    Vendor.find_by!(id: vendor_id)
  end
end
```

Информация была упакована, затем *принудительно разжата* обёрткой.

### 3.2 Формальный контекст

Это отображается на **сложность Колмогорова** и **domain-specific compression**. Язык, чьи примитивы соответствуют домену, может описать то же вычисление в меньшем числе символов — не за счёт лаконичности, а за счёт устранения концепций, которых в домене не существует.

В контрактном домене не существует: идентичности объектов, иерархий диспетчеризации методов, файловых пространств имён, наследования классов для простых трансформаций значений. Контрактно-нативный язык не обязан их представлять.

Предлагаемая метрика для эмпирической проверки — **Semantic Information Ratio**:

```
SIR = (количество различных бизнес-утверждений) / (общий LOC)
```

Гипотеза: для типичных workflow-программ контрактно-нативный язык достигает SIR > 2× по сравнению с Ruby DSL + классы-исполнители.

### 3.3 Почему «уровень процесса» vs «уровень объекта»

Контрактная модель работает на *уровне процесса*: описывает вычисление как поток именованных значений через направленный граф. Ruby работает на *уровне объекта*: описывает вычисление как передачу сообщений между объектами. Принудительное делегирование объявления процессного уровня на реализацию объектного уровня является корневой причиной потери плотности — не недостатком Ruby, а фундаментальным несоответствием уровней.

## 4. Набросок грамматики

### 4.1 Уровень 1 — Contract-First синтаксис (минимально жизнеспособный)

Расширение семантики Igniter inline-выражениями в узлах compute. Не новый язык, но максимальный шаг в рамках DSL:

```
contract PriceQuote {
  in vendor_id: Id
  in zip_code:  Zip
  in as_of:     Time = now()

  compute vendor = http.get("/vendors/{vendor_id}") |> Vendor.parse
                   @cache(60s)

  compute slots  = vendor.slots
                     |> select { |s| s.zip == zip_code && s.available }
                     |> select { |s| s.time > as_of }
                   @coalesce

  branch availability {
    on slots.count > 0 => AvailableQuote { slots: slots, vendor: vendor }
    default            => EmptyQuote     { vendor: vendor }
  }

  out quote  = availability
  out vendor = vendor
}
```

Ключевые синтаксические решения:
- `=` вместо `call:` — выражение прямо на месте связывания
- `@cache(60s)`, `@coalesce` — аннотации как декораторы узлов
- `|>` — оператор pipe для трансформаций (без промежуточных имён)
- `on expr => Contract` — branch на произвольном выражении, не только `eq:` / `in:`

### 4.2 Уровень 2 — Самостоятельный контрактный язык

Самодостаточный язык с типами, функциями и контрактами как равноправными first-class гражданами.

**Скелет EBNF:**

```ebnf
program      = (type_def | fn_def | contract_def)*

type_def     = "type" IDENT "{" field+ "}"
field        = IDENT ":" type_expr

contract_def = "contract" IDENT (":" IDENT)? "{" node* "}"
node         = in_node | compute_node | branch_node
             | compose_node | out_node

in_node      = "in"      IDENT ":" type_expr ("=" expr)?
compute_node = "compute" IDENT "=" expr annotation*
branch_node  = "branch"  IDENT "{" arm+ default_arm? "}"
arm          = "on"      expr "=>" expr
default_arm  = "default"      "=>" expr
compose_node = "compose" IDENT "=" IDENT "{" (IDENT ":" expr)* "}"
out_node     = "out"     IDENT "=" expr

annotation   = "@" IDENT ("(" arg_list ")")?

fn_def       = "fn" IDENT "(" param* ")" "->" type_expr "=" expr

expr         = literal | path | call | pipe | block | if_expr | contract_call
pipe         = expr "|>" expr
block        = "{" stmt* "}"
contract_call = IDENT "{" (IDENT ":" expr)* "}"
```

**Пример полной программы:**

```
type Vendor {
  id:    Id
  name:  String
  slots: [Slot]
}

type Slot {
  time:      DateTime
  zip:       Zip
  available: Bool
}

fn parse_vendor(data: Json) -> Vendor = Vendor {
  id:    data.id
  name:  data.name
  slots: data.slots |> each(Slot.parse)
}

contract PriceQuote {
  in vendor_id: Id
  in zip_code:  Zip
  in as_of:     DateTime = now()

  compute vendor = http.get("/vendors/{vendor_id}") |> parse_vendor
                   @cache(5min)

  compute slots  = vendor.slots
                     |> select { |s| s.zip == zip_code }
                     |> select { |s| s.available }
                     |> select { |s| s.time > as_of }
                   @coalesce

  branch result {
    on slots.count > 0 => AvailableQuote { slots: slots, vendor: vendor }
    default            => EmptyQuote     { vendor: vendor }
  }

  out quote  = result
  out vendor = vendor
}

contract FullOrder {
  in order_id: Id

  compose order   = FetchOrder     { id: order_id }
  compose pricing = PriceQuote     { vendor_id: order.vendor_id,
                                     zip_code:  order.zip }
  compose stock   = InventoryCheck { vendor_id: order.vendor_id }

  out confirmed = merge(pricing.quote, stock.availability)
}
```

## 5. Полнота по Тьюрингу

### 5.1 DAG не полон по Тьюрингу — и это намеренно

Ациклический граф вычисляет строго ограниченный класс функций. Нет цикла → нет нетривиальной рекурсии → нет полноты по Тьюрингу. Это не недостаток; это свойство: **DAG-контракты всегда завершаются**. Они лежат в (не более чем) PTIME и допускают статическое доказательство ограниченности ресурсов, отсутствия бесконечных циклов и полной наблюдаемости графа.

Для целевого домена — бизнес-workflow, финансовая логика, compliance-пайплайны — гарантия завершения является достоинством, которое стоит сохранить.

### 5.2 Три пути к полноте

**Путь A — Рекурсивные контракты**

Контракт может вызывать сам себя. DAG разворачивается в дерево вызовов; завершение — свойство входных данных, а не языка.

```
contract Fibonacci {
  in n: Int

  branch result {
    on n <= 1 => n
    default   => Fibonacci { n: n-1 }.value
              +  Fibonacci { n: n-2 }.value
  }

  out value = result
}
```

**Путь B — Явный узел итерации**

Единственная конструкция с циклом; остальной граф остаётся ациклическим. Полон по Тьюрингу при неограниченном условии завершения.

```
compute primes = iterate(seed: 2, acc: []) { |n, acc|
  next:       next_candidate(n)
  accumulate: acc + [n] when is_prime(n)
  until:      n > limit
}
```

**Путь C — Потоковые / коиндуктивные контракты**

Контракт, чей вход — поток, разворачивается бесконечно. Полон по Тьюрингу через корекурсию.

```
contract LivePricing {
  in vendor_id: Id
  stream events: PriceEvent

  compute quote = fold(events, initial_quote) { |state, event|
    update_quote(state, event)
  }

  out quote = quote
}
```

### 5.3 Рекомендуемая стратегия

Начать с намеренно **неполного по Тьюрингу** ядра (гарантированное завершение, статический анализ, полная наблюдаемость). Добавить полноту через явные opt-in расширения: `iterate`, рекурсивные контракты, потоковые входы. Пользователи, которым нужны циклы, объявляют их явно; остальная часть программы сохраняет свои safety-свойства.

## 6. Формальное позиционирование

### 6.1 Ближайшие академические аналоги

| Модель | Сходство | Отличие |
|--------|----------|---------|
| Kahn Process Networks (1974) | данные текут по рёбрам, процессы в узлах | узлы KPN — бесконечные циклы; узлы контракта — однократные |
| Call-by-need λ-исчисление | ленивые вычисления, sharing | зависимости в λ неявные; в контрактах — явные и именованные |
| Lustre / Esterel | синхронный dataflow для safety-critical систем | ориентированы на hardware/embedded; нет модели композиции и кеширования |
| FRP / Elm | реактивный dataflow | сфокусированы на UI event streams; нет compile-time валидации графа |
| Self-Adjusting Computation (Acar 2002) | инкрементальное повторное вычисление при изменении входа | нет явного объявления графа; propagation автоматическая, но непрозрачная |

### 6.2 Что ново

Такая комбинация не встречается в существующих работах:

1. **Именованные промежуточные результаты** — каждый узел имеет стабильную идентичность, что позволяет интроспекцию, diffing и отслеживание провенанса
2. **Compile-time валидация графа** — ошибки топологии обнаруживаются до исполнения
3. **Структурные аннотации кеширования** — `@cache`, `@coalesce` как first-class атрибуты графа, а не runtime-подсказки
4. **First-class композиция** — контракты компонуются структурно, а не вызовом методов друг друга
5. **Ориентация на бизнес-домен** — спроектировано для workflow, а не для научных вычислений или UI-рендеринга

Центральный тезис:

> Контрактная модель реифицирует *структуру вычисления* как программу.
> Граф И ЕСТЬ код. Интроспекция — не инструмент отладки, а первичное
> представление.

## 7. Почему Ruby DSL — не финальная форма

Три конкретных ограничения:

**7.1 Налог обёртки на каждый узел**  
Каждый `compute` требует класс, наследование и метод `call`. Это структурный налог, навязанный языком-хозяином, а не доменом.

**7.2 Выражения не first-class в узлах**  
`call:` принимает только ссылку на класс. Поддержка лямбд существует, но является второстепенным путём, а не основным — и анонимные callable теряют интроспекцию.

**7.3 Типы декоративные, а не структурные**  
`type:` в `input` / `output` — метаданные для валидации и документации. Они не текут вдоль рёбер графа как статические типы, поэтому компилятор не может вывести, что узел, возвращающий `Money`, питает узел, ожидающий `Numeric` — этот вывод происходит только в runtime.

Контрактно-нативный язык устраняет все три ограничения: inline-выражения, нет классов-обёрток, вывод типов вдоль рёбер.

## 8. Открытые исследовательские вопросы

1. **Типизация эффектов**: как типизировать IO-узлы без monadic overhead? Алгебраические эффекты (OCaml 5, Koka) — наиболее перспективное направление.

2. **Динамическая топология**: `branch` разрывает статическую топологию — подграф, выбираемый в runtime, неизвестен при компиляции. Как типизировать union-of-graphs в точке ветвления?

3. **Транзитивность кеша**: если `PriceQuote` кешируеt `vendor` на 5 минут, а `FullOrder` компонует `PriceQuote` — как cache-политики транзитивно распространяются через границы композиции?

4. **Измерение плотности**: метрика SIR требует референсного корпуса. Предложение: переписать 10 примеров из `examples/` на новом синтаксисе, измерить LOC, SIR и цикломатическую сложность по сравнению с Ruby-оригиналами.

5. **Переиспользование runtime**: может ли парсер, targeting контрактно-нативный синтаксис, компилироваться напрямую в `CompiledGraph` Igniter и работать на существующем Ruby runtime? Это даст рабочий прототип без нового VM.

## 9. Практический путь к прототипу

Самый быстрый способ проверить гипотезу без написания полного компилятора:

**Шаг 1 — Парсер поверх Ruby** (~500 строк)  
PEG или hand-written парсер для компактного синтаксиса, генерирующий вызовы `ContractBuilder`. Разделяет весь Igniter runtime.

**Шаг 2 — Inline-выражения**  
Добавить поддержку `compute :x = expr` где `expr` — минимальный expression language (path, pipe, block). Компилируется в анонимную лямбду.

**Шаг 3 — Измерение плотности**  
Переписать 5–10 существующих примеров; измерить SIR, LOC и cognitive complexity.

**Шаг 4 — Вывод типов** (при подтверждении гипотезы)  
Добавить статический вывод типов вдоль рёбер. Это граница перехода к полноценной языковой работе — type checker над DAG, работающий в compile time.

---

## Резюме

Контрактная модель Igniter — **сильное теоретическое основание** для нового языка. Семантическое ядро уже существует в DSL и компиляторе. Гипотеза об информационной плотности формально обоснована: переход от объектного уровня (классы-исполнители) к процессному уровню (inline-выражения в узлах) устраняет систематический налог языка-хозяина и должен измеримо повысить семантическую плотность.

Ядро DAG даёт гарантированное завершение как свойство безопасности по умолчанию. Полнота по Тьюрингу достижима через явные opt-in расширения. Комбинация именованных промежуточных результатов, compile-time валидации графа и структурного кеширования не встречается совместно в существующих dataflow-языках.

Наиболее критичный открытый вопрос — не осуществимость, а выразительность: как далеко может зайти минимальный expression language внутри узлов до того, как ему нужно стать полноценным языком? Это эмпирический вопрос. Описанный выше путь к прототипу отвечает на него при низких затратах.
