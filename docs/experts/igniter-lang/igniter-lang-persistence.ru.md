## §0 Центральная Идея

В обычных системах персистентность — это инфраструктурная задача: настроить базы
данных, написать миграции, управлять пулами соединений, аннотировать модели.
Система типов ничего не знает о хранилище.

В языке, нативном для контрактов, **тип полностью определяет паттерн доступа**.
`History[T]` — это не тег. Это декларация: только-добавление, индексирование по
времени, сегментная структура, причинная согласованность. `OLAPPoint[T, dims]` —
декларация: колонарное хранилище, прорезка партиций, тяжёлые сканы, scatter-gather
параллелизм. `entity` — декларация: транзакционность, точечные чтения, ограничения
внешних ключей.

Компилятор способен вывести физическую схему хранения, стратегию размещения в
кластере и требования к согласованности **исключительно из типа**. Персистентность
становится свойством типа, а не инфраструктурной конфигурацией. Единственное, что
требует явной настройки — то, что нельзя вывести: фактор репликации, допустимое
отставание, выбор конкретного бэкенда.

Вторичная идея: путь записи унифицируется с лексикой языка. Разрешение конфликтов
при конкурентных записях использует `combines:` — тот же предикат, что и для
конфликтов правил. Условие запечатывания `HistorySegment` — это временной инвариант.
Чекпойнты состояния выполнения — просто `History[ExecutionState]`. Весь язык
применяется к слою персистентности без отдельного словаря.

## §1 Таксономия Форм Хранения

Каждый конструкт языка имеет естественную *форму хранения* — набор паттернов
доступа, который полностью определяет требования к физическому бэкенду.

| Конструкт | Форма | Паттерн доступа | Требование согласованности |
|-----------|-------|-----------------|---------------------------|
| `entity` | Реляционная | Точечные чтения/записи, JOIN, транзакции | Сильная (ACID) |
| `History[T]` | Лог только-добавления | Сканы по диапазону времени, без обновлений | Причинная |
| `BiHistory[T]` | БиTempoральный лог | Сканы по двум осям, исправления через добавление | Причинная |
| `OLAPPoint[T, dims]` | Колонарное | Сканы с прорезкой партиций, агрегации | Eventual |
| `await` / состояние саги | Долговечный лог событий | Последовательное добавление, воспроизведение | At-least-once |
| `cache_ttl` / coalesce | Временное KV | Точечные чтения/записи, TTL-вытеснение | Нет (best-effort) |
| Декларации `rule` | Версионируемый набор правил | Чтение-интенсивные, иммутабельные версии | Причинная |

Компилятор считывает эти формы из аннотаций типов узлов и создаёт *манифест
требований к хранению* как часть артефакта компиляции:

```
CompiledGraph {
  resolution_order: [...],
  storage_requirements: {
    :price_history  => { shape: :timeseries, type: History[Money], ... },
    :revenue_cube   => { shape: :columnar,   type: OLAPPoint[...], ... },
    :order_workflow => { shape: :log,        type: ExecutionState, ... },
  }
}
```

Этот манифест — контракт между языком и инфраструктурным слоем. Среда выполнения
читает его для инстанцирования правильных адаптеров бэкенда.

## §2 `Store[T]` как Языковой Конструкт

Декларация `store` вводит именованное, типизированное, персистентное хранилище в
граф контракта. Оно само является контрактом: `Contract({}) → Store[T]`.

```
store :price_history, History[Money],
  backend:     :timeseries,          # физический движок
  partition:   :by_product,          # стратегия партиционирования
  consistency: :causal,              # уровень согласованности
  replicas:    3,                    # репликация в кластере
  seal_after:  { size: 10_000, time: 1.hour }  # запечатывание головного сегмента

store :revenue_cube, OLAPPoint[Money, { product: String, region: String, month: Date }],
  backend:            :columnar,
  partition:          :by_month,
  source:             :orders,       # мост операционное → аналитическое
  materialization:    :incremental,  # стратегия (см. §5)
  lag:                30.seconds

store :workflow_log, ExecutionState,
  backend:     :log,
  retention:   90.days,
  idempotency: :content_addressed    # дедупликация по хешу входных данных
```

**Варианты бэкендов** (открытый набор, подключаемые):

| Бэкенд | Подходит для | Примеры |
|--------|-------------|---------|
| `:memory` | Разработка, тесты | Встроенный в Igniter |
| `:timeseries` | `History[T]`, `BiHistory[T]` | TimescaleDB, InfluxDB, нативный |
| `:columnar` | `OLAPPoint` | DuckDB, ClickHouse, Parquet |
| `:relational` | `entity` | PostgreSQL, SQLite |
| `:log` | `await`, состояние саги | NATS JetStream, Kafka, нативный |
| `:kv` | `cache_ttl` | Redis, Memcached, нативный |

Маппинг по умолчанию компилятора (используется при неявном `store`):

```
History[T]         → :timeseries  (или :log при отсутствии кластера)
BiHistory[T]       → :timeseries  с поддержкой биtempoральности
OLAPPoint[T, dims] → :columnar
entity             → :relational
await / saga       → :log
cache_ttl          → :kv
```

Явные декларации `store` переопределяют умолчание. При отсутствии декларации и
ненастроенном бэкенде компилятор выдаёт предупреждение и откатывается к `:memory`
(безопасно для разработки, не для продакшена).

## §3 Модель Кластерного Параллелизма

Граф контракта статически кодирует структуру параллелизма. Три уровня:

### Уровень 1 — Параллелизм узлов (внутри контракта)

Уже реализован через runner `thread_pool`. Компилятор вычисляет топологические
слои; все узлы одного слоя независимы и выполняются конкурентно. Дополнительных
изменений не требуется — это базовый уровень.

### Уровень 2 — Параллелизм контрактов (между контрактами)

Целые контракты маршрутизируются на ноды кластера. Два режима:

**Маршрутизация без состояния**: контракты без `await` и персистентного состояния
могут выполняться на любой ноде. Маршрутизатор на основе consistent-hash распределяет
контракты по нодам по хешу входных данных, обеспечивая локальность данных без
координации.

**Маршрутизация с состоянием**: контракты с узлами `await` или доступом к `store`
должны маршрутизироваться на ноду, владеющую соответствующей партицией. Карта
партиций (распространяется через gossip) отображает `(store_name, partition_key)` →
`(node_address, replica_addresses)`.

### Уровень 3 — Параллелизм данных (fanout контракта)

Наиболее мощная форма. Когда контракт читает из партиционированного `store`,
компилятор генерирует *план fanout*:

```
# Пользователь пишет:
contract :monthly_revenue do
  input :year
  olap :revenue, Revenue, rollup: :month, partition: :by_region
  output :by_region_month, from: :revenue
end

# Компилятор генерирует (концептуально):
FanoutPlan {
  scatter: запросить карту партиций → N регионов
  per_shard: выполнить :monthly_revenue на ноде-владельце каждого региона
  gather:    собрать OLAPSlice[N] → объединить через функцию rollup
}
```

Пользователь пишет один контракт. Компилятор создаёт план scatter-gather MapReduce.
Никаких `parallel do`, явной координации, `async/await`. Топология декларирует
параллелизм.

### Размещение с Учётом Топологии

Компилятор вычисляет *граф стоимости коммуникаций*: для каждой пары зависимых узлов
`(A → B)` — каков ожидаемый объём передачи данных? Узлы с высокой стоимостью
коммуникаций должны быть размещены совместно.

Цель размещения: минимизировать суммарный сетевой I/O при соблюдении ограничений
балансировки нагрузки.

```
PlacementPlan {
  :price_lookup   → node_1  (владеет партицией P1 price_history)
  :tax_compute    → node_1  (зависит от :price_lookup, та же нода = 0 сетевых переходов)
  :revenue_olap   → node_2  (владеет партицией R1 revenue_cube)
  :final_total    → gateway (агрегирует с node_1 + node_2)
}
```

Это подсказка компилятора, не жёсткое ограничение. Планировщик может переопределить
её для балансировки, но подсказка управляет размещением по умолчанию.

## §4 Путь Записи для `History[T]`

Чтение `History[T]` решено: контент-адресованные запечатанные сегменты, свободно
распределяемые, доступ O(log n), инвалидация кеша не нужна. Запись сложнее.

### Проблема Головного Сегмента

Каждый `History[T]` содержит:
- **N запечатанных сегментов**: иммутабельные, контент-адресованные, свободно реплицируемые
- **1 мутабельный голова**: открытый сегмент, в который идёт текущая запись

Голова — граница конкурентности. Два конкурентных писателя не могут одновременно
обновлять голову без координации.

### Четыре Стратегии Записи

**Стратегия A — Единственный писатель**

Каждая партиция `History[T]` имеет назначенную ноду-писатель. Все записи в эту
партицию маршрутизируются к ней. Другие ноды перенаправляют записи через карту
партиций.

- Согласованность: сильная (конфликт невозможен)
- Пропускная способность: ограничена единственным писателем
- Отказ: падение писателя → записи ставятся в очередь до выбора нового
- Запечатывание: писатель решает, когда запечатать (по достижении порога размера/времени)

**Стратегия B — Log-Structured Merge (LSM)**

Несколько писателей добавляют в свои локальные микросегменты. Фоновый процесс
слияния консолидирует микросегменты в запечатанные сегменты (как компакция в
LevelDB).

- Согласованность: eventual в окне слияния
- Пропускная способность: высокая (нет координации на пути записи)
- Сложность: процесс слияния должен обрабатывать перекрывающиеся временные интервалы

**Стратегия C — CRDT-сегменты**

Каждая запись немедленно создаёт новый запечатанный микросегмент. Слияние — ленивое,
при чтении. Работает, поскольку сегменты — это интервалы; слияние — хорошо
определённая операция над множествами.

```
Писатель A: сегмент { [t=10, t=10]: price=100, recorded: now }
Писатель B: сегмент { [t=10, t=10]: price=110, recorded: now+1ms }
Чтение при t=10: оба сегмента существуют → применить разрешение конфликта
```

- Согласованность: eventual, но причинно упорядоченная (transaction_time — тайbreaker)
- Координация: отсутствует на пути записи
- Накладные расходы при чтении: слияние при чтении (амортизируется кешем сегментов)

**Стратегия D — Запечатывание через координатор**

Писатели добавляют в Raft-реплицированный лог. Координатор читает лог и создаёт
запечатанные сегменты. Сильная согласованность, координатор — узкое место.

Язык выражает выбор как параметр `store`:

```
store :prices, History[Money],
  write: :single_writer   # Стратегия A — сильная согласованность, проще всего
  # write: :crdt          # Стратегия C — без координации, eventual
  # write: :lsm           # Стратегия B — высокая пропускная способность
```

По умолчанию: `:single_writer` (проще, подходит большинству). `:crdt` — для сценариев
с высоким объёмом записи и требованием partition-tolerance.

### Разрешение Конфликтов Записи

Когда два писателя создают значения для одного `valid_time`:

```
Писатель A: price[valid_time=T] = 100
Писатель B: price[valid_time=T] = 110   (конкурентно)
```

Разрешение использует лексику `combines:` из деклараций правил — тот же предикат
применяется здесь. Это не случайность: конфликт конкурентных записей структурно
идентичен конфликту правил.

```
store :prices, History[Money],
  write_conflict: :last_wins        # естественная модель BiHistory (по transaction_time)
  # write_conflict: :error          # строгий режим — конфликт как ошибка времени выполнения
  # write_conflict: combines: :max  # пользовательский: max(100, 110) = 110
  # write_conflict: combines: :min
```

Унификация конфликта правил и конфликта записи в одной лексике — это проектная цель:
в языке одна концепция для "что делать, когда два значения претендуют на один слот",
применяемая последовательно.

## §5 Материализация Операционного в Аналитическое

Предикат `source:` в хранилище `OLAPPoint` объявляет мост от операционной сущности
к аналитическому хранилищу. Три стратегии материализации:

### Стратегия 1 — Синхронная (write-through)

```
Запись в OLTP → атомарное обновление OLAP-хранилища (та же транзакция)
```

Каждая запись `entity` также пишет в хранилище `OLAPPoint` в той же транзакции.
- Согласованность OLAP: сильная (нулевое отставание)
- Задержка OLTP: увеличивается на накладные расходы записи в колонарное хранилище
- Лучше всего для: низкого объёма записи, чтения-интенсивного OLAP, небольших таблиц фактов

### Стратегия 2 — CDC (захват изменений данных)

```
Запись в OLTP → commit → CDC-событие → асинхронная запись в OLAP
```

Поток событий переносит дельты из операционного лога в OLAP-хранилище. Стандартный
подход (Debezium и т.д.), но в Igniter-Lang он декларируется, а не конфигурируется:

```
store :revenue_cube, OLAPPoint[...],
  source:          :orders,
  materialization: :cdc,
  lag:             30.seconds   # SLA допустимого отставания
```

Компилятор генерирует `MaterializationContract`, подписывающийся на события `orders`
и применяющий проекции к OLAP-хранилищу. Параметр `lag` становится временным
инвариантом на `recorded_time` хранилища.

### Стратегия 3 — Инкрементальный потоковый расчёт

```
Запись в OLTP → дельта → распространение через OLAP-граф → минимальный пересчёт
```

Использует `igniter/extensions/dataflow` (уже реализовано). Пересчитываются только
затронутые агрегационные ячейки. Лучше всего для сложных OLAP-графов со многими
производными измерениями.

```
store :revenue_cube, OLAPPoint[...],
  source:          :orders,
  materialization: :incremental,
  lag:             5.seconds
```

### Материализация как Контракт

Критически важно: пайплайн материализации сам является первоклассным контрактом:

```
MaterializeRevenueCube: Contract({orders_stream: Stream[Order]}) → Void

  input  :orders_stream
  compute :completed, filter: orders_stream by status == :completed
  compute :revenue_records, map: completed → RevenueRecord
  effect  :write_olap, depends_on: :revenue_records,
            call: OLAPStore::Writer,
            idempotent: true,
            store: :revenue_cube
```

Это означает, что стратегия материализации:
- Декларирована в языке, не в инфраструктурных скриптах
- Верифицирована компилятором (типы входа/выхода проверены)
- Наблюдаема через те же инструменты интроспекции, что и любой контракт
- Тестируема той же тестовой инфраструктурой
- Аудируема через то же расширение провенанса

## §6 Модель Согласованности

### CAP в Терминах Контрактов

| Конструкт контракта | Partition tolerance | Требование согласованности | Выбор CAP |
|--------------------|--------------------|-----------------------------|-----------|
| Запись `entity` | По выбору | Сильная (ACID) | CA (одна нода) или CP (Raft) |
| Добавление в `History[T]` | Требуется | Причинная | CP или AP+причинная |
| Чтение `OLAPPoint` | Требуется | Eventual (SLA отставания) | AP |
| Доставка события `await` | Требуется | At-least-once + дедупликация | AP |
| Применение правила | Требуется | Причинная (версия правила) | CP |

Параметр `consistency:` в `store` отображается на эти выборы:

| Значение | Семантика | Требование к сети |
|----------|-----------|------------------|
| `:strong` | Линеаризуемые чтения/записи | Raft-консенсус на записи |
| `:causal` | Причинный порядок сохранён | Векторные часы на всех операциях |
| `:monotonic_reads` | Чтения никогда не идут назад | Привязанная сессия на читателя |
| `:eventual` | Last-write-wins по transaction_time | Без координации |

### Согласованность Связана с Временным `as_of`

Уровень согласованности хранилища определяет семантику запросов `as_of` к нему.
Это прямо связано с моделью распределённого времени:

```
# Хранилище с согласованностью :causal
store :prices, History[Money], consistency: :causal

# Запрос: as_of использует причинные часы, унаследованные от контекста выполнения контракта.
# Если этот контракт был запущен событием от :order_service,
# то as_of причинно следует после всех записей из причинного прошлого этого события.
price[:as_of last_event_from: :order_service]
```

Для хранилищ с `:eventual` запросы `as_of` могут возвращать устаревшие данные
в пределах SLA отставания. Параметр `lag:` становится временным инвариантом:

```
# Инвариант, проверяемый компилятором:
# revenue_cube.recorded_time >= now - lag
# При нарушении запрос вызывает TemporalConsistencyError
```

## §7 Долговечность Состояния Выполнения

### Проблема

Между узлами `await` выполнение контракта приостановлено. При падении ноды
выполнение должно восстановиться в правильную позицию без повторного выполнения
завершённых узлов.

```
class OrderWorkflow < Igniter::Contract
  correlate_by :order_id
  define do
    input   :order_id
    await   :payment,  event: :payment_confirmed   # ← точка приостановки 1
    await   :shipping, event: :shipped              # ← точка приостановки 2
    effect  :notify_customer, depends_on: :shipping, idempotent: true
    output  :complete, from: :shipping
  end
end
```

### Состояние Выполнения как Лог Событий

Состояние выполнения персистируется как упорядоченный лог событий:

```
[start,           { order_id: "o1", inputs: { order_id: "o1" } }]
[node_resolved,   { node: :order_id, value: "o1" }]
[node_suspended,  { node: :payment, awaiting: :payment_confirmed,
                    correlation: { order_id: "o1" } }]
[event_received,  { node: :payment, payload: { amount: 99.99 } }]
[node_resolved,   { node: :payment, value: { amount: 99.99 } }]
[node_suspended,  { node: :shipping, awaiting: :shipped, ... }]
...
```

При восстановлении:
1. Прочитать лог для ключа корреляции `{ order_id: "o1" }`
2. Воспроизвести для восстановления состояния выполнения в памяти
3. Возобновить с последнего приостановленного узла — без повторного выполнения завершённых

### Требование Идемпотентности

Воспроизведение должно быть безопасным. Чистые compute-узлы тривиально идемпотентны
(те же входы → тот же выход, нет побочных эффектов). Эффекты требуют явной
идемпотентности:

```ruby
class SendConfirmationEmail < Igniter::Effect
  idempotent true    # существующий DSL — уже реализован
  effect_type :notification
  def call(order_id:)
    EmailService.send_confirmation(order_id)
  end
end
```

Флаг `idempotent: true` уже существует в системе Effect. Компилятор обязывает,
чтобы все узлы `effect` в долговечном воркфлоу объявляли идемпотентность.

### Протокол Чекпойнтов

Для долгоживущих воркфлоу (дни, недели) воспроизведение с начала дорого. Протокол
чекпойнтов запечатывает снапшоты состояния выполнения:

```
Checkpoint = {
  id:              content_hash(execution_state),   # контент-адресованный
  at_node:         :shipping,                       # после разрешения этого узла
  state_snapshot:  serialized_execution_state,
  log_offset:      1247,                            # возобновить воспроизведение отсюда
  recorded_at:     Time.now
}
```

Чекпойнты запечатаны, иммутабельны и контент-адресованы — точно как `HistorySegment`.
Это не аналогия; это тот же конструкт:

```
ExecutionCheckpoints ≡ History[ExecutionState]
```

Та же инфраструктура `HistorySegment` хранит чекпойнты выполнения. Восстановление
загружает последний чекпойнт и воспроизводит только последующие записи лога.

### Хранилище Долговечности

```
store :order_workflow_state, ExecutionState,
  backend:     :log,
  checkpoint:  { after: :every_await, max_replay: 1000 },
  retention:   90.days,
  idempotency: :content_addressed
```

## §8 Единая Архитектура

Полный стек от декларации типа до физического выполнения:

```
Слой языка
  contract / entity / store / await
         ↓  (время компиляции)
Слой компилятора
  - Манифест требований к хранению (форма → бэкенд)
  - План fanout (карта партиций → scatter-gather)
  - Подсказки размещения (граф стоимости коммуникаций)
  - Контракты материализации (source: → пайплайн)
  - Инварианты согласованности (SLA lag → временной инвариант)
         ↓  (инициализация среды выполнения)
Слой адаптеров хранения
  Реестр бэкендов: { :timeseries → TimescaleAdapter,
                     :columnar   → DuckDBAdapter,
                     :log        → NATSAdapter,
                     :relational → PostgreSQLAdapter,
                     :kv         → RedisAdapter,
                     :memory     → BuiltinAdapter }
         ↓  (выполнение)
Слой кластерного выполнения
  Карта партиций  →  scatter-gather runner
  Карта размещения →  маршрутизация на ноды
  Векторные часы  →  причинная согласованность
  Raft-лог        →  сильная согласованность (опционально)
```

На каждом слое граф контракта — единица работы. Контракт — не "нечто, что
выполняется и использует базу данных". Это типизированный граф потоков данных,
чей артефакт компиляции содержит полное описание требований к хранению, размещению
и согласованности.

## §9 Открытые Вопросы

**В1: Версионирование хранилища.** Когда тип `store` меняется (миграция схемы),
какова модель обновления? `History[T]` — только-добавление; старые сегменты имеют
старый тип. Возможный ответ: запечатанные сегменты иммутабельны, миграция создаёт
новое хранилище с версионированным именем, контракт миграции трансформирует старые
сегменты в новый формат.

**В2: Транзакции между хранилищами.** Может ли один контракт атомарно писать в
хранилище `entity` и добавлять в `History[T]`? Это требует распределённых
транзакций (2PC или сага). Расширение saga уже существует — естественный ответ:
используйте контракт-сагу для записей между хранилищами, а не атомарные транзакции.

**В3: Обнаружение хранилища в динамическом кластере.** Карта партиций
распространяется через gossip. Каково время сходимости? Что происходит при
разделении, когда карта устаревшая? Уровень согласованности должен определять
ответ: хранилища `:strong` отклоняют записи при разделении; `:eventual` принимают
записи и согласовывают позже.

**В4: Портируемость колонарного бэкенда.** DuckDB — отличный встроенный колонарный
движок, но не нативен для кластера. Бэкенд `:columnar` может потребовать двухуровневой
модели: in-process DuckDB для одной ноды + scatter-gather протокол для кластера.
Файлы Parquet на общем объектном хранилище (S3) могут быть простейшим кластерно-
переносимым вариантом.

**В5: Хранилище правил и временная согласованность.** При развёртывании новой версии
`rule` выполнения контрактов в процессе могут находиться в середине вычисления,
используя старый набор правил. Компилятор должен генерировать привязку версии
правила в момент начала контракта, гарантируя, что одно выполнение всегда использует
один согласованный снапшот правил.

## §10 Дорожная Карта POC

### Итерация 1 — Интерфейс персистентности (~500 строк кода)

- Абстрактный интерфейс `Igniter::Storage::Backend`: `read`, `append`, `scan`, `checkpoint`
- `Igniter::Storage::Memory` — in-process бэкенд (уже частично через `NodeCache::Memory`)
- `Igniter::Storage::Log` — append-only лог для долговечности `await`/saga
- Ключевое слово `store` в `ContractBuilder`
- Компилятор генерирует манифест требований к хранению
- Компилятор проверяет: все узлы `await` в контракте имеют долговечный лог

Результат: распределённые контракты переживают перезапуск ноды; выполнение
воспроизводится из лога.

### Итерация 2 — Хранилище History + материализация (~700 строк кода)

- Бэкенд `Igniter::Storage::TimeSeries` (SQLite-based для разработки, TimescaleDB для продакшена)
- Протокол запечатывания `HistorySegment` (сначала стратегия единственного писателя)
- Обнаружение конфликтов записи через предикат `combines:`
- Синхронная материализация для `OLAPPoint source:`
- Применение SLA lag как временного инварианта

Результат: хранилища `History[T]` переживают перезапуск; чтения `OLAPPoint` идут
из материализованного хранилища.

### Итерация 3 — Кластерное распределение (~800 строк кода)

- Протокол карты партиций (расширяет существующую gossip mesh)
- Трансформация fanout компилятора для партиционированных чтений `store`
- Scatter-gather execution runner
- Причинная согласованность через векторные часы при добавлениях в `History[T]`
- Распространение подсказок размещения в планировщик

Результат: OLAP-запросы распределяются по кластеру; добавления в History причинно
упорядочены.

**Общая оценка: ~2000 строк кода** (без существующей инфраструктуры mesh/actor).

## §11 Ключевые Формальные Тождества

Модель персистентности не вводит новых примитивов — она унифицирует существующие
концепции:

```
ExecutionCheckpoint  ≡  HistorySegment[ExecutionState]
Конфликт записи      ≡  Конфликт правил  →  разрешается через combines:
SLA lag              ≡  Временной инвариант на recorded_time
Материализация       ≡  Контракт (source → OLAP-хранилище)
Выбор партиции       ≡  Распределённая таблица поиска  →  как entity
Выбор бэкенда        ≡  Вывод компилятора из типа
```

Язык растёт через унификацию, а не через добавление. Каждая новая концепция
разрешается в существующий словарь. Контрактная модель — универсальный субстрат.
