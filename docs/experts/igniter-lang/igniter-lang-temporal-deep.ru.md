## § 1. Двухвременно́е измерение (Bitemporal)

### § 1.1 Проблема, которую одна ось не решает

Обнаружена ошибка в ценообразовании: последние 3 месяца продукт X стоил 100 у.е.,
хотя должен был стоить 90. Нужно одновременно:

1. Исправить будущее ценообразование — да
2. НЕ изменять суммы прошлых заказов (заморожены, юридически обязательны) — да
3. Ретроактивный анализ: «какой была бы выручка?» — да
4. Журнал аудита: «что система знала на дату D?» — да

Однооосный `History[T]` не может ответить на пункты (3) и (4), не нарушив (2).
Нужны две независимые оси времени.

### § 1.2 Двухвременно́й тип `BiHistory[T]`

```
type BiHistory[T] = [{
  value:          T
  valid_from:     DateTime     -- когда факт стал истинным в бизнес-реальности
  valid_until:    DateTime?    -- когда перестал быть истинным (открытый интервал)
  recorded_from:  DateTime     -- когда система зафиксировала это знание
  recorded_until: DateTime?    -- когда система скорректировала/заменила его
}]
```

Данные образуют 2D-поверхность: действительное время (`vt`) × время транзакции (`tt`).
Точечный запрос `price[vt: t1, tt: t2]` спрашивает: «какова была цена в бизнес-момент
t1, как система понимала это в момент записи t2?»

### § 1.3 Четыре канонических запроса

| Запрос | Смысл | Случай использования |
|--------|-------|---------------------|
| `price[vt: now, tt: now]` | Текущая цена, текущие знания | Отображение в реальном времени |
| `price[vt: order.created_at, tt: order.created_at]` | Цена на момент создания заказа, как тогда понимала система | Замороженная сумма заказа |
| `price[vt: order.created_at, tt: now]` | Цена на момент создания, с исправленными знаниями | Ретроактивный аудит |
| `price[vt: past_date, tt: report_date]` | Цена в прошлом, как было известно на дату отчёта | Регуляторная отчётность |

### § 1.4 Цепочка подтипов

```
T  ⊑  History[T]  ⊑  BiHistory[T]
```

- `T`: константа, без отслеживания (по умолчанию для большинства полей)
- `History[T]`: однооосное, только действительное время (базовая временна́я модель)
- `BiHistory[T]`: полное двухвременно́е отслеживание

Подключается на уровне поля сущности:

```
entity Product {
  name:  String              -- константа (без временно́го отслеживания)
  price: Money @bitemporal   -- полный BiHistory[Money]
  stock: Int   @temporal     -- однооосный History[Int] (только valid time)
}
```

Большинство полей предприятия требуют только `@temporal`. `@bitemporal` нужен
для значений, где ретроактивная корректировка юридически или аналитически важна
(цены, ставки, нормативные параметры).

### § 1.5 Синтаксис двухвременно́го доступа

```
-- Точечный доступ (обе оси)
price[vt: t1, tt: t2]

-- Именованные сокращения
price.current                  -- [vt: now, tt: now]
price.at(t)                    -- [vt: t, tt: now]
price.as_known_at(t)           -- [vt: now, tt: t]
price.as_known_when_valid(t)   -- [vt: t, tt: t]  ← замороженное значение

-- Срез (возвращает BiHistory[T] над областью)
price[vt: period, tt: now]     -- историческое изменение цен, как известно сейчас
price[vt: now, tt: past_year]  -- как менялись наши знания о текущей цене
```

### § 1.6 Двухвременны́е инварианты

```
invariant FrozenOrderTotal : Order {
  -- Заморожено с использованием знаний на момент создания: юридически обязательно
  total == OrderTotal {
    order: self,
    as_of:           created_at,     -- vt: когда был создан заказ
    knowledge_as_of: created_at      -- tt: что мы знали тогда
  }.total
}

invariant AuditableOrderTotal : Order {
  -- Ретроактивный анализ: какой должна была быть сумма
  audited_total == OrderTotal {
    order: self,
    as_of:           created_at,     -- vt: тот же бизнес-момент
    knowledge_as_of: DateTime.now()  -- tt: исправленные знания
  }.total
  -- audited_total != total допустимо и ожидаемо после ценовой коррекции
}
```

Компилятор проверяет, что `total` (замороженный) и `audited_total` (скорректированный)
вычисляются из разных осей `tt` — никогда не путаются.

### § 1.7 Двухвременны́е корректировки

```
-- Записать ценовую корректировку (задним числом)
effect :correct_price =
  PriceCorrection {
    product:       product
    correct_price: 90_00    -- цена, которая должна была быть
    valid_from:    3.months.ago
    valid_until:   Date.today
    reason:        :data_entry_error
  }
  @audit
```

Корректировка вставляет новую запись в измерение `tt`: все будущие запросы с
`tt: now` видят исправленную цену. Запросы с `tt: до_корректировки` видят
оригинальную. `FrozenOrderTotal` (использующий `tt: created_at`) не затронут.

## § 2. Синтез временны́х правил

### § 2.1 Проблема

Бизнес-аналитик говорит: *«Выручка в декабре должна быть на 15% выше ноябрьской»*.
Он хочет правило ценообразования, а не SQL-отчёт.

Синтез правила, управляемый целью: дано временно́е целевое свойство и исторические
данные — синтезировать объявление `rule`, достигающее цели при соблюдении всех
объявленных инвариантов.

### § 2.2 DSL синтеза

```
synthesize rule for OrderManagement {
  goal:
    PeriodReport { period: december }.actual_revenue >=
    PeriodReport { period: november }.actual_revenue * 1.15

  template: rule SynthesizedPromotion : Product {
    applies: { months: [:december] }
    compute: fn(product) -> Float = ?rate    -- ? обозначает цель синтеза
    priority: 60
    combines: :override
  }

  constraints: [PriceFloor, CustomerFairness, MaxDiscountRate(0.30)]

  using: orders |> select { |o| o.created_at.in?(last_year) }
}
```

Синтезатор находит наименьший `?rate` (минимально необходимое изменение) такой,
чтобы цель выполнялась на историческом наборе данных с соблюдением ограничений.

### § 2.3 Сведение к задаче линейного программирования

Для линейных правил ценообразования `effective_price = base × (1 - rate)`:

Пусть `n` — количество заказов в периоде синтеза.
Пусть `tᵢ` — сумма заказа `i` по базовым ценам.
Пусть `r` — ставка скидки (переменная синтеза).

Целевое ограничение:
```
Σᵢ tᵢ × (1 - r) ≥ 1.15 × Σᵢ tᵢ_november
```

Если аналитик хочет *более высокую* выручку, синтезатор может найти **повышение
цены** (`r < 0` = наценка), а не скидку. Это явно отражается в результате.

Ограничение ценового порога:
```
∀ product: base_price × (1 - r) ≥ cost_price × 1.10
→  r ≤ min_across_products(1 - cost_margin)
```

Это **задача линейного программирования** — PTIME через симплекс-метод.
Результат: либо реализуемая ставка `r*`, либо доказательство невыполнимости.

### § 2.4 Результат синтеза

```
-- Вывод синтезатора (предлагаемое правило, не активировано):
rule SynthesizedPromotion_2026_12 : Product {
  applies: { months: [:december] }
  compute: fn(product) -> Float = -0.14    -- повышение цены на 14%
  priority: 60
  combines: :override
  @generated(
    goal:       "december_revenue >= 1.15 * november_revenue",
    confidence: 0.92,                      -- на основе исторической дисперсии
    orders_used: 847,
    feasibility: :satisfied
  )
}

-- Предварительный расчёт влияния (WhatIfAnalysis запускается автоматически):
-- Ожидаемая дельта выручки: +124,000 у.е.
-- Затронуто заказов: 847 (100%)
-- Среднее изменение заказа: +146 у.е.
-- PriceFloor: соблюдён для всех 234 продуктов
-- CustomerFairness: ПРЕДУПРЕЖДЕНИЕ — 12 продуктов превышают порог max_increase
```

Синтезатор выявляет предупреждения до активации. Аналитик может скорректировать
шаблон и запустить синтез повторно.

### § 2.5 Многопеременный синтез

Для более выразительных шаблонов с несколькими `?` целями:

```
template: rule TieredPromotion : Product {
  applies: { months: [:december] }
  compute: fn(product) -> Float =
    if product.category == :premium then ?premium_rate
    else if product.margin > 0.40  then ?high_margin_rate
    else                                ?standard_rate
}
```

Каждый `?` — независимая переменная ЗЛП. Система находит комбинацию, минимизирующую
суммарное изменение цен при выполнении цели по выручке.

## § 3. Обнаружение причинных цепей

### § 3.1 Проблема

Два правила кажутся независимыми, но связаны через общее значение:

```
rule VolumeDiscount : Order {
  applies: { when: order.total > 1_000 }
  compute: fn(order) -> Float = 0.10
}

rule SeasonalDiscount : Product {
  applies: { months: [:december] }
  compute: fn(product) -> Float = 0.20
}
```

В декабре: `SeasonalDiscount` снижает цены → `order.total` падает ниже `1000` →
`VolumeDiscount.applies` становится ложным → `order.total` растёт →
`VolumeDiscount.applies` снова истинно → осцилляция.

Это скрытый **цикл обратной связи** в наборе правил.

### § 3.2 Граф зависимостей правил

Компилятор строит **Граф зависимостей правил (ГЗП)**, где:
- Узлы = правила и временны́е значения, которые они читают/пишут
- Ребро `r → v`: правило `r` пишет (вычисляет) значение `v`
- Ребро `v → r`: предикат `applies:` правила `r` читает значение `v`

```
SeasonalDiscount → Product.effective_price → Order.total → VolumeDiscount.applies
VolumeDiscount   → Order.effective_total   → Order.total   (обратная связь!)
```

Цикл: `Order.total → VolumeDiscount → Order.total` — обнаружен как цикл в ГЗП.

### § 3.3 Классификация циклов

| Тип цикла | Условие | Сходимость |
|-----------|---------|-----------|
| **Монотонно убывающий** | Каждая итерация только снижает значения | Гарантирована |
| **Монотонно возрастающий** | Каждая итерация только повышает значения | Гарантирована |
| **Ограниченная осцилляция** | Цикл проходит конечное число состояний | Сходится к периодической орбите |
| **Неограниченный** | Значения могут расти без предела | Расходимость — ОШИБКА |

Для правил скидок (ставка ∈ `[0, 1]`): любой цикл через цены монотонно убывает
→ сходится за O(log(price/floor)) итераций.

### § 3.4 Диагностика компилятора и разрешение

```
ПРЕДУПРЕЖДЕНИЕ: причинный цикл в оценке правил
  Цикл: SeasonalDiscount → Order.total → VolumeDiscount → Order.total
  Тип:  монотонно убывающий (только скидки)
  Сходимость: гарантирована в 2 итерации

  Варианты разрешения:
  (а) Объявить явный порядок оценки:
        SeasonalDiscount >> VolumeDiscount
      Эффект: VolumeDiscount видит сумму после сезонной скидки

  (б) Заморозить промежуточное значение:
        rule VolumeDiscount { applies: { when: order.pre_rule_total > 1_000 } }
      Эффект: VolumeDiscount проверяет исходную сумму (до применения правил)

  (в) Принять сходимость (добавить аннотацию @converge):
        rule VolumeDiscount { ... @converge(max_iterations: 3) }
      Эффект: среда выполнения итерирует до стабилизации
```

### § 3.5 Стратегии разрешения

**Стратегия А — явный порядок** `>>`

```
system OrderManagement {
  rules: [SeasonalDiscount >> VolumeDiscount, CustomerLoyalty, PriceFloor]
}
```

ГЗП становится DAG. Самый предсказуемый вариант.

**Стратегия Б — снимок**

```
rule VolumeDiscount : Order {
  applies: { when: order.base_total > 1_000 }    -- base_total = значение до правил
  compute: fn(order) -> Float = 0.10
}
```

`base_total` — узел с аннотацией `@snapshot`: его значение заморожено до
применения любых правил в текущем контексте оценки.

**Стратегия В — аннотация сходимости**

```
rule VolumeDiscount : Order {
  applies: { when: order.total > 1_000 }
  compute: fn(order) -> Float = 0.10
  @converge(max_iterations: 5, tolerance: 0.01)
}
```

Среда выполнения итерирует набор правил до тех пор, пока изменение между
итерациями не станет меньше `tolerance` или не будет достигнуто `max_iterations`
(ошибка если не сошлось).

## § 4. Вероятностные временны́е правила

### § 4.1 Правила с неопределённой применимостью

Некоторые правила должны срабатывать на основе вероятностных условий — акция,
ожидающая одобрения менеджера, сезонный прогноз, A/B-эксперимент.

```
rule AnticipatedPromotion : Product {
  applies: { when: ~approval_likelihood > 0.8 }    -- вероятностный предикат
  compute: fn(product) -> Float = 0.15
  priority: 70
  @approximate(confidence: 0.9)
}
```

`~approval_likelihood` — это `~Float` из модели предвычисления
([igniter-lang-precomp.md](igniter-lang-precomp.md)): несёт
`{ value: 0.85, lo: 0.72, hi: 0.94, confidence: 0.9 }`.

Сравнение `~Float > 0.8` в `applies:` возвращает `Bool | Uncertain` —
трёхзначный результат из §1.5 спецификации языка.

### § 4.2 Двухуровневая временна́я оценка

Модель предвычисления напрямую отображается на оценку правил:

| Уровень | Правило применяется? | Тип результата | Использование |
|---------|---------------------|---------------|---------------|
| **Приближённый** | Определено вероятностно | `~T` | Планирование, прогнозирование |
| **Точный** | Правило активировано или нет (Bool) | `T` | Зафиксированная сумма заказа, счёт |

```
contract OrderTotal {
  in order: Order
  in as_of: DateTime = order.created_at

  -- Приближённо: включаем вероятные правила
  compute :expected_total: ~Money =
    apply_rules(order, ActiveRules[as_of] ++ ProbableRules[as_of])
    @approximate(confidence: 0.85)

  -- Точно: только подтверждённые правила
  compute :committed_total: Money =
    apply_rules(order, ActiveRules[as_of] |> filter { .confirmed? })
    @exact

  out total:    Money  = committed_total   -- юридическое обязательство
  out expected: ~Money = expected_total    -- плановая цифра
}
```

### § 4.3 Модель точности, управляемая решением

Из [igniter-lang-precomp.md](igniter-lang-precomp.md): нижестоящие потребители
объявляют требования к точности, запуская точное вычисление только при необходимости.

```
-- Планирование запасов: достаточно приближения
compute :planned_revenue: ~Money =
  orders |> sum { .expected }
  @tolerance(0.05)       -- допустимая погрешность 5%

-- Выставление счёта: должно быть точным
compute :invoice_total: Money =
  order.total            -- форсирует путь committed_total
  @exact
```

### § 4.4 Вероятностные временны́е инварианты

```
invariant ProbabilisticConsistency : Order {
  -- Зафиксированная сумма должна быть в доверительном интервале прогноза
  total in expected.lo..expected.hi
}

invariant ConfidenceThreshold : OrderBatch {
  -- Не фиксировать партию, если агрегированная достоверность недостаточна
  items.avg { .expected.confidence } >= 0.85
}
```

## § 5. Распределённое время

### § 5.1 Проблема

В распределённой системе с узлами A и B:
- Узел A обрабатывает заказ в 10:00:00 по настенным часам
- Узел B обрабатывает событие в 09:59:58 (2 секунды позади)
- `as_of: DateTime.now()` возвращает разные значения на каждом узле
- Один и тот же заказ может оцениваться с несогласованными наборами правил

Настенные часы `DateTime.now()` небезопасны как `as_of` в распределённом контексте.

### § 5.2 Логические часы как `as_of`

**Метки времени Лампорта (1978)**: каждое событие получает целочисленную метку `L`.
Если событие A причинно предшествует B, то `L(A) < L(B)`.

```
type LogicalTimestamp = {
  node_id:  Id
  sequence: Int
}
```

Параметр `as_of` становится `LogicalTimestamp`:

```
in as_of: LogicalTimestamp = LogicalClock.current()
```

`price[as_of: lt]` разрешает историю до значения, видимого в логическое время
`lt`: все записи с `recorded_from ≤ lt` видимы.

### § 5.3 Причинный `as_of`

Конструкция `await` должна нести логическое время отправителя, а не получателя:

```
await :payment_confirmed,
  event:  :payment_received,
  as_of:  event.sender_logical_time    -- причинное время источника события
```

Без этого: событие, пришедшее с задержкой из-за сети, оценивается с `now`,
которое позже его причинного времени — система «видит будущее» относительно
контекста события.

### § 5.4 Векторные часы и уровни согласованности

**Векторные часы** отслеживают причинную историю по узлам:
`VC = { node_1: 5, node_2: 3, node_3: 7 }`

Это даёт частичный порядок: параллельные события несравнимы.

| Уровень согласованности | Семантика `as_of` | Гарантия |
|------------------------|------------------|---------|
| **Причинный** | Векторные часы последней записи | Видит все причинно предшествующие записи |
| **Монотонное чтение** | Максимум увиденных меток | Никогда не идёт назад во времени |
| **Сессионный** | Векторные часы записей этой сессии | Видит собственные записи |
| **Финальный** | `DateTime.now()` на локальном узле | Нет гарантий порядка |

### § 5.5 Временны́е инварианты при распределённом времени

Инвариант `FrozenOrderTotal` усиливается для распределённых контекстов:

```
invariant FrozenOrderTotal : Order {
  -- Заморожено в причинное время создания заказа (не настенные часы)
  total == OrderTotal {
    order:       self,
    as_of:       created_at_logical,     -- LogicalTimestamp
    consistency: :causal
  }.total
}

invariant CausalRuleApplication : Order {
  -- Все правила, применённые к заказу, были активны до его создания (причинно)
  applied_rules.all? { |r|
    r.logical_valid_from <= created_at_logical
  }
}
```

## § 6. Единая временна́я модель

### § 6.1 Полная иерархия типов

```
T                         -- константа (без отслеживания)
  ⊑ History[T]            -- однооосная: действительное время
      ⊑ BiHistory[T]      -- двуосная: valid time × transaction time
          ⊑ ~BiHistory[T]  -- приближённая + двухвременна́я (полная модель)
```

### § 6.2 Параметры временно́го контракта

```
contract C {
  in as_of:           DateTime | LogicalTimestamp = LogicalClock.current()
  in knowledge_as_of: DateTime                    = DateTime.now()
  in consistency:     Symbol                      = :causal
  ...
}
```

Три независимые оси управления:
- `as_of`: **когда** оценивать (ось vt)
- `knowledge_as_of`: **что мы знали** при оценке (ось tt, для bitemporal)
- `consistency`: **насколько строго** соблюдать порядок (для распределённых систем)

### § 6.3 Конвейер оценки набора правил

```
1. СОБРАТЬ   правила где applies(as_of, context) → Bool | ~Bool
2. РАЗРЕШИТЬ вероятностные applies: ~Bool → Bool (или оставить как ~)
3. УПОРЯДОЧИТЬ по priority + явная >> композиция
4. ПРОВЕРИТЬ ГЗП на причинные циклы; применить стратегию разрешения
5. ПРИМЕНИТЬ упорядоченная свёртка со стратегиями combines
6. СОЙТИСЬ  если аннотация @converge: итерировать до стабилизации
7. ВЕРИФИЦИРОВАТЬ временны́е инварианты на выходе
```

Каждый шаг — PTIME для конечных наборов правил роговского фрагмента.

### § 6.4 Свойства корректности

| Свойство | Гарантия |
|----------|---------|
| **Временна́я детерминированность** | Одни входы + один `as_of` + один набор правил → один выход |
| **Причинная согласованность** | Правила применяются только с причинно предшествующими знаниями |
| **Целостность замороженных значений** | `@bitemporal` + `tt: created_at` → значения не меняются задним числом |
| **Безопасность причинных циклов** | Монотонные циклы сходятся; неограниченные — ошибка компиляции |
| **Приближённая корректность** | Результаты `~T` в пределах объявленных доверительных интервалов |
| **Распределённая монотонность** | `as_of` монотонно возрастает в рамках сессии |

## § 7. Пересмотренная дорожная карта POC

### § 7.1 Трёхитерационный план

**Итерация 1 — Однооосное временно́е измерение + правила (фундамент)**
- `History[T]`, оператор `[t]`, аннотация `@temporal`
- Объявления `rule` с `applies:` / `compute:` / `priority:`
- Ортогональность: регистрация правил в системе без изменения контрактов
- Временны́е инварианты: `FrozenOrderTotal`, `TemporalCoverage`
- Компилятор: обнаружение временны́х узлов, инъекция `as_of`, свёртка разрешения правил
- Дымовой тест: `FulfillOrder` + правило `WeekendPrice`

**Итерация 2 — Причинные цепи + bitemporal + синтез**
- Построение ГЗП и обнаружение циклов
- Стратегии `@converge` и `@snapshot`
- `BiHistory[T]`, синтаксис `[vt: t, tt: t]`
- DSL `synthesize rule` + интеграция ЗЛП-решателя (симплекс)
- Расширенный дымовой тест: `PeriodReport` + `WhatIfAnalysis`

**Итерация 3 — Вероятностные правила + распределённое время**
- `~applies` с правилами `@approximate`
- Двухуровневая оценка (ожидаемое vs. зафиксированное)
- `LogicalTimestamp` как тип `as_of`
- Инвариант `CausalRuleApplication`
- Дымовой тест распределённости: многоузловая обработка заказов
