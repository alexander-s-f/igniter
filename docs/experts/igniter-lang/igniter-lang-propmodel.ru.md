# Контрактный язык Igniter — Модели свойств и синтез алгоритмов

Дата: 2026-04-27.
Перспектива: теория языков программирования; представление знаний; спецификационно-управляемый синтез.
Тема: Слой моделей свойств над графом контракта — объявлять типы объектов и отношения между свойствами, автоматически выводить алгоритмы; LLM как автор моделей.

*Статус: экспериментальное исследование — теоретическое предложение, не feature track.*
*Продолжает: [igniter-lang-precomp.md](igniter-lang-precomp.md) — вероятностное предвычисление.*
*Каноничная версия: [igniter-lang-propmodel.md](igniter-lang-propmodel.md)*

---

## 1. Фундаментальная проблема

В каждой программе-workflow происходят две вещи одновременно:

1. **Указание того, как должен выглядеть мир** для достижения цели
2. **Реализация того, как проверить и произвести это состояние мира**

Сегодня эти две вещи написаны на одном языке, на одном уровне, перемешаны вместе. Результат: *спецификация* (что) невидима — она неявно живёт в реализации (как). Чтобы понять что означает код, нужно реконструировать спецификацию, читая код.

Предложение: **разделить их на отдельные уровни**. Объявить спецификацию явно. Вывести реализацию из неё.

## 2. Три уровня

### 2.1 Уровень 0 — Онтология (что за объекты существуют и какие свойства у них есть)

```
type Product {
  available: Bool
  stock:     Int where stock >= 0
  price:     Money where price > 0
  category:  Enum[:physical, :digital]
}

type Marketing {
  campaign_active:  Bool
  discount_applied: Bool
  discount_rate:    Float where 0.0 <= discount_rate <= 1.0
}

type Manager {
  approved:         Bool
  approval_limit:   Money
}

type Order {
  status: Enum[:pending, :processing, :complete, :cancelled]
  total:  Money
}
```

Это типизированная **онтология**: объекты, их свойства и ограничения на значения свойств (уточняющие типы).

### 2.2 Уровень 1 — Модель свойств (какие отношения существуют между свойствами)

```
model OrderFulfillment {
  goal: order.status = :complete

  requires:
    product.available   = true
    product.stock       > 0
    manager.approved    = true
    order.total         <= manager.approval_limit

  consistency:
    marketing.campaign_active → marketing.discount_applied = true

  derives:
    order.total = product.price * (1.0 - discount_rate_if_any)
}
```

Это **спецификация свойств**: что должно быть истинным об объектах для достижения цели, и какие отношения существуют между свойствами. Никакой реализации. Никакого «как».

### 2.3 Уровень 2 — Алгоритм (выведенный или написанный)

Из модели система синтезирует:

```
contract FulfillOrder {
  in product:   Product
  in marketing: Marketing
  in manager:   Manager

  // выведен из правила consistency модели
  guard :discount_consistent {
    when marketing.campaign_active
    then marketing.discount_applied must be true
  }

  // выведен из requires модели
  guard :fulfillable {
    product.available = true
    product.stock > 0
    manager.approved = true
  }

  // выведен из derives модели
  compute :effective_price =
    product.price * (1.0 - marketing.discount_rate
                          .if_present(when: marketing.discount_applied))

  compute :within_limit = effective_price <= manager.approval_limit

  // выведен из goal + requires + derives модели
  compute :status =
    fulfillable && within_limit ? :complete : :pending

  out result = Order { status: status, total: effective_price }
}
```

Алгоритм не пишется программистом — он **синтезируется компилятором** из модели свойств. Программист пишет только уровень 0 (типы) и уровень 1 (модель). Уровень 2 выводится.

### 2.4 Уровень 3 — Выполнение с предвычислением

Синтезированный контракт выполняется с использованием рантайма из предыдущего репорта:
- `product.available` — дешёвая булева проверка свойства → предвычислить первой
- `manager.approval_limit` — запрос в БД → вычислять только если продукт доступен
- `effective_price` → точно или приближённо в зависимости от требований потребителя

Четыре уровня соединяются в единый пайплайн: **онтология → модель → контракт → выполнение**.

## 3. Что синтез может и не может

### 3.1 Автоматически синтезируемое

| Шаблон | Объявление в модели | Синтезированная форма |
|--------|--------------------|-----------------------|
| Проверка свойства | `requires: x.p = true` | `guard :x { x.p = true }` |
| Условная согласованность | `a.p → b.q = true` | `guard` с `when:` |
| Вывод свойства | `derives: x = f(a, b)` | `compute :x = f(a, b)` |
| Цель из свойств | `goal: x.status = :complete` | `compute :status` + `out` |
| Агрегация | `derives: total = sum(items.price)` | `aggregate :total` |
| Условная цель | `requires: if a.p then b.q` | узел `branch` |

### 3.2 За пределами синтеза — требуются функции

Некоторые вычисления не могут быть выведены из отношений между свойствами. Они требуют явных определений функций:

```
// Это не синтезируемо — это настоящая трансформация
fn shipping_cost(product: Product, destination: Address) -> Money = {
  base_rate(destination.zone) * product.weight_kg * urgency_multiplier
}
```

Определения функций — уровень 0.5 — между онтологией и моделью. Они определяют как вычисляются свойства, а не что должно быть о них истинным.

### 3.3 Граница синтеза

```
Декларативная область (синтезируемо):
  ├── проверки свойств (requires)
  ├── условная согласованность (if P then Q)
  ├── арифметические выводы (total = price * quantity)
  └── условия цели (status = :complete)

Вычислительная область (требует fn):
  ├── нетривиальные трансформации (алгоритмы маршрутизации)
  ├── вызовы внешних систем (запросы в БД, HTTP)
  ├── предметно-специфические вычисления (формула стоимости доставки)
  └── обученные модели (ML-предсказания)
```

Гипотеза: **для большинства бизнес-workflow декларативная область покрывает ≥70% узлов**. Вычислительная область — меньшинство — и это именно то, что сегодня реализуют классы-исполнители. Язык делает это разделение явным.

## 4. Теоретические основания (указатель к следующему документу)

Уровень модели свойств соединяется с хорошо установленной формальной теорией. Этот раздел отображает конструкты; следующий документ в серии строго разрабатывает теоретические основания.

### 4.1 Ситуационное исчисление (МакКарти, 1963)

Ситуационное исчисление формализует, как **действия изменяют свойства объектов** во времени. «Ситуация» — снимок всех значений свойств. Действия переводят между ситуациями.

Модель свойств — ограниченное ситуационное исчисление: «ситуация» — текущий набор свойств объектов, «действие» — выполнение контракта, «цель» — целевые значения свойств.

### 4.2 Дескриптивные логики / OWL

Уровень онтологии (уровень 0) — дескриптивная логика: типизированная схема с ограничениями свойств и объявлениями отношений. OWL (Web Ontology Language) — наиболее широко используемая дескриптивная логика, применяемая в семантической сети и графах знаний.

Модель свойств (уровень 1) соответствует аксиомам OWL: необходимые и достаточные условия принадлежности классу, ограничения свойств, ограничения согласованности.

### 4.3 Логика Хоара

Каждый синтезированный контракт соответствует тройке Хоара:

```
{ предусловие(A, B, C) }  контракт  { постусловие(X) }
```

Объявления `requires` — предусловие. Объявление `goal` — постусловие. Задача синтеза: найти программу, которая переводит состояние предусловия в состояние постусловия.

### 4.4 Уточняющие типы

Ограничения свойств (`stock >= 0`, `price > 0`) — **уточняющие типы**: типы, дополненные логическими предикатами. Языки LiquidHaskell и Dafny используют уточняющие типы для спецификационно-управляемой верификации.

В контрактном языке уточняющие типы распространяются через граф контракта: если `compute :total = sum(items.price)` и `price > 0` для всех элементов, компилятор может статически вывести `total > 0`.

### 4.5 Планирование (STRIPS / PDDL)

Задача синтеза — имея свойства объектов и целевое состояние свойств, найти контракт, достигающий цели — является экземпляром **автоматического планирования**. PDDL (Planning Domain Definition Language) — стандартный формализм.

Критическое отличие: PDDL-планирование последовательное (найти последовательность действий), тогда как синтез контрактов — граф потоков данных. Правильный формализм — **иерархические сети задач (HTN-планирование)**, расширенные на DAG-структуры.

## 5. LLM как автор моделей

### 5.1 Почему LLM лучше работает на уровне 1

LLM обучены на естественном языке, который выражает знание о мире преимущественно как **отношения между свойствами**: «заказ завершён, когда товар доступен, менеджер одобрил, и оплата прошла». Это словарь уровня 1, а не код реализации.

Когда LLM напрямую генерирует код уровня 3, он должен:
- Придумывать детали реализации, которых нет в обучающих данных
- Делать неявные предположения о структурах данных
- Производить невалидируемый вывод

Когда LLM генерирует модели уровня 1, он:
- Выражает то, что реально знает (отношения свойств)
- Производит формально верифицируемый вывод
- Остаётся в границах своей компетенции

### 5.2 Контракт LLM-компилятор

```
Ответственность LLM:       корректно выразить отношения свойств
                            (goal, requires, consistency, derives)

Ответственность компилятора: синтезировать корректный контракт
                              верифицировать согласованность модели
                              отклонять противоречивые или неполные модели

Ответственность рантайма:  выполнять с предвычислением,
                            обратным распространением точности
```

Это **разделение труда**, где каждая сторона делает то, в чём сильна. LLM вносит знание. Компилятор обеспечивает корректность. Рантайм обеспечивает эффективность.

### 5.3 Верифицируемый вывод LLM

Критическое свойство: модель уровня 1 **формально проверяема** до выполнения.

Компилятор может верифицировать:
- **Согласованность**: ни одно свойство не требуется одновременно истинным и ложным
- **Полнота**: каждый путь к цели имеет все необходимые объявленные свойства
- **Завершаемость**: синтез завершается (свойство модели DAG — гарантировано)
- **Достижимость**: цель достижима из объявленных входов

Если LLM производит несогласованную модель, компилятор отклоняет её с конкретной ошибкой — не рантайм-краш, не неверный вывод. Это **верификация вывода LLM на семантическом уровне**.

### 5.4 Итеративное уточнение модели

Цикл LLM-компилятор:

```
1. LLM генерирует модель уровня 1 из описания на естественном языке
2. Компилятор проверяет согласованность и полноту
3. При ошибках: компилятор возвращает структурированную обратную связь LLM
4. LLM уточняет модель
5. Повторять до принятия модели
6. Компилятор синтезирует контракт
7. Контракт выполняется с полным провенансом и интроспекцией
```

Этот цикл имеет формальное условие остановки (принятие компилятором), в отличие от текущей генерации кода LLM, у которой формального условия остановки нет.

## 6. Информационная плотность на уровне модели

### 6.1 SIR на каждом уровне

Расширение метрики Semantic Information Ratio из первого документа:

| Уровень | Описание | Оценочный SIR |
|---------|----------|---------------|
| Ruby DSL + классы-исполнители | Текущее состояние | 1.0× (базовый) |
| Контрактно-нативный (уровень 2) | Предыдущие документы | ~2× |
| Модель свойств (уровень 1) | Этот документ | ~4–6× (гипотеза) |

Уровень модели плотнее потому что:
- Нет деталей реализации
- Нет структурного boilerplate
- Семантическое содержание максимально сконцентрировано
- Одна модель генерирует несколько контрактов для разных стратегий выполнения

### 6.2 Аргумент Колмогорова

Модель свойств — это **минимальное описание** намерения workflow. Нельзя описать намерение «выполнение заказа» в меньшем числе концептов, чем: участвующие объекты, какими свойствами они должны обладать, и какова цель. Любая дополнительная информация — деталь реализации.

Это теоретико-информационная нижняя граница описания программы: модель свойств приближается к Минимальной Длине Описания (MDL) спецификации workflow.

## 7. Архитектура полной системы

Полная четырёхуровневая архитектура с новым слоем:

```
┌─────────────────────────────────────────────────────┐
│  Уровень 0: Онтология                               │
│  Типы объектов + ограничения свойств + определения fn│
│  Автор: разработчик (или LLM)                       │
│  Инструмент: проверка типов + валидатор ограничений │
└────────────────────────┬────────────────────────────┘
                         │ объявляет
┌────────────────────────▼────────────────────────────┐
│  Уровень 1: Модель свойств                          │
│  goal + requires + consistency + derives            │
│  Автор: разработчик ИЛИ LLM                        │
│  Инструмент: model checker (согласованность, полнота)│
└────────────────────────┬────────────────────────────┘
                         │ синтезирует
┌────────────────────────▼────────────────────────────┐
│  Уровень 2: Граф контракта                          │
│  Узлы input/compute/guard/branch/output             │
│  Автор: синтезатор (или разработчик для fn-узлов)   │
│  Инструмент: GraphCompiler + вывод типов            │
└────────────────────────┬────────────────────────────┘
                         │ выполняется через
┌────────────────────────▼────────────────────────────┐
│  Уровень 3: Рантайм с предвычислением               │
│  Двухуровневое выполнение, обратное распростр. точн.│
│  Автор: рантайм (автоматически)                     │
│  Инструмент: Resolver + приближённый evaluator      │
└─────────────────────────────────────────────────────┘
```

## 8. Открытые вопросы

### 8.1 Полнота синтеза

Для каких классов моделей свойств корректный контракт всегда существует? Когда синтез завершается неудачей, и какую обратную связь должен давать компилятор?

Формальный вопрос: **для какого фрагмента логики первого порядка над отношениями свойств синтез разрешим?**

Предварительный ответ (из теории планирования): если модель свойств монотонна (добавление фактов никогда не убирает цели), синтез разрешим за полиномиальное время. Немонотонные модели (например, отмена, отзыв) требуют более мощного аппарата.

### 8.2 Неоднозначность модели

Модель свойств может быть реализуема несколькими контрактами. Какой из них должен выбрать синтезатор? Критерии выбора:
- Минимальная стоимость (меньше дорогостоящих узлов)
- Максимальный параллелизм (больше конкурентных узлов)
- Максимальная устойчивость к отсутствующим данным (наиболее graceful degradation)

Это задача **оптимизации синтеза** — не просто «найди контракт», а «найди лучший контракт».

### 8.3 Эволюция свойств

Когда онтология меняется (например, `Manager` получает свойство `delegation_level`), какие модели свойств затронуты? Какие синтезированные контракты нужно перегенерировать?

Это **анализ воздействия изменений** на уровне модели — формально разрешимый, потому что модель явна.

### 8.4 Вероятностные модели свойств

В сочетании с документом о предвычислении: что если свойства вероятностны?

```
model FraudDetection {
  goal: transaction.flagged = true
  requires:
    location.anomaly_score > 0.8    // вероятностное свойство
    amount.deviation > 3.0          // статистическое свойство
    velocity.rate > threshold       // приближённое
}
```

Вероятностная модель свойств синтезирует контракт с приближёнными узлами, используя механизм предвычисления из предыдущего документа. Это объединяет оба исследовательских направления.

## Резюме

Слой модели свойств добавляет declaration-first уровень над графом контракта:
- **Онтология** (уровень 0): какие объекты и свойства существуют
- **Модель свойств** (уровень 1): какие отношения между свойствами достигают цели
- **Контракт** (уровень 2): синтезированный из модели
- **Выполнение** (уровень 3): рантайм с предвычислением

Уровень модели — естественный интерфейс для LLM: он выражает знание о мире как отношения свойств — именно то, на чём обучены LLM — в форме, которая формально верифицируема и автоматически компилируема.

Информационная плотность на уровне модели приближается к теоретической нижней границе MDL для спецификации workflow: никаких избыточных деталей реализации, только семантическое содержание.

Ключевой открытый вопрос — граница синтеза: для каких моделей свойств автоматический синтез контракта разрешим и практически эффективен? Следующий документ в серии обращается к этому через строгие теоретические основания — ситуационное исчисление, дескриптивные логики, логику Хоара и теорию планирования.
