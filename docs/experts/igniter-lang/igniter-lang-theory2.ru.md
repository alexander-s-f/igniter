## Преамбула: Что нужно доказать

Предложение о моделях свойств в предыдущем документе сделало три утверждения без доказательства:

1. **Синтез разрешим** — компилятор всегда может определить, существует ли корректный контракт для данной модели
2. **Синтез корректен (soundness)** — если компилятор производит контракт, тот гарантированно правилен относительно модели
3. **Синтез полон (completeness)** — если корректный контракт существует, синтезатор его найдёт

Это не опциональные свойства. Без разрешимости компилятор может зациклиться. Без корректности синтезированные контракты могут ошибаться. Без полноты синтезатор молча пропускает корректные решения.

Этот документ доказывает все три — в точно определённом логическом фрагменте — и показывает, какие расширения нарушают какие гарантии и почему.

## 1. Логический фрагмент

### 1.1 Проблема полной логики первого порядка

Полная логика первого порядка (FOL) выразительна, но неразрешима (Чёрч, 1936; Тьюринг, 1936). Если бы модели свойств были произвольными формулами FOL, синтез был бы неразрешим в общем случае — компилятор не мог бы гарантировать завершение.

Нужен **ограниченный фрагмент**, который:
- Достаточно выразителен для описания реальных бизнес-моделей
- Достаточно слаб для разрешимого синтеза

### 1.2 Хорновы дизъюнкты и Datalog

**Хорнов дизъюнкт** — это формула FOL вида:

```
B₁ ∧ B₂ ∧ ... ∧ Bₙ → H
```

где каждый `Bᵢ` и `H` — атомарная формула (предикат, применённый к термам), а `H` — голова (не более одного положительного литерала в консеквенте).

**Datalog** — язык хорновых дизъюнктов над фиксированным конечным доменом, без функциональных символов. Его ключевые свойства:

- **Разрешим**: вычисление всегда завершается
- **PTIME-полный**: вычисление занимает время, полиномиальное по размеру входа
- **Конфлюэнтен**: результат не зависит от порядка вычисления
- **Монотонен**: добавление входных фактов может только добавлять выведенные факты, никогда не убирать их

**Теорема 1.1** (Характеризация через Datalog): Модель свойств M находится в *хорновом фрагменте* тогда и только тогда, когда каждое правило M может быть записано как хорнов дизъюнкт Datalog над доменом свойств.

Хорнов фрагмент покрывает:
- `requires: x.p = true` → `holds(p, x) :- given(x)`
- `derives: y = f(a, b)` → `val(y, V) :- val(a, A), val(b, B), apply(f, A, B, V)`
- `consistency: a.p → b.q = true` → `holds(q, b) :- holds(p, a)`
- `goal: x.status = :complete` → `complete(x) :- holds(status_complete, x)`

Хорнов фрагмент **не** покрывает:
- Неограниченное отрицание: `NOT product.recalled` (требует стратификации — см. §6)
- Дизъюнктивные требования: `a.p OR b.q` (требует дизъюнктивного Datalog — неразрешим в общем случае)
- Арифметические агрегации с неограниченными кванторами (требует ограничения до стратифицированной агрегации)

### 1.3 Фрагмент синтеза

**Определение**: Модель свойств M находится в *фрагменте синтеза*, если:
1. Все правила — хорновы дизъюнкты (нет неограниченной дизъюнкции)
2. Домен каждого типа свойства конечен или ограничен объявленными ограничениями
3. Модель **стратифицируема**: существует функция ранжирования `rank: предикаты → ℕ` такая, что для каждого правила `B₁ ∧ ... ∧ Bₙ → H`, `rank(H) > rank(Bᵢ)` для всех `i`

Стратифицируемость гарантирует отсутствие циклических зависимостей — модель свойств сама является DAG.

## 2. Разрешимость

### 2.1 Задача синтеза

**Определение** (задача синтеза): Дана модель свойств M в фрагменте синтеза и набор входных типов I, найти контракт C такой, что:

```
∀ входов v ∈ ⟦I⟧ : ⟦C⟧(v) удовлетворяет goal(M)
```

где `⟦I⟧` — множество всех значений входных типов, а `⟦C⟧` — денотация C.

**Теорема 2.1** (Разрешимость): Задача синтеза для фрагмента синтеза разрешима, и алгоритм синтеза работает за время O(|M|² · |dom|^k), где |dom| — максимальный размер домена, k — максимальная арность правил.

**Доказательство (набросок)**:

Алгоритм синтеза — восходящее вычисление Datalog (Уллман, 1988):

```
Алгоритм Synthesize(M, I):
  S ← ground(I)                  // инициализировать входными фактами
  repeat
    S ← S ∪ { H | (B₁∧...∧Bₙ→H) ∈ M, {B₁,...,Bₙ} ⊆ S }
  until нет новых фактов           // достигнута фиксированная точка
  if goal(M) ⊆ S:
    return derivation_tree(goal(M), S, M)   // контракт
  else:
    return UNREACHABLE
```

Завершение гарантировано потому что:
1. S монотонно возрастает (хорновы дизъюнкты монотонны)
2. Общее число наземных фактов ограничено |M| · |dom|^k (конечно)
3. Каждая итерация добавляет хотя бы один новый факт или завершается

Завершение + конечное число итераций → разрешимость. ∎

**Следствие 2.2**: Недостижимость синтеза тоже разрешима (алгоритм возвращает UNREACHABLE за конечное время).

### 2.2 Дерево вывода

Когда синтез успешен, алгоритм производит **дерево вывода**: DAG, листья которого — входные факты, корень — цель, а каждый внутренний узел помечен правилом, которое его вывело.

```
goal: complete(order)
  ← available(product)    [правило: requires]
  ← stock_ok(product)     [правило: requires]
  ← approved(manager)     [правило: requires]
    ← within_limit(order) [правило: derives, из: price, limit]
      ← price(order)      [правило: derives, из: product, marketing]
```

Дерево вывода И ЕСТЬ граф контракта. Трансляция синтеза:

| Узел дерева вывода | Узел графа контракта |
|-------------------|---------------------|
| Входной факт | узел `input` |
| Применение правила | узел `compute` |
| Проверка согласованности | узел `guard` |
| Факт цели | узел `output` |
| Условное правило | узел `branch` |

Синтез — не поиск. Это **единый восходящий проход** с последующей **обратной трассировкой** от цели. Оба прохода линейны по размеру вывода.

## 3. Корректность (Soundness)

### 3.1 Логика Хоара и слабейшие предусловия

**Логика Хоара** (Хоар, 1969) рассуждает о корректности программ с помощью троек:

```
{ P }  программа  { Q }
```

означает: если предусловие P выполняется до исполнения, то постусловие Q выполняется после. Это **частичная корректность**: ничего не говорит о завершении.

Для синтеза контрактов нужно:

```
{ requires(M) }  C  { goal(M) }
```

**Слабейшее предусловие Дейкстры** `wp(C, Q)` — это слабейшая формула P такая, что `{P} C {Q}` выполняется. Корректность синтеза требует:

```
requires(M) ⊢ wp(C, goal(M))
```

т.е. клауза `requires` модели логически влечёт слабейшее предусловие синтезированного контракта.

### 3.2 wp для узлов контракта

Алгоритм синтеза назначает условие верификации (VC) каждому шагу синтеза. Исчисление wp для узлов контракта:

```
wp(output :x = v,    Q) = Q[x ↦ v]                        // подстановка
wp(compute :x = f(deps), Q) = wp(f, Q)[x ↦ f(deps)]       // wp функции
wp(guard { P },      Q) = P → Q                            // guard влечёт Q
wp(branch { on c => C₁; default => C₂ }, Q)
                       = (c → wp(C₁, Q)) ∧ (¬c → wp(C₂, Q))
wp(compose A then B, Q) = wp(B, wp(A, Q))                  // композиция
```

### 3.3 Теорема корректности синтеза

**Теорема 3.1** (Корректность синтеза): Пусть M — модель свойств в фрагменте синтеза, и C = Synthesize(M, I) — синтезированный контракт. Тогда:

```
requires(M) ⊢ wp(C, goal(M))
```

**Доказательство по структурной индукции по дереву вывода**:

*База*: Входные узлы. Если `v ∈ I` требуется, то `requires(M) ⊢ holds(v, ·)` по определению. Узел `input :v` имеет `wp(input :v, Q) = Q[v ↦ given_v]`, и `requires(M)` — это в точности предположение о том, что данные значения удовлетворяют ограничениям. ✓

*Инд. переход — правило derives*: Правило `B₁ ∧ ... ∧ Bₙ → derives(y, F(b₁,...,bₙ))`.
По инд. гипотезе, `requires(M) ⊢ wp(compute :bᵢ, Q)` для каждого `bᵢ`.
Синтезированный узел: `compute :y = F(b₁,...,bₙ)`.
`wp(compute :y, Q) = wp(F, Q[y ↦ F(b₁,...,bₙ)])`.
Поскольку `F` — чистая функция без побочных эффектов: `wp(F, Q') = Q'[b₁,...,bₙ ↦ given_values]`.
По инд. гипотезе, все значения `bᵢ` удовлетворяют своим requires. ✓

*Инд. переход — правило requires*: Правило `requires: x.p = v`.
Синтезированный узел: `guard :g { x.p = v }`.
`wp(guard { x.p = v }, Q) = (x.p = v) → Q`.
По определению, `requires(M) ⊢ x.p = v` (это требование).
Следовательно `requires(M) ⊢ wp(guard { x.p = v }, Q)`. ✓

*Инд. переход — правило goal*: Корень дерева вывода.
По индукции все промежуточные VC выполняются. Факт цели находится в S (фиксированная точка), поэтому `goal(M)` выводим. `wp(output :goal, goal(M)) = True`. ✓

По структурной индукции, все VC выполняются. ∎

### 3.4 Что гарантирует корректность

Корректность означает: **синтезированный контракт не может произвести неверный ответ**. Если входы удовлетворяют `requires(M)`, выход всегда удовлетворяет `goal(M)`.

Что корректность **не** гарантирует:
- Поведение на входах, нарушающих `requires(M)` (не ограничено)
- Эффективность синтезированного контракта (корректен, но не обязательно быстр)
- Единственность (могут существовать несколько корректных контрактов)

## 4. Полнота (Completeness)

### 4.1 Теорема Эрбрана

**Теорема Эрбрана** (1930): Множество хорновых дизъюнктов S выполнимо тогда и только тогда, когда оно имеет *эрбранову модель* — модель, построенную из наземных термов (без переменных), где предикат P(t₁,...,tₙ) истинен в точности тогда, когда он выводим из S.

Эрбранова модель **каноническая**: это уникальная минимальная модель S (наименьшая фиксированная точка восходящего вычисления).

**Следствие**: Для множества хорновых дизъюнктов S наземный факт F выводим из S тогда и только тогда, когда F находится в эрбрановой модели S.

### 4.2 Теорема полноты синтеза

**Теорема 4.1** (Полнота синтеза): Пусть M — модель свойств в фрагменте синтеза. Если существует хоть какой-либо контракт C такой, что `{requires(M)} C {goal(M)}`, то Synthesize(M, I) возвращает контракт C' такой, что `{requires(M)} C' {goal(M)}`.

**Доказательство**:

Пусть контракт C удовлетворяет `{requires(M)} C {goal(M)}`. Тогда денотация `⟦C⟧` — функция, которая при входах, удовлетворяющих `requires(M)`, производит выходы, удовлетворяющие `goal(M)`.

Трасса выполнения C на любом входе v, удовлетворяющем `requires(M)`, производит последовательность промежуточных значений — вывод `goal(M)` из входных фактов.

По характеризации через хорновы дизъюнкты (Теорема 1.1), этот вывод может быть выражен как последовательность применений правил из M. Каждая такая последовательность соответствует пути в эрбрановой модели.

По теореме Эрбрана, наземный факт `goal(M)` находится в эрбрановой модели M ∪ {ground(I)}. Восходящий алгоритм (доказательство Теоремы 2.1) вычисляет в точности эрбранову модель. Следовательно `goal(M) ∈ S` в фиксированной точке, и синтез успешен. ∎

**Следствие 4.2**: Алгоритм синтеза **относительно полон**: он пропускает решение только когда решения не существует.

### 4.3 Предположение о замкнутом мире

Полнота опирается на **предположение о замкнутом мире** (CWA): свойства, не выводимые из входа, считаются ложными.

При CWA: если `product.available` отсутствует во входе, оно ложно. Guard `requires: product.available = true` заблокирует.

Это стандарт в Datalog и в большинстве контекстов бизнес-логики (если нам неизвестно, что товар доступен, считаем его недоступным). Для открытых миров модель должна явно объявлять значения по умолчанию.

## 5. Ситуационное исчисление: изменения свойств во времени

### 5.1 Почему статических моделей недостаточно

Модели свойств в предыдущем документе предполагали единственный снимок: все свойства имеют фиксированные значения в момент выполнения контракта. Но реальные workflow включают **переходы состояний**: менеджер одобряет заказ, изменяя `manager.approved` с false на true.

**Ситуационное исчисление** (МакКарти, 1963; Рейтер, 1991) формализует это:

- **Флюенты** `F(x, s)`: свойство F объекта x в ситуации (состоянии) s
- **Действия** `a`: события, изменяющие значения флюентов
- **Do**: `do(a, s)` — ситуация после выполнения действия a в ситуации s
- **Poss**: `Poss(a, s)` выполняется, когда действие a возможно в ситуации s
- **Аксиома последующего состояния** для каждого флюента F и действия A:
  ```
  F(x, do(A, s)) ↔ γ⁺_F(x, A, s) ∨ (F(x, s) ∧ ¬γ⁻_F(x, A, s))
  ```
  где γ⁺ — условие, при котором A делает F истинным, γ⁻ — при котором делает ложным.

### 5.2 Проблема рамки

**Проблема рамки** (МакКарти и Хейс, 1969): специфицировать, какие свойства НЕ изменяются после действия, так же трудно, как и то, что изменяется.

Наивная спецификация требует аксиом для каждой пары действие-флюент. Для n действий и m флюентов: O(n·m) аксиом.

**Решение Рейтера** (1991): *Аксиомы последующего состояния* заменяют аксиомы рамки. Для каждого флюента F одна аксиома покрывает все действия:

```
F(x, do(A, s)) ↔
  (∃a: A = причина_F_истина(a, x)) ∨         // действие сделало F истинным
  (F(x, s) ∧ ¬∃a: A = причина_F_ложь(a, x))  // F был истинен и действие его не опровергло
```

Итого O(m) аксиом — линейно по числу флюентов.

### 5.3 Модели свойств с переходами состояний

Для контрактного языка ситуационное исчисление даёт **динамические модели свойств**:

```
model OrderLifecycle {
  fluent: order.status

  action: submit_order(order, product, manager) {
    possible when: product.available && product.stock > 0
    effect:        order.status := :processing
  }

  action: approve_order(order, manager) {
    possible when: order.status = :processing && manager.approved
    effect:        order.status := :complete
  }

  goal: order.status = :complete
}
```

Синтез из динамической модели производит **workflow-контракт** — контракт, последовательно выполняющий необходимые действия и проверяющий условия на каждом шаге.

**Теорема 5.1**: Синтез динамических моделей свойств разрешим, когда пространство действий конечно и домен флюентов конечен. Задача PSPACE-полна (эквивалентна STRIPS-планированию).

Для DAG-структурированных планов (параллельные workflow): **полиномиальное планирование частичного порядка** (МакАллестер и Розенблитт, 1991) сводится к вычислению Datalog при монотонных эффектах действий — восстанавливая PTIME-сложность.

## 6. Стратифицированное отрицание

### 6.1 Проблема с отрицанием

Чистые хорновы дизъюнкты не могут выражать отрицание: `NOT product.recalled` нельзя выразить хорновым дизъюнктом. Но отрицание необходимо для реальных моделей.

**Стратифицированный Datalog** (Апт, Блэйр и Уокер, 1988) расширяет Datalog стратифицированным отрицанием:

Программа Datalog P **стратифицирована**, если существует разбиение предикатов на страты `P₁, P₂, ..., Pₖ` такое, что:
1. Если правило `B₁ ∧ ... ∧ ¬Bᵢ ∧ ... ∧ Bₙ → H`, то `stratum(Bᵢ) < stratum(H)` (отрицаемые предикаты вычисляются в более ранних стратах)
2. Если правило `B₁ ∧ ... ∧ Bₙ → H` (без отрицания), то `stratum(Bᵢ) ≤ stratum(H)`

**Теорема 6.1** (Стратифицированный Datalog): Стратифицированный Datalog имеет уникальную **совершенную модель** (Пшимусинский, 1988), вычислимую за PTIME путём восходящего вычисления страт.

**Фрагмент синтеза, расширенный до стратифицированного отрицания**:

```
model FulfillmentWithExclusions {
  requires:
    product.available   = true
    NOT product.recalled          // стратифицированное отрицание: вычисляется до проверки requires
    manager.approved    = true
    NOT order.cancelled           // стратифицированное отрицание

  goal: order.status = :complete
}
```

Алгоритм синтеза вычисляет отрицаемые предикаты в более ранних стратах:
- Страта 1: вычислить `product.recalled`, `order.cancelled` (положительные выводы)
- Страта 2: применить `NOT product.recalled` как guard (отрицание из страты 1)
- Страта 3: вывести `goal` из результатов страты 2

Синтезированный контракт:
```
compute :recalled   = product.recalled     // страта 1
compute :cancelled  = order.cancelled      // страта 1
guard :not_recalled  { NOT recalled  }     // страта 2
guard :not_cancelled { NOT cancelled }     // страта 2
...
```

### 6.2 Что нарушает стратификацию

Модель **не стратифицируема** (и синтез в общем случае становится неразрешимым) при наличии **отрицательных циклов**:

```
// НЕВЕРНО: цикл через отрицание
model Bad {
  derives: a.approved = NOT b.blocked
  derives: b.blocked  = NOT a.approved    // ← отрицательный цикл
}
```

Компилятор обнаруживает это при проверке согласованности и отклоняет модель с конкретной ошибкой. Для отрицательно-цикличной модели не существует ни корректного синтеза, ни корректной семантики.

## 7. Дескриптивные логики и фрагмент EL

### 7.1 DL как язык онтологий

**Дескриптивные логики** (DL) — фрагменты логики первого порядка, разработанные для представления знаний. Они задают формальную семантику уровню онтологии (уровень 0) архитектуры моделей свойств.

База знаний DL имеет два компонента:
- **TBox** (терминологический): определения классов и свойств
  ```
  Product ⊑ ∀available.Bool           // Product имеет свойство available типа Bool
  AvailableProduct ≡ Product ⊓ available = true   // определённый класс
  ```
- **ABox** (ассерциональный): факты об экземплярах
  ```
  product1 : Product
  available(product1) = true
  ```

**Сервисы рассуждения** над базами знаний DL соответствуют в точности операциям синтеза:

| Задача рассуждения DL | Операция синтеза |
|----------------------|------------------|
| Согласованность TBox | Проверка согласованности модели |
| Согласованность ABox | Проверка корректности входа |
| Субсумпция концептов (C ⊑ D?) | Следует ли свойство из `requires(M)`? |
| Проверка экземпляра (a : C?) | Удовлетворяет ли вход requires? |
| Реализация концепта | Минимальный набор свойств для goal? |

### 7.2 Фрагмент EL

Не все DL трактабельны. **EL** (Бадер и др., 2005) — хорнов фрагмент дескриптивной логики:
- Только экзистенциальная квантификация (∃), конъюнкция (⊓) и top (⊤)
- Нет универсальной квантификации (∀), дизъюнкции (⊔) или отрицания (¬)

**Теорема 7.1** (Трактабельность EL): Субсумпция концептов в EL разрешима за **полиномиальное время** (PTIME).

**Теорема 7.2** (EL = хорновый Datalog): Фрагмент EL соответствует в точности хорновому Datalog над доменом свойств. Рассуждение в EL эквивалентно восходящему вычислению Datalog.

Это теоретический якорь: **онтологии EL + хорновые модели свойств = PTIME-синтез**.

OWL 2 EL — стандартный профиль DL W3C, используемый для крупномасштабных биомедицинских онтологий (SNOMED CT имеет 300K+ концептов, всё в EL). Его полиномиальное рассуждение доказано и реализовано. Мы наследуем этот аппарат.

### 7.3 За пределами EL

Для более богатых моделей:
- **ELI** (обратные роли): свойства можно обходить в обратном направлении → PTIME
- **ALC** (полная DL): отрицание + универсальная квантификация → PSPACE-полная
- **SROIQ** (OWL 2 Full): разрешима, но NEXPTIME

Для фрагмента синтеза: **оставаться в EL для гарантированного PTIME-синтеза**. Использовать более богатые фрагменты только при необходимости, с задокументированными затратами на сложность.

## 8. Уточняющие типы и синтез, управляемый типами

### 8.1 Уточняющие типы

**Уточняющий тип** (Фриман и Пфеннинг, 1991) дополняет базовый тип логическим предикатом:

```
{ x : Int | x >= 0 }        // неотрицательное целое
{ m : Money | m > 0 }        // положительные деньги
{ p : Product | p.available } // доступный продукт
```

В контрактном языке:
```
type AvailableProduct = Product where available = true
type ApprovedManager  = Manager  where approved = true && approval_limit > 0
```

### 8.2 Синтез, управляемый типами: оптимизация

Если входной тип — уточняющий тип, синтезатор может устранить guards:

**Пример**: Если вход объявлен как `AvailableProduct` (тип гарантирует `available = true`), требование модели `requires: product.available = true` **статически выполняется типом**. Узел `guard` в синтезированном контракте не нужен.

**Теорема 8.1** (Устранение guards, управляемое типами): Пусть M — модель свойств, I — входные типы с уточняющими предикатами R(I). Для каждого требования `requires: x.p = v` в M, если `R(I) ⊢ x.p = v` (выводимо из уточняющего типа входа), соответствующий узел guard может быть устранён из синтезированного контракта.

Это **оптимизация времени компиляции**: чем точнее входные типы, тем меньше guards синтезируется, тем дешевле контракт.

### 8.3 Liquid types и SMT-решение

**Liquid types** (Рондон, Кавагути и Джала, 2008) расширяют уточняющие типы ограничениями линейной арифметики и проверяют их с помощью SMT-решателей.

Для фрагмента синтеза:
- Уточняющие предикаты с линейной арифметикой: `price > 0`, `stock >= min_stock`
- Проверяются SMT-решателем (Z3, CVC5) при проверке типов
- DPLL(T)-разрешимо для линейной арифметики (PTIME для фиксированного числа переменных)

Это даёт нам **устранение арифметических guards в время компиляции** — контракты над числовыми свойствами могут иметь свои guards, разрядившиеся до выполнения.

## 9. Единая теоретическая картина

### 9.1 Пять теорий, один объект

Фрагмент синтеза находится на пересечении пяти хорошо изученных теорий:

```
Дескриптивная логика EL ─────────────────────── Рассуждение об онтологии
         │                                        (PTIME субсумпция)
         │
Хорновый Datalog / Стратифицированный Datalog ── Алгоритм синтеза
         │                                        (восходящее вычисление)
         │
Логика Хоара / исчисление wp ────────────────── Верификация корректности
         │                                        (разрядка VC)
         │
Теорема Эрбрана ─────────────────────────────── Полнота
         │                                        (минимальная эрбранова модель)
         │
Ситуационное исчисление + STRIPS ────────────── Динамические модели
                                                 (синтез переходов состояний)
```

Каждая теория вносит разную гарантию:

| Теория | Предоставляемая гарантия |
|--------|--------------------------|
| EL DL | Проверка согласованности — PTIME; вывод на уровне типов |
| Хорновый Datalog | Алгоритм синтеза завершается, PTIME, конфлюэнтен |
| Логика Хоара | Синтезированный контракт корректен (soundness) |
| Теорема Эрбрана | Синтезатор ничего не пропускает (completeness) |
| Ситуационное исчисление | Динамические модели с переходами состояний обрабатываются |
| Уточняющие типы | Устранение guards во время компиляции |

### 9.2 Центральная теорема

**Теорема 9.1** (Главный результат): Для моделей свойств в фрагменте синтеза (хорновый Datalog + стратифицированное отрицание + онтология EL + уточняющие типы):

1. **(Разрешимость)** Задача синтеза разрешима за PTIME
2. **(Корректность)** Синтезированный контракт удовлетворяет своей спецификации: `requires(M) ⊢ wp(C, goal(M))`
3. **(Полнота)** Если существует хоть один корректный контракт, синтез его найдёт (полнота при CWA)
4. **(Оптимальность)** Синтезированный контракт — это DAG-контракт минимальной глубины (оптимальный критический путь)

**Условия теоремы**:
- Нет отрицательных циклов в модели свойств
- Нет дизъюнктивных требований (только конъюнктивные)
- Конечные или ограниченные домены свойств
- Чистые определения функций в `derives` (без побочных эффектов)

**При нарушении условий**: компилятор отклоняет модель с конкретной, осмысленной ошибкой, объясняющей, какое условие нарушено и как исправить.

## 10. Граница: что выходит за пределы фрагмента

### 10.1 Дизъюнктивные требования

```
// ВНЕ ФРАГМЕНТА
model TwoSuppliers {
  requires: supplier_a.available OR supplier_b.available
  goal:     order.fulfillable = true
}
```

Дизъюнктивная хорновая логика (Datalog с дизъюнкцией в голове) — **NP-полна** (вместо PTIME). Синтез всё ещё разрешим, но экспоненциально сложнее.

**Практический обходной путь**: разбить на отдельные модели (`ModelA`, `ModelB`) и ветвить:
```
branch fulfillment {
  on supplier_a.available => FulfillViaA { ... }
  on supplier_b.available => FulfillViaB { ... }
}
```
Это всегда выразимо, когда дизъюнкция — над свойствами входа.

### 10.2 Рекурсивные модели

```
// ВНЕ ФРАГМЕНТА (в DAG-ядре)
model TreeAggregation {
  derives: node.total = node.value + sum(node.children.total)  // самоссылка
}
```

Рекурсивные модели соответствуют **рекурсивному Datalog** — разрешимому, но уже не PTIME (сложность возрастает до PSPACE для Datalog с рекурсией над бесконечными доменами).

**Практический путь**: разрешить рекурсивные модели как явный opt-in (аннотированный `@recursive`). Синтезатор генерирует рекурсивный контракт с известными затратами на сложность.

### 10.3 Вероятностные свойства

```
// РАСШИРЕНИЕ — требует вероятностного Datalog
model RiskModel {
  requires: location.anomaly_score > 0.8   // вероятностный порог
}
```

**ProbLog** (Де Рэдт и др., 2007) расширяет Datalog вероятностными фактами. Инференс ProbLog — **PP-полный** (точный инференс #P-трудный, приближённый — PTIME).

Фрагмент синтеза расширяется до вероятностных моделей путём замены точного эрбранова вывода **взвешенным подсчётом моделей** — соединяясь с документом о предвычислении.

## 11. Исследовательская программа: что остаётся доказать

### Открытая проблема 1 — Оптимальность синтеза
Текущий синтез находит *какой-то* корректный контракт. Доказать, что восходящее вычисление Datalog производит **DAG минимальной глубины** (оптимальный критический путь) среди всех корректных контрактов.

*Гипотеза*: Алгоритм синтеза оптимален по критическому пути, поскольку порядок стратификации соответствует порядку зависимостей, а восходящее вычисление Datalog производит минимальный вывод.

*Подход к доказательству*: Редукция к кратчайшему пути в DAG вывода, используя ранг страты как вес ребра.

### Открытая проблема 2 — Инкрементальный синтез
Когда модель свойств изменяется (например, добавляется новый clause `requires`), доказать, что синтезированный контракт может быть обновлён инкрементально без полного пересинтеза.

*Гипотеза*: Полунаивное вычисление Datalog применимо к синтезу контрактов: пересчитываются только правила, затронутые изменением модели.

*Подход к доказательству*: Адаптировать теорему о полунаивном вычислении (Банкильон, 1986) к контексту синтеза.

### Открытая проблема 3 — Полнота при управлении типами
Доказать, что устранение guards через уточняющие типы сохраняет полноту: синтезированный контракт без устранённых guards всё ещё корректен.

*Гипотеза*: Если `R(I) ⊢ x.p = v`, то контракт без `guard :x` имеет ту же денотацию, что контракт с ним, на всех входах, удовлетворяющих R(I).

*Подход к доказательству*: Вычисление слабейшего предусловия с уточнёнными типами входа.

### Открытая проблема 4 — Корректность вероятностного синтеза
Расширить теорему корректности до вероятностных моделей: если синтезированный контракт использует приближённые узлы, доказать, что выход удовлетворяет цели с вероятностью ≥ α (объявленный уровень доверия).

*Подход к доказательству*: Теория вероятностей + слабейшее предусловие, расширенное до распределений.

### Открытая проблема 5 — Сложность EL + стратифицированное отрицание
Доказать точную сложность синтеза для полного фрагмента синтеза (онтология EL + стратифицированное отрицание + уточняющие типы). Гипотеза: всё ещё PTIME, поскольку стратифицированное отрицание добавляет не более полиномиального множителя на страту.

## Резюме

Теоретические основания устанавливают, что фрагмент синтеза обладает всеми тремя необходимыми свойствами — разрешимостью, корректностью, полнотой — при PTIME-сложности.

Цепочка доказательств:

```
Восходящее вычисление хорнового Datalog
  → Разрешимость (Теорема 2.1)
  → Завершение + корректность алгоритма синтеза

Логика Хоара / исчисление wp
  → Корректность (Теорема 3.1)
  → Синтезированный контракт корректен по построению

Теорема Эрбрана
  → Полнота (Теорема 4.1)
  → Синтезатор ничего не пропускает во фрагменте

Дескриптивная логика EL = хорновый Datalog
  → PTIME рассуждение об онтологии
  → Проверка типов и устранение guards — PTIME

Стратифицированный Datalog / совершенная модель Пшимусинского
  → Стратифицированное отрицание обрабатывается корректно
  → Уникальная семантика, PTIME вычисление

Ситуационное исчисление + STRIPS
  → Динамические модели: PSPACE для последовательных планов
  → PTIME для монотонных DAG-планов (параллельный workflow)
```

Фрагмент не произволен — он определяется требованиями трактабельности. Каждое расширение (дизъюнкция, неограниченная рекурсия, вероятность) возможно, но несёт задокументированные затраты на сложность. Язык делает эти затраты видимыми и требует явного opt-in.

Это и есть научный фундамент: не «мы думаем, что это работает», а «мы знаем, что это работает, в этих границах, с этими доказательствами».
