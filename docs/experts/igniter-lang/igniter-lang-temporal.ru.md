## § 0. Проблема

Менеджер изменяет цену продукта на выходные. Во всех языках и фреймворках, которые
приходилось видеть, это превращает одностраничный алгоритм в разросшуюся систему:

- `effective_date` добавляется в каждую модель, работающую с ценой
- таблица `price_histories` со сложными join-ами
- логика «as-of» запросов разбросана по репозиториям
- специальные случаи в вычислении суммы заказа
- новые столбцы в каждом отчёте
- журнал аудита пристёгнут как довесок
- тесты, которые проходят, но пропускают временны́е граничные случаи

Причина архитектурна: **время — второсортный гражданин**. Цена типизирована
как `Float`. Но цена — это не `Float` — цена — это **функция от времени**:
`DateTime → Money`.

Ответ языка контрактов: сделать временно́е измерение **первоклассным**,
**ортогональным** графу вычислений и **компонуемым** с системой инвариантов.
Добавление правила цены на выходные не должно касаться контракта заказа, вычисления
суммы или контракта отчёта.

## § 1. Временны́е типы

### § 1.1 `History[T]` — значение во времени

```
type History[T] = [{ value: T, from: DateTime, until: DateTime? }]
```

`History[T]` — упорядоченная последовательность интервалов, каждый несёт значение
типа `T` и полуоткрытый интервал `[from, until)`, в течение которого оно действительно.
Последовательность непересекающаяся и покрывает непрерывную область.

Константы — вырожденные истории:

```
T ⊑ History[T]
```

Обычное значение `Money` — это `History[Money]` с одним интервалом `[epoch, ∞)`.
Это означает: **существующие контракты работают без изменений** — временно́е
измерение сугубо аддитивно.

### § 1.2 Оператор временно́го доступа `[t]`

```
expr '[' time_expr ']'    -- разрешить History[T] до T в момент t
expr '[' range_expr ']'   -- срез History[T] за интервал → History[T]
```

`price[t]` вычисляет историю в момент `t` и возвращает значение типа `T`,
действительное в этот момент. Если ни один интервал не покрывает `t`, результат `Null`.

```
product.price[order.created_at]          -- цена в момент создания заказа
product.price[Date.today]                -- текущая цена
product.price[period.start..period.end]  -- история цен за период
product.price.current                    -- псевдоним для product.price[DateTime.now()]
```

**Правило типизации:**
```
Γ ⊢ h : History[T]   Γ ⊢ t : DateTime
──────────────────────────────────────────
Γ ⊢ h[t] : T?                            -- Null если t не покрыт
```

### § 1.3 Объявление временны́х узлов

Узел `compute` или `in`, значение которого меняется со временем, объявляется
с аннотацией `@temporal`:

```
in product_id: Id

compute :product = fetch(product_id)   @temporal   -- значение может меняться

compute :price   = product.price       @temporal
compute :rules   = ActiveRules         @temporal
```

Узел с аннотацией `@temporal` автоматически типизируется как `History[T]`
на уровне контракта. Внутри временны́х выражений доступ осуществляется через `[t]`.

### § 1.4 Контекст оценки `as_of`

Каждый контракт несёт неявный параметр `as_of: DateTime`.

```
in as_of: DateTime = DateTime.now()
```

Когда среда выполнения вычисляет узел с аннотацией `@temporal`, она автоматически
применяет `[as_of]` для разрешения истории до точечного значения. Пользователь пишет:

```
compute :effective_price = product.price
```

Компилятор переписывает это в:

```
compute :effective_price = product.price[as_of]
```

Вызывающий код, которому нужна конкретная точка оценки, передаёт `as_of` явно:

```
compose :historical = OrderTotal { order: order, as_of: order.created_at }
compose :current    = OrderTotal { order: order, as_of: Date.today }
```

## § 2. Объявления правил

### § 2.1 Синтаксис

```
rule_decl ::=
  'rule' IDENT ':' type_name '{'
    'applies' ':' applies_spec
    'compute' ':' fn_expr
    ['requires' ':' IDENT]
    ['priority' ':' INT]
    ['combines' ':' SYMBOL]
  '}'

applies_spec ::=
  | '{' {applies_clause ','} '}'
  | 'always'

applies_clause ::=
  | 'from' ':' date_expr
  | 'until' ':' date_expr
  | 'days' ':' '[' {SYMBOL ','} ']'
  | 'months' ':' '[' {SYMBOL ','} ']'
  | 'when' ':' predicate
  | 'manual' ':' BOOL
```

### § 2.2 Примеры

```
rule WeekendManagerPrice : Product {
  applies: { days: [:saturday, :sunday], manual: true }
  compute: fn(product, override) -> Money = override.weekend_price
  requires: ManagerApproval
  priority: 100
}

rule SeasonalDiscount : Product {
  applies: { from: :december_1, until: :january_15 }
  compute: fn(product) -> Float = 0.20     -- скидка 20%
  priority: 50
}

rule CustomerLoyaltyDiscount : Customer {
  applies: { when: customer.orders_count > 10 }
  compute: fn(customer) -> Float = 0.05    -- лояльность 5%
  priority: 30
  combines: :additive                      -- складывается с другими скидками
}

rule PriceFloor : Product {
  applies: always
  compute: fn(product) -> Money = product.cost_price * 1.10   -- порог маржи 10%
  priority: 1000                           -- порог всегда побеждает
  combines: :clamp_min
}
```

### § 2.3 Регистрация правил в системе

```
system OrderManagement {
  entities: [Order, Product, Customer]
  rules: [
    WeekendManagerPrice
    SeasonalDiscount
    CustomerLoyaltyDiscount
    PriceFloor
  ]
}
```

Добавление нового правила — **одна строка** в объявлении системы.
Никакие другие контракты не изменяются.

## § 3. Временна́я оценка контракта

### § 3.1 Контракт OrderTotal — до и после

**До временны́х правил** (оригинал, не изменялся):

```
contract OrderTotal {
  in order: Order

  compute :items = order.items
  aggregate :total = sum(items |> map { |i| i.product.price * i.qty })

  out total: Money = total
}
```

**После временны́х правил** (компилятор переписывает; исходный код не изменился):

Компилятор видит, что `product.price` аннотирован `@temporal`, и автоматически
инъецирует `as_of`:

```
contract OrderTotal {
  in order:  Order
  in as_of:  DateTime = order.created_at    -- инъецирован компилятором

  compute :items = order.items
  compute :effective_prices =
    items |> map { |i|
      i.product.price[as_of]
      |> apply(ActiveRules[as_of], for: i.product)
      |> apply(CustomerRules[as_of], for: order.customer)
      |> enforce(PriceFloor[as_of], for: i.product)
    }

  aggregate :total = sum(effective_prices |> map { |p, i| p * i.qty })

  out total:         Money      = total
  out as_of:         DateTime   = as_of
  out rules_applied: [RuleName] = active_rules_applied
}
```

Исходный код пользователя **не изменился**. Временна́я механика инъецируется
компилятором при объявлении временны́х узлов и правил.

### § 3.2 Порядок применения правил

Для каждого временно́го узла среда выполнения формирует **стек правил** в точке
оценки `t`:

1. Собрать все правила, предикат `applies` которых выполняется в момент `t`
2. Отсортировать по `priority` (по убыванию)
3. Применить по порядку согласно стратегии `combines` каждого правила

Стратегии `combines`:

| Стратегия | Смысл |
|-----------|-------|
| `:override` (по умолчанию) | Более высокий приоритет заменяет низший |
| `:additive` | Стекируется с предыдущим результатом (напр., сумма скидок) |
| `:clamp_min` | Результат = max(предыдущий, значение этого правила) |
| `:clamp_max` | Результат = min(предыдущий, значение этого правила) |
| `:first_match` | Остановиться после первого совпавшего правила |

### § 3.3 Алгоритм разрешения временно́го узла

```
resolve_temporal(node, t):
  base_value  = node.compute_at(t)
  rule_stack  = system.rules_for(node.type, t)
               |> sort_by(:priority)
  result      = base_value
  for rule in rule_stack:
    result = combine(result, rule.compute(result, t), rule.combines)
  return result
```

Для конечных наборов правил — PTIME: простая упорядоченная свёртка.

## § 4. Принцип ортогональности

Центральная архитектурная гарантия: **временны́е правила ортогональны графу
вычислений контракта**.

Добавление правила:
- НЕ изменяет топологию графа (нет новых рёбер, нет новых узлов)
- НЕ изменяет сигнатуры типов существующих контрактов
- НЕ требует касаться исходного кода существующих контрактов
- Изменяет только **то, какое значение резолвит временно́й узел** в заданный момент

Формально: пусть `G` — граф контрактов, `R` — набор правил. Оценка `G` с набором
`R ∪ {r}` отличается от оценки `G` с `R` только в разрешённых значениях временны́х
узлов — не в структуре графа, порядке зависимостей или результате проверки типов.

Это **свойство фрейма** временно́й модели: всё, чего не касается правило, остаётся
неизменным. Прямое соответствие решению Рейтера для проблемы фрейма в ситуационном
исчислении:

> *Аксиома следующего состояния*: значение флюента после действия — либо значение,
> установленное действием, либо предыдущее значение (если действие не затрагивает
> флюент).

В терминах контрактов: правило устанавливает значение временно́го узла на интервале.
Все остальные узлы не затронуты.

## § 5. Временны́е инварианты

### § 5.1 Замороженные значения

Некоторые значения должны быть **заморожены в момент создания** — они не должны
изменяться даже при последующем изменении базовых временны́х данных.

```
invariant FrozenOrderTotal : Order {
  -- Сумма заказа фиксируется в момент создания, не пересчитывается задним числом
  total == OrderTotal { order: self, as_of: created_at }.total
}

invariant FrozenItemPrices : OrderItem {
  -- Цена позиции — цена, действовавшая в момент создания заказа
  unit_price == product.price[order.created_at]
               |> apply(ActiveRules[order.created_at], for: product)
}
```

Компилятор проверяет это статически: добавление нового ценового правила не может
задним числом изменить `total` уже созданного заказа.

### § 5.2 Временна́я согласованность

```
invariant RuleIntegrity : Order {
  -- Все применённые к заказу правила были активны в момент его создания
  applied_rules.all? { |r| r.applies_at(created_at) }
}

invariant TemporalCoverage : Product {
  -- В истории цен нет пробелов — цена существует всегда
  price.covers?(DateRange.all)
}
```

### § 5.3 Монотонные ограничения

```
invariant SalePriceIsLower : Product {
  when sale_active_at(t) {
    price[t] <= price[t - 1.day]    -- цена акции ниже, чем до акции
  }
}

invariant PriceFloorRespected : Product {
  price.all? { |interval| interval.value >= cost_price * 1.10 }
}
```

### § 5.4 Временны́е инварианты и компилятор

Временны́е инварианты распространяются через композицию так же, как статические
(из [igniter-lang-invariants.md](igniter-lang-invariants.md)), с одним дополнением:
компилятор проверяет временны́е инварианты по всему объявленному набору правил.

При добавлении нового правила `r`:
1. Компилятор вычисляет значения временны́х узлов при наборе `R ∪ {r}`
2. Проверяет все инварианты `FrozenValue` — убеждается, что замороженные значения не изменились
3. Проверяет все инварианты `TemporalConsistency`
4. Выдаёт диагностику при нарушении любого инварианта

При нарушении правило отклоняется с конкретной трассой инварианта — не исключение
во время выполнения.

## § 6. Отчёты и контрфактуальный анализ

### § 6.1 Отчёт по периоду

```
contract PeriodReport {
  in period:  DateRange
  in orders:  [Order]

  -- Фактическая выручка: каждый заказ оценивается на момент его создания
  collection :actuals =
    map(orders, OrderTotal)             -- as_of по умолчанию = order.created_at

  -- Контрфактуал: те же заказы с сегодняшними правилами
  collection :counterfactual =
    map(orders, OrderTotal { as_of: Date.today })

  -- Разбивка по влиянию правил
  collection :rule_impact =
    map(orders, fn(order) -> {
      order_id: order.id
      actual:   OrderTotal { order: order, as_of: order.created_at }.total
      current:  OrderTotal { order: order, as_of: Date.today }.total
      delta:    current - actual
    })

  aggregate :actual_revenue        = sum(actuals         |> map { .total })
  aggregate :counterfactual_revenue = sum(counterfactual  |> map { .total })
  aggregate :price_impact          = counterfactual_revenue - actual_revenue

  out summary: PeriodSummary = {
    actual:         actual_revenue
    counterfactual: counterfactual_revenue
    price_impact:   price_impact
    period:         period
  }
  out detail: [RuleImpact] = rule_impact
}
```

Этот контракт отвечает на вопросы:
- «Какова фактическая выручка за этот месяц?» → `actual_revenue`
- «Какой была бы выручка при сегодняшних ценах?» → `counterfactual_revenue`
- «Как изменения цен на выходные повлияли на выручку?» → `price_impact`
- «Какие заказы пострадали больше всего?» → `detail`

Без SQL. Без join-ов. Без отдельного аналитического сервиса.

### § 6.2 Анализ «что если»

```
contract WhatIfAnalysis {
  in orders:        [Order]
  in proposed_rule: Rule
  in period:        DateRange

  -- Текущая выручка с существующими правилами
  collection :current =
    map(orders |> select { |o| period.covers?(o.created_at) }, OrderTotal)

  -- Гипотетическая выручка с добавленным предлагаемым правилом
  collection :hypothetical =
    map(
      orders |> select { |o| period.covers?(o.created_at) },
      OrderTotal { extra_rules: [proposed_rule] }
    )

  aggregate :current_revenue      = sum(current      |> map { .total })
  aggregate :hypothetical_revenue  = sum(hypothetical |> map { .total })

  out impact: RuleImpactAnalysis = {
    revenue_delta:    hypothetical_revenue - current_revenue
    orders_affected:  hypothetical |> count { |h, c| h.total != c.total }
    avg_order_change: (hypothetical_revenue - current_revenue) / orders.count
  }
}
```

Менеджер по ценообразованию может оценить предлагаемое правило на исторических
данных **до его активации** — без изменения кода, без промежуточной среды.

## § 7. Алгебра композиции правил

### § 7.1 Операции

Правила компонуются с той же алгеброй, что и контракты:

```
-- Последовательность: rule_2 видит результат rule_1
rule_1 >> rule_2

-- Выбор более специфичного (более узкая временна́я область побеждает)
rule_1 | rule_2

-- Оба применяются независимо, результаты объединяются
rule_1 ∧ rule_2

-- Исключительность: только одно может совпасть в данном интервале
rule_1 ⊕ rule_2

-- Условный запасной: rule_2 только если rule_1 не совпал
rule_1 else rule_2
```

### § 7.2 Обнаружение конфликтов

Компилятор статически обнаруживает конфликты правил:

**Конфликт перекрытия**: два правила одного приоритета и типа применяются к
перекрывающимся интервалам — результат неоднозначен:

```
-- КОНФЛИКТ: оба применяются в декабрьские выходные, одинаковый приоритет
rule WeekendDiscount : Product { applies: { days: [:saturday, :sunday] } ... }
rule DecemberSale    : Product { applies: { from: :december_1 } ... }
```

Компилятор: «временно́й конфликт — WeekendDiscount ∩ DecemberSale в интервале
[dec_1..jan_15 ∩ субботы/воскресенья]; разрешите через priority или combines».

**Нарушение порога**: правило скидки конфликтует с правилом порога:

```
rule AggressiveDiscount : Product {
  applies: { when: marketing.campaign_active }
  compute: fn(product) -> Money = product.price * 0.40   -- скидка 60%
}

-- PriceFloor объявлен с priority:1000 и combines: :clamp_min
-- Компилятор предупреждает: AggressiveDiscount может нарушить PriceFloor
-- при активной кампании
```

### § 7.3 Порядок специфичности правил

При отсутствии явного приоритета компилятор упорядочивает правила по специфичности:

1. **Временна́я специфичность**: более узкий интервал специфичнее
   (`{ days: [:saturday] }` > `{ months: [:december] }` > `always`)
2. **Специфичность предиката**: больше условий = специфичнее
3. **Специфичность типа**: правило подтипа > правило надтипа
   (правило `VipCustomer` > правило `Customer`)

## § 8. Формальное основание

### § 8.1 Ситуационное исчисление (McCarthy 1963, Reiter 1991)

Из [igniter-lang-theory2.md](igniter-lang-theory2.md), §7: ситуационное исчисление —
формальная модель для рассуждений об изменяющихся мирах.

| Ситуационное исчисление | Временна́я контрактная модель |
|------------------------|------------------------------|
| Ситуация `S` | Контекст оценки `(contract, as_of: t)` |
| Флюент `F(S)` | Значение временно́го узла в `as_of` |
| Действие `A` | Применение правила |
| Аксиома следующего состояния | Спецификация `applies:` + функция `compute:` |
| Аксиома фрейма | Принцип ортогональности: нетронутые узлы неизменны |
| Начальная ситуация `S₀` | Базовое значение `History[T]` до применения правил |

Решение Рейтера для проблемы фрейма: O(m) аксиом следующего состояния для m
действий. В терминах контрактов: O(r) объявлений правил для r правил. Без взрыва аксиом.

### § 8.2 Оценка за PTIME

Временна́я оценка выполняется за PTIME при:
- Конечных наборах правил (гарантировано для любого конечного объявления системы)
- Конечных интервалах истории (гарантировано объявлениями `from`/`until`)
- PTIME-предикатах `applies:` (линейная арифметика на DateTime — да; рекурсия — нет)

Разрешение правил — упорядоченная свёртка конечного отсортированного списка:
O(r log r) для сортировки + O(r) для применения = O(r log r) суммарно.

Срез истории `h[t]` — бинарный поиск по списку интервалов — O(log n) для n интервалов.

### § 8.3 Связь с временны́м Datalog

Временна́я оценка отображается на **временно́й Datalog** (Chomicki & Imielinski 1988):
правила — это клозы Datalog с временны́м аргументом. Оценка — нижняя точка
над временно́й областью, которая остаётся PTIME для ограниченных областей.

Это сохраняет гарантию разрешимости Теоремы 2.1 из [igniter-lang-theory2.md](igniter-lang-theory2.md)
для временно́го фрагмента.

## § 9. Обещание компактности

Тестовый сценарий: менеджер изменяет цену продукта на выходные. Что меняется?

| Изменение | Классическая система | Временна́я контрактная модель |
|-----------|---------------------|------------------------------|
| Добавить цену на выходные | +миграция, +таблица price_history, +join в запросе заказа, +столбцы отчёта | +1 объявление `rule WeekendPrice` |
| Добавить скидку лояльности | +таблица customer_discounts, +логика проверки условий, +join в отчёте | +1 объявление `rule CustomerLoyalty` |
| Добавить сезонное правило | +таблица seasonal_prices, +логика диапазона дат, +спецслучаи в суммах | +1 объявление `rule SeasonalDiscount` |
| Заморозить цены заказа | +логика снимков, +журнал аудита, +миграция | автоматически: инвариант `FrozenOrderTotal` |
| Отчёт по периоду | +новый SQL-запрос, +отдельный аналитический сервис, +конвейер данных | +контракт `PeriodReport` на базе существующего `OrderTotal` |
| Анализ «что если» | +аналитический сервис, +копия данных для тестирования | +контракт `WhatIfAnalysis` (~20 строк) |
| Отчёт о влиянии цен | +нестандартный запрос, +ETL в хранилище | +коллекция `counterfactual` в существующем `PeriodReport` |

В каждой строке: классический подход = новые файлы, новые таблицы, изменения
существующего кода. Подход контрактов = новые объявления, ноль изменений в
существующих контрактах.

## § 10. Открытые направления исследований

### § 10.1 Двухвременно́е измерение (Bitemporal)

Большинство систем отслеживают одну ось времени (действительное время: когда факт
истинен в мире). Двухвременны́е системы отслеживают две независимые оси:

- **Действительное время** (`vt`): когда факт был истинен в бизнес-реальности
- **Время транзакции** (`tt`): когда система зафиксировала его

```
product.price[vt: order.created_at, tt: DateTime.now()]
-- «Какова была цена в момент создания заказа с учётом всего, что мы знаем сейчас?»

product.price[vt: order.created_at, tt: order.created_at]
-- «Какова была цена с учётом только информации, доступной в момент создания?»
```

Двухвременно́й доступ позволяет:
- Исправлять исторические ошибки ценообразования без изменения сумм исторических заказов
- Журнал аудита: «что система знала в момент T?»
- Соответствие регуляторным требованиям: «что было истинным на дату D для отчётного периода R?»

Для этого требуется расширить `History[T]` до 2D-поверхности вместо 1D-последовательности.

### § 10.2 Синтез временны́х правил

Дано: *«В декабре выручка должна вырасти на 15% по сравнению с ноябрём»*
Дано: текущий набор правил и история ценообразования
Синтезировать: правило скидки или ценообразования, достигающее цели по выручке

Это связывает временну́ю модель с синтезом на основе моделей свойств из
[igniter-lang-propmodel.md](igniter-lang-propmodel.md). Цель становится временны́м
свойством; синтезатор ищет в пространстве правил.

Реализуемость: для линейных ценовых правил (`base * factor + constant`) синтез
является задачей линейного программирования — PTIME через симплекс-метод.

### § 10.3 Причинные цепи между правилами

Когда правило A изменяет значение X, а правило B зависит от X, порядок применения
имеет значение. Два правила, кажущихся независимыми, могут быть причинно связаны:

```
rule VolumeDiscount : Order {
  applies: { when: order.total > 1_000 }
  compute: fn(order) -> Float = 0.10
}

rule SeasonalDiscount : Product {
  applies: { from: :december_1, until: :january_15 }
  compute: fn(product) -> Float = 0.20
}

-- После применения SeasonalDiscount сумма заказа может упасть ниже 1_000
-- VolumeDiscount тогда перестаёт применяться — причинная зависимость!
```

Компилятор должен обнаруживать и предупреждать о таких причинных циклах в
оценке правил.

### § 10.4 Вероятностные временны́е правила

Правила с неопределёнными предикатами `applies:` — например, «применять, если
вероятность одобрения менеджером высока» — связывают временны́е правила с моделью
приближённых вычислений из [igniter-lang-precomp.md](igniter-lang-precomp.md):

```
rule AnticipatedPromotion : Product {
  applies: { when: ~approval_likelihood > 0.8 }   -- вероятностный applies
  compute: fn(product) -> Float = 0.15
  @approximate(confidence: 0.9)
}
```

### § 10.5 Распределённое время

В распределённой системе часы расходятся. `as_of: DateTime.now()` неоднозначно
между узлами. Вопросы для решения:

- Часы какого узла определяют «сейчас» для оценки правил?
- Как конструкция `await` взаимодействует с временно́й оценкой?
- Могут ли два параллельных рабочих процесса оценить один заказ при разных `as_of`
  и получить несогласованные результаты?

Связь: логические часы (Lamport 1978) как временна́я ось для распределённой
оценки контрактов.
