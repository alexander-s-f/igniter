# Контрактный язык Igniter — Вероятностное предвычисление

Дата: 2026-04-27.
Перспектива: теория языков программирования; приближённые вычисления; decision-directed evaluation.
Тема: Двухуровневая модель выполнения, где каждый узел вычислительного графа имеет дешёвый вероятностный режим предвычисления, а рантайм определяет минимально необходимую точность из потребностей downstream-потребителей.

*Статус: экспериментальное исследование — теоретическое предложение, не feature track.*
*Продолжает: [igniter-lang-theory.md](igniter-lang-theory.md) — теоретические основания.*
*Каноничная версия: [igniter-lang-precomp.md](igniter-lang-precomp.md)*

## 1. Центральная идея

Традиционные вычисления стирают различие между «знать ответ» и «вычислить его полностью». Каждый узел вычислительного графа либо выполняется полностью, либо не выполняется вовсе.

Предложение — третий режим:

> **Предвычисление**: произвести дешёвое вероятностное приближение результата
> узла — распределение, доверительный интервал, статистическую сводку — не
> выполняя полного вычисления. Переходить к точному вычислению только если
> приближение недостаточно для downstream-решения.

Для тензора `A` с миллиардом параметров:

```
Традиционно:    X = A + B*C   →  загрузить A (дорого), вычислить (дорого), вернуть X
Предвычисление: X̃ ≈ Ã + B*C  →  использовать статистическое описание A,
                                  вернуть (X̃, доверие, интервал)
                               → escalate до точного только если потребителю
                                  нужна точность за пределами интервала
```

Ключевой сдвиг: **потребитель определяет требуемую точность, а не производитель**.

## 2. Формальная структура

### 2.1 Двухуровневые домены

Каждое вычислимое значение существует одновременно в двух доменах:

```
Точный домен    V   : реальное значение (Real, Tensor, Record, ...)
Абстрактный     Ṽ   : описание V (распределение, интервал, форма, ...)
```

Абстрактный домен — **звуковое приближение**: если точное значение равно `v`, то `v ∈ γ(ṽ)`, где `γ` — функция конкретизации. Приближение никогда не ошибается — оно только может быть недостаточно точным.

Для распространённых абстрактных доменов:

| Абстрактный домен | Представление | Конкретизация |
|-------------------|---------------|---------------|
| Интервал | `[lo, hi]` | `{x : lo ≤ x ≤ hi}` |
| Гауссово | `(μ, σ)` | `N(μ, σ)` — бесконечное множество, конечное описание |
| Доверительный интервал | `(оценка, ε, α)` | с вероятностью `α`, `|X − оценка| ≤ ε` |
| Форма (тензоры) | `[d₁, d₂, ..., dₙ]` | все тензоры этой формы |
| Знак | `{+, −, 0}` | все числа данного знака |

### 2.2 Поднятые операции

Каждая функция `f : V → W` имеет соответствующую **поднятую** версию `f̃ : Ṽ → W̃`, работающую с описаниями:

```
f    :  V  →  W       (точная)
f̃   :  Ṽ  →  W̃      (приближённая)
```

Методы подъёма, по возрастанию стоимости и точности:

| Метод | Как работает | Когда применять |
|-------|-------------|-----------------|
| **Shape inference** | Алгебраические правила на размерностях | Тензорные операции, всегда бесплатно |
| **Интервальная арифметика** | Правила `[a,b] op [c,d]` | Монотонные функции, символьно |
| **Метод дельта** | Линеаризация `f` вокруг `μ_A` | Гладкая `f`, дёшево |
| **Монте-Карло** | Сэмплировать `Ã` k раз, прогнать `f` на каждом | Всегда применим |
| **Обученный суррогат** | Дешёвая модель, аппроксимирующая `f` | `f` вызывается очень часто |

Подъём не требует полного значения `A` — только его описания `Ã`. Для тензора с миллиардом параметров `Ã` может быть средним вектором и ковариацией — вычисленным один раз и переиспользуемым.

### 2.3 Decision-Directed Evaluation

Ключевой инсайт: **downstream-потребители объявляют требуемую точность, а не производитель**.

Узел ветвления, сравнивающий `X > threshold`, нуждается лишь в том, чтобы знать, находится ли интервал `[X_lo, X_hi]` целиком выше, целиком ниже или включает порог:

```
Случай 1:  X_hi < threshold  →  исход ветвления определён (False), точное не нужно
Случай 2:  X_lo > threshold  →  исход ветвления определён (True), точное не нужно
Случай 3:  threshold ∈ [X_lo, X_hi]  →  неопределённость → escalate до точного X
```

Это **decision-directed lazy evaluation**: структура решений в графе вычислений определяет минимально необходимую точность в каждом узле. Точно вычисляются только те узлы, чьего приближения недостаточно для downstream-решений.

## 3. Двухуровневые узлы контракта

### 3.1 Объявление узла

В контрактно-нативном языке узел compute объявляет оба уровня:

```
compute :total_risk {
  approximate: portfolio.sample(1_000) |> estimate_risk(rates)
               @confidence(0.95) @method(:monte_carlo)

  exact: portfolio |> calculate_exact_risk(rates)
}

compute :revenue {
  approximate: transactions |> statistical_summary |> estimate_revenue(fx_rates)
               @confidence(0.90) @tolerance(0.01)   // в пределах 1% от точного

  exact: transactions |> calculate_exact_revenue(fx_rates)
}
```

Уровень `approximate` необязателен. Узлы без него используют точное вычисление.

### 3.2 Объявление требований потребителя

Downstream-узлы объявляют, какая точность им нужна:

```
branch decision {
  on total_risk > 1_000_000 => TriggerAlert { risk: total_risk @exact }
  default                   => NoAction     {}
}

compute :report_line = format_revenue(revenue @tolerance(0.05))
// отчёт нуждается лишь в 5% tolerance → приближение всегда достаточно
```

### 3.3 Стратегия разрешения рантайма

```
Для каждого узла X:
  1. Собрать требования точности от всех прямых потребителей
  2. Если хоть один потребитель требует @exact → запустить точное вычисление
  3. Иначе → запустить approximate:
       а. Выполнить approximate уровень → получить (X̃, confidence, interval)
       б. Для каждого downstream-решения, использующего X̃:
            - Можно ли принять решение на основе интервала? → да: продолжать
            - Нет? → escalate: запустить точный X, инвалидировать X̃
  4. Вернуть минимально-дорогой достаточный результат
```

## 4. Импликации на уровне графа

### 4.1 Распространение приближения

Абстрактная интерпретация распространяется по графу автоматически. Если `X̃` известно и `Y = g(X)`, то `Ỹ = g̃(X̃)` вычислимо без `X`. Это означает:

```
contract RiskPipeline {
  in portfolio: Portfolio        // Ã = форма + выборочная статистика
  in rates:     Rates            // точные (маленькие)

  compute :raw_exposure  = exposure_matrix(portfolio, rates)
                           // Ã_exposure = форма [N, M], границы [0, rate_max]

  compute :total_risk    = sum(raw_exposure)
                           // Ã_total = (сумма границ) = [0, N*M*rate_max]
                           // плюс: сэмпл 1К строк → более жёсткая оценка

  compute :risk_category = classify(total_risk)
                           // можно решить: total_risk > 1M? вероятно да/нет

  out decision = risk_category
}
```

Если приближение на `total_risk` достаточно для классификации, ни `raw_exposure`, ни `total_risk` не нужно вычислять точно. Решение каскадируется от выхода назад к входам: **обратное распространение требований точности**.

### 4.2 Раннее прореживание ветвей

В контракте с `branch` уровень предвычисления может определить, какая ветвь будет выбрана, до запуска любых ветвь-специфичных вычислений:

```
branch routing {
  on estimated_revenue(summary) > threshold => HighValuePath { revenue: revenue }
  default                                   => StandardPath  { }
}
```

Если `estimated_revenue(summary) >> threshold` с доверием 0.99, выбирается `HighValuePath`. Узел `revenue` разрешается лениво только если контракт `HighValuePath` действительно его использует — и только до необходимой ему точности.

## 5. Связь с теоретическими основаниями

Это предложение соединяется со всеми пятью теоретическими столпами из предыдущего репорта.

### 5.1 Абстрактная интерпретация (Кузо, 1977)

Вероятностное предвычисление — это и есть абстрактная интерпретация с вероятностным абстрактным доменом. Фреймворк Кузо обеспечивает:

- **Гарантию звуковости**: абстрактный результат всегда является корректным сверхприближением
- **Монотонность**: уточнение абстрактного значения может только улучшать точность
- **Сходимость**: для DAG-контрактов предвычисление всегда завершается (фиксированная точка не нужна)

Вероятностный домен `(μ, σ, α)` — конкретный экземпляр фреймворка Кузо, хорошо изученный под названием **вероятностная абстрактная интерпретация** (Монно, 2000).

### 5.2 Атрибутные грамматики (Кнут, 1968)

Двухуровневая модель прямо отображается на структуру атрибутной грамматики:

- **Точные атрибуты**: стандартные синтезируемые/наследуемые атрибуты (вычисляются сверху вниз или снизу вверх)
- **Абстрактные атрибуты**: второй набор атрибутов, несущих приближения
- **Стратегия вычисления**: абстрактные атрибуты вычисляются первыми; точные — только когда абстрактных недостаточно

Это **стратифицированная атрибутная грамматика**: одна страта вычисляет абстрактные атрибуты, вторая — точные, обусловленные первой.

### 5.3 CCP (Сарасват, 1989)

Предвычисление добавляет новый вид `tell` в хранилище ограничений:

- `tell_exact(X = v)`: добавить точное ограничение (стандартный CCP)
- `tell_approx(X ∈ [lo, hi], α)`: добавить вероятностное ограничение

Операции `ask` в узлах решений проверяют оба вида:
- Если `ask(X > threshold)` разрешимо из приближённого ограничения → продолжать
- Иначе → блокировать до появления точного ограничения

Это **вероятностное конкурентное программирование в ограничениях** — известное расширение CCP (Гупта и др., 1994).

### 5.4 Байесовские сети

Уровень вероятностного предвычисления — это байесовский инференс над графом контракта. Если входы имеют априорные распределения, абстрактное вычисление распространяет эти распределения вперёд по графу, производя апостериорное распределение над выходом.

Двухуровневая модель — прагматическая версия: распространять распределения дёшево (предвычисление), и переходить к точному инференсу только там, где апостериорное слишком неопределённо для требуемого решения.

### 5.5 Datalog (разрешимость)

Уровень абстрактного вычисления сам является программой на Datalog над абстрактным доменом — разрешимой и в PTIME. Уровень точных вычислений тоже является Datalog (для DAG-ядра). Оба уровня завершаются. Протокол escalate завершается, потому что:
1. Предвычисление завершается (Datalog)
2. Escalate к точному монотонен (однажды точный, остаётся точным)
3. Структура DAG гарантирует отсутствие циклов

## 6. Что существует и что ново

### 6.1 Существующие работы

| Система / область | Что делает | Отношение к предложению |
|-------------------|-----------|-------------------------|
| Haskell lazy evaluation | Откладывает вычисление до необходимости | Откладывает, но вычисляет точно когда нужно |
| JAX / TensorFlow tracing | Shape inference до выполнения | Абстрактный домен = только формы |
| Approximate computing (Карбин, 2012+) | Допускает приближённые результаты | Фокус на железе, не на языковой семантике |
| Probabilistic programming (Stan, Pyro) | Работают с распределениями | Полный инференс, не слой предвычислений |
| Abstract interpretation (Кузо, 1977) | Анализировать программы без запуска | Инструмент статического анализа, не рантайм-стратегия |
| Speculative execution (CPU) | Выполнять до знания о нужности | Уровень железа, не семантики |

### 6.2 Что ново

Эта комбинация не была построена:

> Контрактно-нативный язык, где:
> 1. Каждый узел объявляет и точное, и приближённое вычисление
> 2. Downstream-потребители объявляют требуемую точность
> 3. Рантайм распространяет требования точности назад по графу
> 4. Только узлы, чьего приближения недостаточно для downstream-решений, вычисляются точно
> 5. Весь механизм является частью языковой семантики, а не отдельным слоем оптимизации

Ключевая новизна: **обратное распространение требований точности через граф вычислений как first-class языковая функция**. Существующие системы делают это либо статически (абстрактная интерпретация, вывод типов), либо вообще не делают. Ни один существующий язык не выражает это как рантайм-контракт между производителем и потребителем.

## 7. Проблема подъёма (открытые исследования)

Самый трудный теоретический вопрос: **как поднять произвольные вычисления до их приближений?**

### 7.1 Таксономия подъёма

Не все функции могут быть подняты дёшево. Таксономия по сложности подъёма:

| Класс функции | Метод подъёма | Стоимость | Звуковость |
|---------------|--------------|-----------|------------|
| Линейная (`f(x) = ax + b`) | Точное интервальное распространение | Бесплатно | Точная |
| Монотонная | Интервальная арифметика | Бесплатно | Звуковая |
| Гладкая, дифференцируемая | Метод дельта (1-й порядок Тейлора) | Низкая | Приближённая |
| Липшицева | Интервал с константой Липшица | Низкая | Звуковая |
| Чёрный ящик | Монте-Карло сэмплирование | Средняя | Статистическая |
| Разрывная, негладкая | Монте-Карло или отказ | Высокая | Статистическая |

### 7.2 Гипотеза автоподъёма

Гипотеза: для значительной доли реальных workflow-вычислений (финансовых, логистических, compliance) функции гладкие или монотонные, что делает автоматический звуковой подъём осуществимым.

Доказательства в пользу гипотезы:
- Агрегации (`sum`, `avg`, `max`) линейны/монотонны → точный интервальный подъём
- Бизнес-правила (`price * quantity * rate`) линейны → точный подъём
- Пороги классификации (`risk > 1M`) нуждаются только в границах, не в распределениях
- Действительно негладкие случаи (вызовы LLM, внешних API) и так являются приближёнными

### 7.3 Пользовательский объявленный подъём

Для функций, которые не могут быть подняты автоматически, программист объявляет подъём:

```
fn calculate_risk(portfolio: Portfolio, rates: Rates) -> Money {
  // точная реализация
  portfolio.reduce { |acc, pos| acc + pos.value * rates[pos.currency] }
}

fn ~calculate_risk(portfolio: ~Portfolio, rates: Rates) -> ~Money {
  // приближённый подъём: сэмпл 1000 позиций, масштабировать
  portfolio.sample(1_000).reduce { |acc, pos|
    acc + pos.value * rates[pos.currency]
  } * (portfolio.size / 1_000.0)
  @confidence(0.95) @method(:sampling)
}
```

Префикс `~` обозначает приближённый уровень. Язык обеспечивает контракт звуковости: если пользователь объявляет подъём, компилятор проверяет (статически или при тестировании), что он является корректным приближением.

## 8. Модель стоимости

Слой предвычисления вводит мета-уровневую модель стоимости. Каждый узел имеет:

```
Cost(X, level) = { approx_cost(X), exact_cost(X) }
Value(X, decision) = точность, необходимая для разрешения decision
```

Рантайм оптимизирует:

```
минимизировать Σ cost(Xᵢ, levelᵢ)
при условии: для каждого downstream-решения D,
             приближение его входов достаточно для разрешения D
```

Это задача **resource-constrained computation planning** — NP-трудная в общем случае, но разрешимая для DAG-контрактов с ограниченными деревьями решений (типичный случай).

На практике эвристика такова:
1. Попробовать approximate для всех узлов
2. Пройти дерево решений; отметить узлы, нуждающиеся в escalate
3. Вычислить точно только отмеченные узлы

Это даёт план выполнения с минимальной стоимостью за один прямой + обратный проход по графу.

## 9. Исследовательская программа

### Приоритет 1 — Вероятностный абстрактный домен (2–3 месяца)

Определить формальный абстрактный домен:
- Какие семейства распределений поддерживать (Гауссово, равномерное, эмпирическое)
- Доказательство звуковости для стандартных поднятых операций
- Интеграция с формальной моделью контракта из теоретического репорта

### Приоритет 2 — Исчисление подъёма (3–4 месяца)

Определить правила автоматического подъёма:
- Полные правила подъёма для линейных/монотонных операций
- Правила метода дельта для гладких функций
- Монте-Карло как универсальный fallback со статистическими гарантиями звуковости

### Приоритет 3 — Обратное распространение точности (2–3 месяца)

Формализовать обратный проход:
- Как требования точности потребителей распространяются по графу
- Доказать, что распространение завершается и является звуковым
- Точно определить протокол escalate

### Приоритет 4 — Языковая интеграция (4–6 месяцев)

Спроектировать конкретный синтаксис для двухуровневых узлов в контрактном языке:
- Объявления узлов `approximate:` / `exact:`
- Аннотации потребителей `@tolerance` / `@confidence`
- Синтаксис `~fn` для пользовательских подъёмов
- Расширения системы типов для приближённых значений

### Приоритет 5 — Эмпирическая валидация (непрерывно)

Построить бенчмарк:
- Выбрать 10 workflow-контрактов с дорогостоящими узлами вычислений

## Резюме

Модель предвычисления добавляет вертикальное измерение в граф контракта:

```
Уровень 0 (Предвычисление):  дёшево, приближённо, вероятностно
                              ↓ если недостаточно для решения
Уровень 1 (Точный):          дорого, точно, детерминированно
```

Направление выполнения — **назад от решений к данным**: требования потребителей определяют минимальную точность, которая распространяется через граф, и определяет, что должно быть вычислено точно, а что может остаться приближённым.

Это абстрактная интерпретация (Кузо) с вероятностными доменами, запускаемая как рантайм-стратегия оптимизации, а не как статический анализ. Она естественно вписывается в контрактную модель, потому что граф зависимостей уже делает отношения потребитель-производитель явными — обратное распространение требований точности — это просто ещё один обход того же самого графа.

Новый вклад — сделать это first-class языковой функцией: декларативные приближённые уровни, типизированные доверительные интервалы и рантайм, распространяющий точность — не надстройка-оптимизация, а часть языковой семантики с самого начала.
