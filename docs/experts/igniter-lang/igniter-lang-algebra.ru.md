## § 1. Максимальная унификация: всё — контракт

### § 1.1 Типы как контракты

Стандартный взгляд трактует типы как объекты, а контракты — как морфизмы между ними.
Максимальная унификация идёт дальше: **типы сами являются контрактами**.

```
contract Int {
  in  wire: Bit[64]
  guard :valid { is_int(wire) }
  out value: Int = decode(wire)
}

contract Add {
  in  a: Int
  in  b: Int
  out result: Int = a + b
}
```

В этой модели:
- Проверка типов = композиция контрактов
- Вывод типов = синтез контракта (модель свойств в действии)
- Ошибка типа = «нет контракта с такой сигнатурой» — не категорический сбой,
  а промах синтезатора; система может предложить альтернативы

Выражение `a + b` превращается в `Add { a: a, b: b }.result`. Программа —
граф контрактов до самого дна.

### § 1.2 Алгебра контрактов

Если всё — контракт, все программные операции становятся операциями над
контрактами. Алгебра замкнута:

| Операция | Обозначение | Смысл |
|----------|-------------|-------|
| Последовательность | `A >> B` | выход A подаётся на вход B |
| Параллель | `A ⊗ B` | независимые графы, объединённые выходы |
| Выбор | `A \| B` | ветвление по условию |
| Уточнение | `A where guard` | добавить ограничение к существующему контракту |
| Подъём | `~A` | приближённая версия A |
| Проекция | `A.field` | выбрать один выход из A |
| Итерация | `A*` | повтор до условия (opt-in, не DAG) |

Это законы **симметричной моноидальной категории**, где контракты — морфизмы,
а `⊗` — тензорное произведение. Последовательная композиция ассоциативна
с единицей `id[τ]`; параллельная коммутативна с точностью до изоморфизма.

### § 1.3 Связь с Arrows

Тайпкласс `Arrow` в Haskell — это в точности данная абстракция: обобщение функций
до «вычислений с типизированными входами и выходами»:

```haskell
class Arrow a where
  arr   :: (b -> c) -> a b c        -- поднять функцию до контракта
  (>>>) :: a b c -> a c d -> a b d  -- последовательная композиция
  first :: a b c -> a (b,d) (c,d)  -- параллель (левая компонента)
```

Контракты Igniter — это стрелки (arrows). Законы стрелок дают бесплатные теоремы
о композиции контрактов: ассоциативность, тождество и закон перестановки
выполняются по построению из свойства DAG.

### § 1.4 Аксиомная граница

У каждой системы есть необоснованное дно. В арифметике Пеано — аксиома
преемника; в теории типов — правила формирования. Контракты не исключение:
на какой-то глубине они упираются в примитивы языка-хоста.

Вопрос не в том, *существует ли* аксиомный слой — он существует всегда.
Вопрос в том, **где провести границу и как сделать её невидимой**.

Три позиции для границы:

| Позиция | Что является аксиомой | Опыт пользователя |
|---------|----------------------|-------------------|
| **Тонкая** | Только арифметика и IO | Вся доменная логика — контракты |
| **Средняя** | Функции/классы на хост-языке (текущий Igniter) | Контракты компонуют функции |
| **Толстая** | Целые сервисы — аксиомы | Контракты — только клей |

Исследуемое здесь видение — **тонкая** позиция: опустить границу как можно ниже,
чтобы энтерпрайз-пользователи никогда не касались аксиомного слоя.

## § 2. Органичный аксиомный слой

### § 2.1 Проблема словаря

Аксиомный слой, говорящий на языке математики (`Add`, `Compare`, `Multiply`),
не органичен для разработки уровня предприятия. Энтерпрайз-мышление использует
другой словарь:

- *«Согласовать заявку на закупку»*
- *«Зарезервировать инвентарь под прогноз»*
- *«Уведомить стейкхолдеров об изменении статуса»*
- *«Откатить упавшую платёжную сагу»*

Сделать аксиомный слой органичным означает сделать так, чтобы **словарь
библиотеки** говорил на языке домена, а аксиомный слой стал невидимым.

### § 2.2 Трёхуровневая архитектура без швов

```
╔══════════════════════════════════════════════════════╗
║  СИСТЕМА    OrderManagement, InventorySystem, ...   ║  ← пользователь думает здесь
╠══════════════════════════════════════════════════════╣
║  БИБЛИОТЕКА Approve, Notify, Reserve, Audit, Saga   ║  ← стандартная библиотека
╠══════════════════════════════════════════════════════╣
║  АКСИОМЫ    Add, Compare, Fetch, Store, Emit        ║  ← невидимое дно
╠══════════════════════════════════════════════════════╣
║  ПЛАТФОРМА  Ruby / LLVM / wire                      ║  ← металл
╚══════════════════════════════════════════════════════╝
```

Пользователь работает на верхнем уровне и изредка обращается к БИБЛИОТЕКЕ.
Контракты АКСИОМНОГО уровня он никогда не пишет вручную. ПЛАТФОРМА никогда
не упоминается напрямую.

Принцип «без швов»: граница между БИБЛИОТЕКОЙ и АКСИОМАМИ должна быть
неощутимой. Библиотечные контракты должны восприниматься как встроенные
конструкции языка, а не как сторонние зависимости.

### § 2.3 Аналогия с SQL

SQL скрывает B-деревья, сканирование индексов и планировщик запросов. Вы пишете:

```sql
SELECT order_total FROM orders WHERE status = 'complete'
```

Не: «открой листовую страницу B-дерева, сканируй вперёд пока status = complete,
спроецируй столбец order_total».

Язык контрактов должен делать то же самое для бизнес-логики. Пользователь
объявляет **что** система должна вычислить на доменном уровне; граф контрактов
и его выполнение — это план запроса.

## § 3. Энтерпрайз-примитивы как контракты первого класса

Уровень БИБЛИОТЕКИ должен покрывать канонические паттерны корпоративных систем.
Каждый паттерн — макрос, разворачивающийся в базовые конструкции контрактов.

### § 3.1 Сущность с жизненным циклом

```
entity Order {
  lifecycle: pending → processing → complete | cancelled
  invariants: [
    total >= 0
    items.count > 0
    status in OrderStatus
  ]
}
```

Разворачивается в:
- Узел `branch`, охраняющий допустимые переходы жизненного цикла
- Узлы `guard`, проверяющие инварианты при каждом переходе
- Узлы `effect`, генерирующие события `order.status_changed`
- Узел `await` для асинхронных переходов

### § 3.2 Рабочий процесс (Сага)

```
workflow Fulfillment {
  steps:        [ValidateOrder, ReserveInventory, ChargePayment, Ship]
  compensation: [ReleaseInventory, RefundPayment]
  timeout:      48.hours
}
```

Разворачивается в:
- Последовательную цепь `compose` для счастливого пути
- Связи `effect @compensate` для каждого компенсирующего шага
- Верхнеуровневый `await` с таймаутом
- Автоматический контракт отката при сбое любого шага

### § 3.3 Политика (сквозное правило)

```
policy ApprovalRequired {
  when:     order.total > 10_000
  requires: ManagerApproval { manager: order.manager }
}
```

Разворачивается в:
- Узел `guard`, инъецируемый в каждый контракт, касающийся `order.total`
- Подконтракт `ManagerApproval`, компонуемый условно
- Аудитный `effect` при каждом согласовании или отказе

Политики — контрактный эквивалент аспектно-ориентированного программирования,
но объявленный явно и проверяемый во время компиляции.

### § 3.4 Объявление системы

```
system OrderManagement {
  entities:  [Order, Product, Customer]
  workflows: [Fulfillment, Return, Cancellation]
  policies:  [ApprovalRequired, FraudCheck, RateLimit]
  events:    [OrderPlaced, OrderShipped, PaymentFailed]
}
```

Объявление `system` — верхнеуровневая единица композиции. Оно:
- Называет ограниченный контекст (bounded context)
- Объявляет граф сущностей
- Связывает рабочие процессы с переходами жизненного цикла сущностей
- Применяет политики ко всем контрактам в области видимости
- Регистрирует каналы событий для привязок `await` / `effect`

## § 4. Идеал: `compute` исчезает

### § 4.1 `compute` как замаскированный `compose`

Каждый узел `compute` с нетривиальным телом — это в действительности `compose`
к библиотечному контракту, которому ещё не дали имени.

| Как пишут сегодня | В идеале алгебры контрактов |
|------------------|------------------------------|
| `compute :price = item.price * qty` | `compose :price = LineTotal { item: item, qty: qty }` |
| `compute :risk = score > 0.8 ? :high : :low` | `compose :risk = RiskBand { score: score }` |
| `compute :filtered = items.select(&:available)` | `collection :available = items \|> Select[available]` |
| `compute :total = items.sum(&:price)` | `aggregate :total = sum(items \|> map { .price })` |

Логика не исчезает — она перемещается из тел `compute` в библиотечные контракты,
где именована, протестирована и переиспользуется.

### § 4.2 Топология против шагов

На уровне предприятия пользователь задаёт **топологию** вычисления — что от чего
зависит, — а не шаги выполнения.

Это качественный сдвиг в режиме разработки:

- **Императивно**: «итерируй по позициям, для каждой умножь цену на количество, накапливай»
- **Контрактная топология**: «применить `LineTotal` к каждой позиции, агрегировать через `Sum`»

Второй вариант: более высокоуровневый, статически верифицируемый и напрямую
параллелизируемый средой выполнения. Он также читается как описание бизнеса,
а не техническая инструкция.

### § 4.3 Когда `compute` остаётся

`compute` не исчезает полностью. Он остаётся для:
- **Инлайн-выражений**, которые действительно единичны и не стоит именовать
- **Адаптерных узлов** для согласования выходов библиотечных контрактов
- **Исследовательского кода** до того, как паттерн выявлен и вынесен в библиотеку

Практическое правило: если одно и то же тело `compute` написано дважды — это
должен стать именованным библиотечным контрактом.

## § 5. Скелет стандартной библиотеки контрактов

### § 5.1 Целевое покрытие словаря

Минимальный словарь контрактов для корпоративных систем должен покрывать:

**Доступ к данным**
`Fetch[T]`, `FetchMany[T]`, `Store[T]`, `Update[T]`, `Delete[T]`,
`Query[T]` (параметризованы типом сущности)

**Валидация и авторизация**
`Validate[T]`, `Authorize[subject, action, resource]`, `RateLimit[key, limit]`,
`Sanitize[T]`

**Паттерны вычисления**
`Transform[A, B]`, `Classify[T, Label]`, `Score[T]`, `Rank[T]`,
`Deduplicate[T]`, `Reconcile[A, B]`

**Коммуникация и интеграция**
`Notify[channel, recipient, template]`, `Emit[event]`, `Call[service, op]`,
`Webhook[url, payload]`

**Рабочие процессы и состояние**
`Approve[subject, actor]`, `Escalate[subject, deadline]`, `Schedule[at, contract]`,
`Remind[at, recipient]`

**Наблюдаемость**
`Audit[entity, action, actor]`, `Trace[node]`, `Metric[name, value]`

### § 5.2 Вывод контракта

Когда пользователь пишет:

```
compose :result = CreditCheck { customer: customer, amount: order.total }
```

и `CreditCheck` не существует в области видимости, система должна:

1. Найти в библиотеке контракты с подходящей сигнатурой входов
2. Если найдено: предложить каноническое имя с дифом
3. Если не найдено: запустить синтезатор на основе модели свойств, используя
   объявленные типы входов/выходов как спецификацию цели
4. Если синтез успешен: предложить сгенерированный контракт на ревью

Это **синтез контракта по типу** — компилятор выступает парным программистом,
заполняя отсутствующие библиотечные контракты по требованию.

### § 5.3 Прогрессивное раскрытие сложности

Три уровня поддерживают разные режимы разработки без конфликтов:

| Тип пользователя | Режим разработки | Используемые контракты |
|-----------------|-----------------|------------------------|
| Доменный эксперт | Объявлять системы и процессы | Только СИСТЕМА + БИБЛИОТЕКА |
| Разработчик приложений | Связывать доменные контракты | БИБЛИОТЕКА + редко АКСИОМЫ |
| Платформенный инженер | Строить библиотеку | АКСИОМЫ + ПЛАТФОРМА |

Каждый уровень полностью читаем самостоятельно. Доменный эксперт, читающий
объявление `system`, понимает бизнес-логику, не зная реализации `ManagerApproval`
или `FraudCheck`.

## § 6. Формальные свойства

### § 6.1 Композициональность

Поскольку всё — контракт, все законы композиции выполняются единообразно:

```
-- Ассоциативность
(A >> B) >> C  =  A >> (B >> C)

-- Тождество
id[τ] >> A  =  A  =  A >> id[τ]

-- Коммутативность параллели (с точностью до переупорядочения выходов)
A ⊗ B  ≅  B ⊗ A
```

Эти законы — не только теория. Они управляют оптимизациями компилятора
(переупорядочение, слияние, параллелизация) без семантических изменений.

### § 6.2 Монотонность уточнения

Добавление `policy` к `system` может только **усилить** ограничения системы,
но не ослабить их. Это свойство монотонности модели CCP:

```
system S  ⊑  system S + policy P
```

Компилятор проверяет это статически: политика — это инъекция `guard`, а узлы
guard только сужают пространство достижимых выполнений.

### § 6.3 Сохранение разрешимости

Трёхуровневая архитектура сохраняет результаты разрешимости из
[igniter-lang-theory2.md](igniter-lang-theory2.md):

- Развёртки СИСТЕМЫ и БИБЛИОТЕКИ производят контракты в роговом фрагменте → PTIME
- Жизненный цикл `entity` разворачивается в стратифицированный `branch`/`guard` → стратифицированный Datalog
- `workflow` разворачивается в `compose` + `await` → разрешимо при конечном алфавите событий
- Инъекция `policy` монотонна → не вводит новых неподвижных точек

Энтерпрайз-поверхность — синтаксический сахар над разрешимым ядром.

## § 7. Открытые исследовательские вопросы

1. **Минимальный словарь библиотеки** — какой наименьший набор библиотечных
   контрактов покрывает 80% реальных корпоративных кодовых баз?
   (Требуется эмпирическое исследование.)

2. **Полнота вывода контрактов** — при каких условиях синтезатор гарантированно
   найдёт контракт `CreditCheck`, зная только его сигнатуру типов?

3. **Композиция политик** — когда две политики конфликтуют
   (напр., `ApprovalRequired` и `AutoApproveBelow500`), как язык обнаруживает
   и разрешает противоречие во время компиляции?

4. **Жизненный цикл сущности и параллелизм** — два параллельных рабочих процесса
   пытаются перевести один и тот же `Order` в новое состояние; какая конструкция
   контракта моделирует оптимистическую vs. пессимистическую блокировку?

5. **Контракты границ системы** — как объявление `system` выражает внешний API,
   чтобы две системы могли безопасно компоноваться через границы ограниченных
   контекстов?

6. **Эволюция библиотеки** — когда сигнатура библиотечного контракта меняется,
   какие правила миграции сохраняют корректность всех зависящих от него системных
   контрактов?
