## § 0. Ключевой инсайт

Узел `guard` локален: он проверяет условие один раз, в ходе одного выполнения
контракта. **Инвариант** универсален: это свойство, которое должно выполняться
для **всех экземпляров** типа и **всех переходов** состояния сущности.

Если охранники — локальные контракты, инварианты — **глобальные контракты**,
и именно они превращают язык из типобезопасной среды выполнения в лёгкий
формальный верификатор.

```
-- Охранник: проверяется один раз, в данном выполнении контракта
guard :within_budget {
  total <= manager.approval_limit
}

-- Инвариант: должен выполняться для каждого Account, везде, всегда
invariant PositiveBalance : Account {
  balance >= 0
}
```

Разница — в **области квантификации**: охранник = экзистенциальный (это выполнение),
инвариант = универсальный (все экземпляры, всё время).

## § 1. Инварианты как контракты

### § 1.1 Контрактная форма инварианта

Инвариант над типом `T` с предикатом `P` — это контракт:

```
Contract({ x: T }) → { verified: T where P(x) }
```

Он принимает значение типа `T`, утверждает `P` и возвращает **уточнённый тип**
`T where P`. Тип выхода строже типа входа.

Конкретное раскрытие:

```
invariant PositiveBalance : Account {
  balance >= 0
}

-- Разворачивается в:
contract PositiveBalance {
  in  account: Account
  guard :check { account.balance >= 0 }
  out  verified: Account where balance >= 0 = account
}
```

Выход — не просто `Account`, а `Account where balance >= 0`. Инвариант работает
как **приведение `T` к `T where P`** — операция сужения типа с проверкой
во время выполнения.

### § 1.2 Синтаксис объявления инварианта

```
invariant_decl ::=
  'invariant' IDENT ':' type_name '{' {invariant_clause} '}'

invariant_clause ::=
  | expr                            -- предикат
  | IDENT ':' expr                  -- именованный предикат
  | 'when' expr '{' {expr} '}'     -- условный предикат
  | 'on' SYMBOL '{' {expr} '}'     -- предикат для конкретного состояния жизненного цикла
```

**Примеры:**

```
invariant PositiveBalance : Account {
  balance >= 0
}

invariant ValidOrder : Order {
  items.count > 0
  total >= 0
  total == sum(items |> map { .price * .qty })    -- согласованность полей
}

invariant ShippedOrder : Order {
  when status == :shipped {
    shipped_at != null
    tracking_number != null
  }
  on :cancelled {
    cancelled_at != null
    refund_issued == true
  }
}
```

## § 2. Алгебра инвариантов

### § 2.1 Операции

Инварианты — значения первого класса. Над ними определены следующие операции:

```
-- Конъюнкция (усиление): оба должны выполняться
inv_a ∧ inv_b

-- Дизъюнкция (ослабление): хотя бы один должен выполняться
inv_a ∨ inv_b

-- Импликация: если inv_a выполняется, выполняется и inv_b
inv_a → inv_b

-- Параметризация: семейство инвариантов
inv(param: type)

-- Условная активация
inv when condition

-- Область жизненного цикла: активен только в именованных состояниях
inv on [:state_a, :state_b]
```

### § 2.2 Решётка инвариантов

Инварианты над типом `T` образуют **ограниченную решётку**:

```
          True  (нет ограничений — любое значение проходит)
            │
         inv_a
        /       \
    inv_a ∨ inv_b  ...
        \       /
         inv_b
            │
    inv_a ∧ inv_b  (оба должны выполняться — более ограничительно)
            │
         False  (ни одно значение не может удовлетворить — мёртвый тип)
```

Частичный порядок: `inv_a ≤ inv_b` тогда и только тогда, когда предикат `a`
влечёт предикат `b`. Более сильный инвариант = ниже в решётке инвариантов =
выше специфичность.

Решётка — это **связь Галуа** между множеством значений, удовлетворяющих инварианту,
и его предикатом: добавление ограничений монотонно сужает удовлетворяющее множество.

### § 2.3 Арифметика инвариантов

```
-- Ослабление: убрать конъюнкт (больше значений проходят)
ValidOrder ∧ InStock  →  ValidOrder

-- Усиление: добавить конъюнкт (меньше значений проходят)
ValidOrder  →  ValidOrder ∧ InStock

-- Композиция: проверить оба инварианта последовательно
PositiveBalance >> ValidLimit    -- эквивалентно PositiveBalance ∧ ValidLimit

-- Параметрическое семейство
invariant WithinLimit(limit: Money) : Order {
  total <= limit
}

-- Специализация: зафиксировать параметр
WithinLimit(10_000)   -- конкретный инвариант
```

## § 3. Инварианты — это уточнённые типы

### § 3.1 Формальное тождество

Объявление инварианта — синтаксический сахар для **псевдонима уточнённого типа**:

```
invariant P on T  ≡  type T_P = T where P
```

Операции над инвариантами — это операции над решёткой уточнённых типов:

| Операция над инвариантами | Операция над уточнёнными типами | Направление подтипа |
|---------------------------|--------------------------------|---------------------|
| `inv_a ∧ inv_b` | `{ x: T \| P_a(x) ∧ P_b(x) }` | подтип обоих |
| `inv_a ∨ inv_b` | `{ x: T \| P_a(x) ∨ P_b(x) }` | надтип обоих |
| Ослабление | Убрать конъюнкт из предиката | `T_PQ ⊑ T_P` |
| Усиление | Добавить конъюнкт | `T_PQ ⊑ T_P` (PQ уже подтип P) |

Правило подтипизации:

```
P(x) ∧ Q(x) → P(x)   для всех x
─────────────────────────────────────────────────
{ x: T | P(x) ∧ Q(x) } ⊑ { x: T | P(x) }
```

Это правило **ослабления уточнения** из § 1.4 спецификации языка — теперь показано
как алгебраическая операция ослабления в решётке инвариантов.

### § 3.2 Связь с Liquid Types

Данная модель формально эквивалентна **Liquid Types** (Rondon, Kawaguchi, Jhala 2008):
уточнённые типы, где предикаты берутся из ограниченного логического фрагмента
(квалификаторов), допускающего разрешимую проверку типов через SMT-решатель.

Ключевая граница: предикаты роговского фрагмента, используемые в модели контрактов
(линейная арифметика, структурный доступ к полям, без универсальной квантификации),
это в точности те квалификаторы, с которыми вывод типов Liquid Types работает
за полиномиальное время.

Это означает: **проверка инвариантов в контрактной модели разрешима за PTIME**
для роговского фрагмента — для общего случая SMT-решатель не нужен; он требуется
только для пользовательских числовых свойств на границе.

## § 4. Логика Хоара и распространение инвариантов

### § 4.1 Инварианты как пред/постусловия

Тройки Хоара `{P} C {Q}` (предусловие P, команда C, постусловие Q) напрямую
отображаются на узлы контракта:

```
-- Предусловие: что должно выполняться на входе
guard :pre { account.balance >= 0 }     -- или: объявленный инвариант на типе входа

-- Команда: вычисление
compute :new_balance = account.balance - withdrawal

-- Постусловие: что должно выполняться на выходе
out result: Account where balance >= 0  -- или: объявленный инвариант на типе выхода
```

Компилятор контрактов проверяет тройку Хоара: данны входной инвариант и вычисление,
выполняет ли тип выхода объявленный инвариант?

### § 4.2 Последовательная композиция и слабейшие предусловия

Для последовательной композиции `A >> B`:

```
wp(B, Q) = слабейшее предусловие на вход B, при котором Q выполняется на выходе B
wp(A >> B, Q) = wp(A, wp(B, Q))
```

В терминах контрактов:
- `A` производит выходной тип `τ_A`
- `B` требует входной тип `τ_B`
- Композиция безопасна тогда и только тогда, когда `τ_A ⊑ τ_B`

Компилятор проверяет: влечёт ли выходной инвариант `A` входной инвариант `B`?

```
A производит:  Account where balance >= 0
B требует:     Account where balance >= 0 ∧ balance <= credit_limit

-- НЕБЕЗОПАСНО: гарантия A слабее требования B
-- Ошибка компилятора: "разрыв инвариантов — A гарантирует PositiveBalance,
--                       B требует PositiveBalance ∧ WithinLimit"
```

### § 4.3 Сохранение инвариантов в рабочих процессах

```
workflow Fulfillment {
  steps:       [ValidateOrder, ReserveInventory, ChargePayment, Ship]
  preserves:   [ValidOrder, PositiveInventory]
  establishes: [PaymentReceived, TrackingAssigned]
}
```

Объявление `preserves` — это **доказательное обязательство**: компилятор проверяет,
что для каждого шага `S` сохранение инвариантов до `S` гарантирует их сохранение
после `S`.

Объявление `establishes` — это **постусловие**: при завершении рабочего процесса
эти дополнительные инварианты выполняются.

Алгоритм верификации (PTIME для роговского фрагмента):
1. Предположить, что инварианты `preserves` выполняются на входе рабочего процесса
2. Для каждого шага вычислить распространённое через wp множество инвариантов
3. На выходе каждого шага проверить, что сохраняемые инварианты выполняются
4. На выходе рабочего процесса проверить инварианты `establishes`
5. Выдать диагностику с первым шагом, нарушившим инвариант

## § 5. Энтерпрайз-паттерны инвариантов

### § 5.1 Статические инварианты

Свойства, которые должны выполняться для каждого значения типа, всегда:

```
invariant ValidProduct : Product {
  price > 0
  stock >= 0
  name.length > 0
}

invariant ValidCustomer : Customer {
  email.contains("@")
  credit_limit >= 0
  status in [:active, :suspended, :closed]
}
```

### § 5.2 Инварианты жизненного цикла

Свойства, которые должны выполняться в конкретных состояниях сущности:

```
invariant OrderLifecycle : Order {
  on :processing {
    reserved_at != null
    inventory_reserved == true
  }
  on :complete {
    payment_received == true
    shipped_at != null
    tracking_number != null
  }
  on :cancelled {
    cancelled_at != null
    cancellation_reason != null
    when total > 0 { refund_issued == true }
  }
}
```

### § 5.3 Межсущностные инварианты (уровень системы)

Свойства, охватывающие несколько сущностей — наиболее мощный вид:

```
invariant OrderConsistency {
  order.total == sum(order.items |> map { .price * .qty })
  order.customer.credit_used >= order.total when order.payment_method == :credit
}

invariant InventoryConsistency {
  product.stock == product.initial_stock
                 - sum(fulfilled_orders |> map { .qty })
                 + sum(received_shipments |> map { .qty })
}
```

Межсущностные инварианты нельзя проверить локально — для них нужна область
объявления `system`:

```
system OrderManagement {
  invariants: [OrderConsistency, InventoryConsistency]
  -- компилятор проверяет их на каждой границе контракта в системе
}
```

### § 5.4 Безопасность и живость (классические формальные свойства)

Модель инвариантов напрямую связана с классическими свойствами формальной
верификации:

| Формальное свойство | Форма контрактного инварианта |
|--------------------|-----------------------------|
| **Безопасность** (`□ ¬bad`) | `invariant NeverBad : T { not bad_condition }` |
| **Живость** (`◇ good`) | `workflow W { establishes: GoodCondition }` |
| **Сохранение** | `workflow W { preserves: [inv_a, inv_b] }` |
| **Прогресс** | `entity E { lifecycle: must_reach [:complete, :cancelled] }` |

## § 6. Компилятор как верификатор

### § 6.1 Что компилятор может проверить статически

Инварианты как первоклассные объекты + алгебра + распространение через логику Хоара
дают компилятору следующие возможности:

| Проверка | Техника | Сложность |
|----------|---------|-----------|
| Согласованность инварианта | Выполнимость предиката (нет типа `False`) | PTIME (Horn) |
| Сохранение инварианта в compose | `τ_out ⊑ τ_in` на каждом ребре `compose` | PTIME |
| Сохранение в рабочем процессе | wp-распространение по цепи шагов | PTIME |
| Согласование охранника с инвариантом | Предикат охранника ⊆ требуемый инвариант | PTIME |
| Межсущностная согласованность | Нижняя точка Datalog по графу сущностей | PTIME |
| Полнота жизненного цикла | Покрытие ветвями состояний жизненного цикла | PTIME |

Все проверки выполняются за PTIME для роговского фрагмента. Это **формальный
верификатор, а не просто проверщик типов** — он доказывает семантические свойства,
а не только синтаксические. И работает достаточно быстро для запуска при каждой
компиляции.

### § 6.2 Гарантия PTIME

Из [igniter-lang-theory2.md](igniter-lang-theory2.md), Теорема 2.1: синтез
в роговском фрагменте выполняется за PTIME. Проверка инвариантов — это синтез
в обратную сторону: дан контракт и инвариант, проверить, сохраняет ли контракт
инвариант.

Формально: проверка сохранения инвариантов сводится к вычислению нижней точки
Datalog + лемма Фаркаша для линейной арифметики (SMT не нужен для линейных
предикатов). Совокупная процедура — PTIME.

Это означает: компилятор может **доказывать** свойства уровня предприятия
за то же время, что требуется на проверку типов — без интерактивного доказательства
теорем от пользователя.

### § 6.3 Градиент верификации

Система поддерживает градиент от «лёгких проверок» до «полной верификации»:

```
Уровень 0  -- инварианты не объявлены: только ошибки во время выполнения (текущий Igniter)
Уровень 1  -- инварианты объявлены, проверяются во время выполнения: ловим нарушения
Уровень 2  -- инварианты + согласование охранников: статическая проверка в точках compose
Уровень 3  -- инварианты + preserves в рабочих процессах: PTIME формальная верификация
Уровень 4  -- межсущностные инварианты + система: верификация системы целиком
```

Пользователи подключают нужный уровень. Энтерпрайз-команды могут внедрять постепенно.

## § 7. На пути к POC

### § 7.1 Итерация 1: усиление модели

Исследовательские задачи перед реализацией:
1. **Глубина квантора инварианта** — какие предикатные фрагменты остаются в PTIME?
   (Линейная арифметика: да. Полиномиальная: пограничная. Универсальная квантификация: нет.)
2. **Область межсущностных инвариантов** — как объявление `system` ограничивает
   домен верификации?
3. **Наследование инвариантов** — при `entity Child : Parent` каковы правила
   наследования инвариантов?
4. **Параметрические инварианты** — `inv(param)` — как параметризация взаимодействует
   со структурой решётки?
5. **Приближённые инварианты** — `~inv` — инвариант над `~T` (вероятностное
   выполнение, граница достоверности)?

### § 7.2 Итерация 2: проектирование POC

Минимальная реализуемая поверхность:
```
-- Добавления в DSL:
invariant NAME : TYPE { ... }         -- объявление
workflow W { preserves: [...] }        -- обязательство сохранения
compose :x = C { ... }                -- проверяется на ребре compose (τ_out ⊑ τ_in)
system S { invariants: [...] }         -- область видимости системы

-- Добавления в компилятор:
- Реестр инвариантов (имя → предикат)
- Представление уточнённого типа (T where P)
- SubtypeChecker — проверка τ_out ⊑ τ_in на каждом ребре compose
- WorkflowVerifier — wp-распространение для шагов с preserves
- Расширение диагностического отчёта — отчёты о нарушении инвариантов с трассой доказательства
```

### § 7.3 Итерация 3: реализация POC (Ruby)

Цель: расширить существующий компилятор Igniter:
1. Ключевое слово DSL `invariant` → создаёт именованный `InvariantNode` в модели
2. Представление уточнённого типа в `CompiledGraph`
3. `SubtypeChecker` — проверка `τ_out ⊑ τ_in` на каждом ребре `compose`
4. `WorkflowVerifier` — wp-распространение для обязательств `preserves`
5. Расширение диагностического отчёта — отчёты о нарушении инвариантов с трассой доказательства

Цель дымового теста: пример `FulfillOrder`/`ProcessOrder` с явными инвариантами
на `Order`, `Account` и `Product` — верифицированный компилятором без ошибок
времени выполнения.

## § 8. Открытые вопросы

1. **Отрицательные инварианты** — может ли инвариант утверждать *отсутствие*
   свойства (`account.frozen == false`)? Как это взаимодействует с монотонностью
   модели CCP?

2. **Темпоральные инварианты** — `inv always` vs. `inv eventually` — нужен ли
   контрактной модели слой линейной темпоральной логики, или достаточно инвариантов
   с областью жизненного цикла?

3. **Обнаружение конфликтов инвариантов** — если два инварианта в одной системе
   взаимно несовместимы (`balance > 0` и `balance < 0`), может ли компилятор
   обнаружить это без перебора?

4. **Вероятностные инварианты** — `~inv` с достоверностью `0.95` — для узлов `~T`,
   где точная проверка инвариантов слишком дорога, можно ли распространять
   приближённое выполнение инварианта?

5. **Синтез, направляемый инвариантами** — дан набор инвариантов и целевой предикат,
   может ли синтезатор на основе модели свойств сгенерировать контракт, доказуемо
   достигающий цели при сохранении всех инвариантов?
