# Igniter в Науке, Робототехнике, Космосе и Медицине

*Отчёт о валидации — критические и научные домены.*
*Дата: 2026-04-27.*

---

## §0 Гипотеза и Методология Валидации

**Гипотеза**: Igniter — не только корпоративный фреймворк. Его базовые примитивы —
валидируемые графы зависимостей, `History[T]`, инварианты времени компиляции,
распределённые контракты, временны́е правила и OLAP Point — это именно те примитивы,
которых не хватало стекам программного обеспечения в науке, робототехнике, космосе
и медицине. Корпоративный use case — лишь один из экземпляров более общего паттерна.

**Метод валидации**: для каждого домена — определить глубинную боль, напрямую
сопоставить её с конкретным примитивом Igniter, набросать DSL, который устранит боль,
выявить то, что Igniter может делать здесь лучше любого другого инструмента,
и собрать инсайты для развития платформы.

**Сигнал анти-валидации** — случаи, где Igniter добавляет накладные расходы без
пользы, или где существующие доменные инструменты (Snakemake, ROS, MATLAB/Simulink,
Epic) просто лучше. Честность здесь важнее продвижения.

## §1 Наука — Воспроизводимость и Целостность Пайплайнов

### Боль

Кризис воспроизводимости в науке — не культурная проблема, а проблема архитектуры
программного обеспечения. Реальные первопричины:

1. **Параметры предобработки изменяются незаметно.** Анализ 2019 года невозможно
   воспроизвести, потому что функция нормализации была обновлена в общей библиотеке.
   Никто не знает, какая версия использовалась.

2. **Пайплайны — это ad-hoc DAG-и.** В каждой лаборатории свои Python-скрипты,
   объединённые shell-вызовами. Между стадиями нет проверки типов. Переименование
   колонки ломает пайплайн в рантайме, а не при определении.

3. **Контроль качества данных — поздний или отсутствующий.** Контрольные точки
   качества существуют в комментариях ("TODO: validate that pval < 0.05") или в
   постфактумных скриптах — не в определении пайплайна.

4. **Многомерный анализ непрозрачен.** Анализ по условиям × время × пациент × ген
   требует индивидуальных трюков с pandas. Размерная структура невидима в коде.

### Соответствие Igniter

| Боль | Примитив Igniter | Механизм |
|------|-----------------|----------|
| Незаметные изменения параметров | `BiHistory[T]` + `as_of` | Каждый вход версионирован; `as_of: original_date` воспроизводит точное состояние |
| Поломки пайплайна в рантайме | Типизированный граф контракта | Несоответствие типов обнаруживается при компиляции, а не на шаге N 12-часовой задачи |
| Поздние контрольные точки качества | `invariant` как первоклассный узел | Ограничения качества вшиты в определение пайплайна, проверяются до выполнения |
| Непрозрачный многомерный анализ | `OLAPPoint[T, dims]` | Измерения декларированы; `slice`/`rollup`/`drill` — нативные операции |

### Набросок DSL — Пайплайн Анализа Сворачивания Белка

```ruby
store :raw_sequences,     BiHistory[Sequence],   backend: :log
store :model_registry,    BiHistory[ModelState], backend: :log
store :analysis_results,  History[FoldResult],   backend: :timeseries

contract :protein_fold_analysis do
  input :sequence_id, String
  input :as_of, DateTime, default: -> { Time.now }

  lookup :sequence,      from: :raw_sequences,  key: :sequence_id, as_of: :as_of
  lookup :model,         from: :model_registry, key: "alphafold-v2", as_of: :as_of

  compute :preprocessed, with: [:sequence],             call: Preprocessing
  compute :prediction,   with: [:preprocessed, :model], call: FoldPredictor
  compute :quality,      with: [:prediction],            call: QualityScorer

  # Контрольные точки качества вшиты в определение — не в отдельный скрипт
  invariant "quality.score >= 0.70",
    on: :quality, message: "Качество сворачивания ниже допустимого порога"
  invariant "prediction.confidence_intervals.all? { |ci| ci.width < 0.1 }",
    on: :prediction

  output :fold,    from: :prediction
  output :quality, from: :quality
end

# Точное воспроизведение запуска 2023 года — та же модель, те же параметры, те же данные
result = ProteinFoldAnalysis.call(
  sequence_id: "P12345",
  as_of: DateTime.new(2023, 1, 15, 14, 0, 0)
)

# "Какой был бы результат сегодня с прошлогодней моделью?"
counterfactual = ProteinFoldAnalysis.call(
  sequence_id: "P12345",
  as_of: DateTime.now,
  model: model_registry[:as_of Date.new(2023, 1, 1)]
)
```

### OLAPPoint для Мультиомного Анализа

```ruby
olap_point :ExpressionLevel, Float,
  dimensions: { gene: GeneId, sample: SampleId, condition: Condition, time: DateTime }

store :expression_data, OLAPPoint[Float, { gene: GeneId, sample: SampleId,
                                            condition: Condition, time: DateTime }],
  backend: :columnar,
  source:  :sequencing_pipeline,
  materialization: :incremental

contract :differential_expression do
  input :gene_set,    Array[GeneId]
  input :condition_a, Condition
  input :condition_b, Condition
  input :time_point,  DateTime

  olap :expr_a, ExpressionLevel,
    slice: { condition: :condition_a, time: :time_point },
    rollup: :mean, partition: :by_gene

  olap :expr_b, ExpressionLevel,
    slice: { condition: :condition_b, time: :time_point },
    rollup: :mean, partition: :by_gene

  compute :log_fold_change, with: [:expr_a, :expr_b], call: LogFoldCalc

  invariant "log_fold_change.finite?", on: :log_fold_change

  output :lfc, from: :log_fold_change
end
```

### Уникальные Преимущества в Науке

**Что не делает ни один существующий инструмент:**

- **Snakemake / Nextflow**: декларируют топологию пайплайна, но не имеют системы
  типов, инвариантов, временного версионирования. Воспроизведение через `as_of`
  невозможно — нужно вручную управлять файлами окружения.
- **DVC / MLflow**: отслеживают артефакты, но не вычислительную структуру.
  Инварианты отсутствуют. Контрольные точки качества — постфактумные скрипты.
- **Igniter**: топология пайплайна IS типизированная программа. `as_of` даёт
  воспроизводимость бесплатно. Инварианты — это контрольные точки качества.
  `BiHistory[T]` — полный аудит-трейл, удовлетворяющий большинству требований
  воспроизводимости научных журналов.

**Главное преимущество**: `as_of` на входах `BiHistory[T]` превращает воспроизводимость
из процессной дисциплины в свойство языка. Вы не можете "забыть" зафиксировать,
какая версия использовалась — контракт захватывает это по конструкции.

### Собранные Инсайты для Платформы

- **Типы физических единиц**: наука постоянно работает с `Kelvin`, `Meter`,
  `Pascal`, `Mole`. `Float` с меткой "температура" может быть °C или K. Неправильные
  единицы стали причиной потери Mars Climate Orbiter. Типы единиц — это уточняющие
  типы: `Kelvin = Float where value >= 0.0`. Система инвариантов уже поддерживает
  это — но встроенная алгебра единиц сделала бы их первоклассными.

- **Неопределённость как тип**: научные измерения всегда несут неопределённость.
  `~Float` (вероятностный тип) из документа по предвычислению нуждается в физическом
  обосновании: `~Float = { value: Float, uncertainty: Float }`. Неопределённость
  должна распространяться через граф контракта по правилам стандартного распространения
  ошибок. Тип `~T` — правильная модель, нужна лишь физическая привязка.

## §2 Робототехника — Критически Важные Контуры Управления

### Боль

Программное обеспечение для роботов — сложнейшее для понимания, поскольку работает
в физическом мире с ограничениями реального времени и катастрофическими режимами
отказа:

1. **Ограничения безопасности рассыпаны по кодовой базе.** "Скорость не должна
   превышать MAX_VELOCITY" встречается в трёх разных файлах, удаляется при
   рефакторинге — и робот врезается в человека.

2. **Поведенческие режимы — неявные конечные автоматы.** Аварийная остановка,
   деградированный режим, нормальная работа — каждый режим имеет разные правила.
   Реализация через вложенные цепочки `if/elsif/else` делает режимы невидимыми и
   непроверяемыми.

3. **Связность сенсор-актуатор.** Контуры управления смешивают измерение, вычисление
   и воздействие в монолитные классы. Тестирование любой части требует мокирования
   всего остального. Поток данных нечитаем.

4. **Нет формальной связи между спецификацией и кодом.** Документ требований
   безопасности говорит "Робот не должен двигаться быстрее 0,5 м/с в картографированной
   среде". В коде есть `MAX_VELOCITY = 0.5` где-то. Являются ли они одним и тем же
   ограничением — неизвестно.

### Соответствие Igniter

| Боль | Примитив Igniter | Механизм |
|------|-----------------|----------|
| Рассыпанные ограничения безопасности | `invariant` как первоклассный | Единый источник истины; компилятор проверяет размещение |
| Неявные конечные автоматы | Временна́я система `rule` | Поведенческие режимы как именованные правила с предикатами `applies:` |
| Связность сенсор-актуатор | Топология графа контракта | Сенсор → вычисление → актуатор декларированы как типизированный DAG |
| Разрыв спецификация-код | Инварианты с `label:` | Имя инварианта = пункт спецификации; компилятор обеспечивает соответствие |

### Набросок DSL — Автономная Навигация

```ruby
contract :navigation_step do
  input :sensor_fusion,  SensorState
  input :mission_plan,   MissionPlan
  input :robot_state,    RobotState

  compute :obstacle_map,     with: [:sensor_fusion],                             call: ObstacleMapper
  compute :path_candidate,   with: [:obstacle_map, :mission_plan, :robot_state], call: PathPlanner
  compute :velocity_command, with: [:path_candidate, :robot_state],              call: VelocityController

  # Инварианты безопасности — это и ЕСТЬ документ требований безопасности в исполняемой форме
  invariant "sensor_fusion.staleness < 100.milliseconds",
    on: :sensor_fusion, label: "REQ-SENSE-01"
  invariant "velocity_command.linear.abs <= MAX_LINEAR_VEL",
    on: :velocity_command, label: "REQ-SAFE-01"
  invariant "velocity_command.angular.abs <= MAX_ANGULAR_VEL",
    on: :velocity_command, label: "REQ-SAFE-02"
  invariant "path_candidate.clearance >= SAFETY_MARGIN",
    on: :path_candidate, label: "REQ-SAFE-03"

  output :command, from: :velocity_command
end
```

Поведенческие режимы как временны́е правила — без цепочек `if/elsif`:

```ruby
# Аварийная остановка — наивысший приоритет, перекрывает всё
rule :emergency_stop do
  applies_to :velocity_command
  applies:   -> { sensor_fusion.collision_probability > 0.8 }
  compute:   -> { VelocityCommand.zero }
  priority:  1000
end

# Режим экономии батареи
rule :low_battery_crawl do
  applies_to :velocity_command
  applies:   -> { robot_state.battery_level < 0.15 }
  compute:   -> (cmd) { cmd.scale(0.3) }
  priority:  10
end

# Режим деградированных сенсоров — медленнее и осторожнее
rule :degraded_sensing do
  applies_to :path_candidate
  applies:   -> { sensor_fusion.active_sensors < MINIMUM_SENSORS }
  compute:   -> { PathPlanner.conservative_mode(obstacle_map, mission_plan, robot_state) }
  priority:  50
end
```

### Набросок DSL — Координация Роёв Роботов

```ruby
contract :swarm_coordination do
  input :swarm_id,    String
  input :robot_count, Integer

  await :all_ready, event: :robot_ready, count: :robot_count,
        correlate_by: :swarm_id

  compute :formation,       with: [:all_ready],            call: FormationPlanner
  compute :task_assignment, with: [:formation, :all_ready], call: TaskAssigner

  invariant "task_assignment.all? { |r| r.assigned? }", on: :task_assignment
  invariant "formation.no_collisions?", on: :formation

  effect :broadcast_plan, depends_on: :task_assignment,
           call: SwarmBroadcast, idempotent: true

  output :plan, from: :task_assignment
end
```

### Уникальные Преимущества в Робототехнике

**Что не делает ни один существующий инструмент:**

- **ROS 2**: публикация-подписка без типизированных пайплайнов и инвариантов.
  Безопасность — полностью ответственность программиста.
- **Simulink**: блок-диаграммы с ограничениями, но генерируемый код непрозрачен,
  непроверяем и привязан к MATLAB.
- **Igniter**: граф контракта — это архитектура управления. Инварианты с `label:`
  трассируемы до документа требований безопасности. Временны́е правила заменяют
  неявные конечные автоматы. Тот же фреймворк работает и на роботе, и в тест-сьютах
  — мокирование не требуется.

**Главное преимущество**: инварианты с аннотациями `label:` создают живой,
тестируемый, аудируемый мост между опубликованными требованиями и работающим кодом.
Разрыв между "что говорит спецификация" и "что делает ПО" — причина аварий —
становится ошибкой компилятора.

### Собранные Инсайты для Платформы

- **Контракты с дедлайном реального времени**: в робототехнике "этот контур
  управления выполняется за < 10 мс" — не цель производительности, а требование
  корректности. Языку нужны контракты с дедлайном: `contract :navigation_step, deadline: 10.milliseconds`.
  Структура DAG делает анализ WCET (наихудшее время выполнения) более решаемым,
  чем для произвольных программ — критический путь вычислим статически.

- **Метки инвариантов как артефакты сертификации**: `label: "REQ-SAFE-01"`
  связывает инвариант с идентификатором управления требованиями. С базой требований
  компилятор может проверить 100% покрытие требований. Это именно то, что требуют
  DO-178C (авиация), ISO 26262 (автомобили) и IEC 62443 (промышленность).

## §3 Космос — Телеметрия, Миссионные Контракты и Наземные Системы

### Боль

Космические системы имеют одни из высочайших требований к качеству ПО — и одни
из наиболее стойких проблем:

1. **Коррекция телеметрии после обновлений калибровки.** Калибровка датчика
   на космическом аппарате обновляется. Шесть месяцев исторической телеметрии
   нужно переинтерпретировать. Исходные "неправильные" значения должны сохраняться
   для анализа миссии. Стандартные базы данных временных рядов либо перезаписывают,
   либо требуют нестандартных схем коррекций.

2. **Корреляция событий земля-аппарат.** Команды, отправленные с Земли, летят
   20 минут до Марса. Ответы прибывают через 20 минут. Корреляция команд с их
   последствиями требует причинной бухгалтерии через 40-минутное окно
   неопределённости.

3. **Правила миссии закодированы в коде, а не в декларациях.** "Включить
   нагреватель, если температура ниже 250K И мы не в затмении" похоронено
   в C-подпрограмме. Обновление требует сборки ПО, загрузки и верификационного цикла.

4. **Разрыв операционного/аналитического.** Служебная телеметрия (операционная)
   питает отчёты об аномалиях, планирование миссии и инженерный анализ
   (аналитический). Разрыв управляется ручными экспортами и специализированными
   скриптами.

### Соответствие Igniter

| Боль | Примитив Igniter | Механизм |
|------|-----------------|----------|
| Коррекция телеметрии | `BiHistory[T]` | Коррекции добавляются; оригиналы сохраняются; запрос в любой точке (valid, transaction) |
| Причинная корреляция событий | Распределённые контракты + causal `as_of` | `await` коррелирует события; векторные часы отслеживают причинный порядок через световую задержку |
| Правила миссии в коде | Декларации temporal `rule` | Правила — это данные, не код; возможна загрузка без сборки ПО |
| Операционный/аналитический разрыв | `OLAPPoint source:` | Контракт материализации связывает поток телеметрии с OLAP-хранилищем |

### Набросок DSL — Тепловое Управление с BiHistory

```ruby
store :temperature_telemetry, BiHistory[Kelvin],
  backend:        :timeseries,
  write:          :single_writer,   # телеметрия поступает через одну наземную станцию
  consistency:    :causal,
  write_conflict: :last_wins        # более поздняя калибровка заменяет более раннюю

contract :thermal_control_decision do
  input :component_id, String
  input :command_time,  DateTime

  lookup :temperature, from: :temperature_telemetry,
    # Критично для расследования инцидентов:
    # "Какую температуру МЫ ДУМАЛИ видеть, когда отправляли команду?"
    as_of:           :command_time,
    knowledge_as_of: :command_time

  lookup :limits, from: :component_specifications, key: :component_id

  compute :margin, with: [:temperature, :limits], call: ThermalMarginCalc

  invariant "temperature <= limits.max_operating",
    on: :temperature, label: "THERM-01", message: "Компонент превысил тепловой лимит"
  invariant "margin >= 0.0",
    on: :margin, label: "THERM-02"

  branch :recommendation do
    on { margin < 0.05 } => :emergency_shutdown
    on { margin < 0.15 } => :reduce_power
    on { margin >= 0.15 } => :nominal
  end

  output :recommendation, from: :recommendation
  output :margin,         from: :margin
end
```

### Набросок DSL — Правила Миссии как Загружаемые Декларации

Координация земля-аппарат с причинным `as_of`:

```ruby
contract :command_sequence_execution do
  correlate_by :sequence_id

  input  :sequence_id, String
  input  :commands,    Array[UplinkedCommand]

  await  :execution_ack, event: :telemetry_confirms_execution,
           correlate_by: :sequence_id,
           timeout: 3.hours   # двойной световой переход до Марса + запас

  compute :outcome_analysis, with: [:commands, :execution_ack], call: OutcomeAnalyzer

  invariant "execution_ack.all_commands_confirmed?",
    on: :execution_ack, label: "CMD-ACK-01"

  output :outcome, from: :outcome_analysis
end
```

### Уникальные Преимущества в Космосе

**Что не делает ни один существующий инструмент:**

- **CCSDS / XTCE телеметрические схемы**: определяют форматы данных, но не имеют
  вычислительной модели, инвариантов, временно́го рассуждения.
- **MATLAB/SIMULINK для анализа миссий**: хорош для симуляции, ужасен для
  операционных систем. Нет аудит-трейла, нет типизированных пайплайнов.
- **Igniter**: `BiHistory[T]` — корректная модель для коррекций телеметрии.
  Причинный `as_of` с векторными часами решает проблему корреляции через световую
  задержку. Правила миссии как временны́е декларации загружаются без цикла сборки ПО.

**Главное преимущество**: `BiHistory[T]` с `knowledge_as_of` позволяет критический
запрос для расследования инцидентов — "Что мы знали и когда мы это знали?" Это
недостижимо со стандартными базами данных временных рядов и устраняет целые
категории сложности расследования миссий.

### Собранные Инсайты для Платформы

- **Сертифицированный экспорт из скомпилированного графа**: космические агентства
  (ESA, NASA) требуют формальных спецификаций для критически важного ПО.
  Скомпилированный граф контракта уже является формальным DAG с типизированными
  рёбрами и верифицированными инвариантами. Проход компилятора мог бы экспортировать
  его как AADL, SysML или TLA+ — сертификационный артефакт уже содержится в
  артефакте компиляции.

- **Загружаемые декларации правил**: правила миссии как временны́е декларации (не
  код) можно сериализовать, передать по радиоканалу и применить на приёмнике без
  обновления ПО. Это требует, чтобы система правил была данными — что структурно
  уже так. Платформе нужен формат сериализации правил и безопасный API инъекции правил.

## §4 Медицина — Клиническая Безопасность и Аудит-Трейлы

### Боль

Медицинское ПО убивает людей, когда в нём неверная логика, и создаёт правовую
ответственность, когда оно не может доказать, что логика верна:

1. **Проверка лекарственных взаимодействий — навесная, не структурная.** Базы
   данных взаимодействий запрашиваются при вводе назначения и забываются. Прошла
   ли проверка, какая версия использовалась и что она нашла — не является частью
   медицинской документации.

2. **Логика дозирования похоронена в коде, не трассируема к руководствам.** Клинические
   руководства говорят "снизить дозу на 50% при рСКФ < 30". В коде есть
   `if egfr < 30 then dose * 0.5`. Являются ли они одним ограничением — неизвестно.

3. **Коррекция медицинских записей требует полного аудит-трейла.** HIPAA и стандарты
   ЭМК требуют, чтобы каждая коррекция сохраняла оригинал с указанием кто исправил
   и когда. Стандартные базы данных поддерживают это через нестандартные таблицы
   аудита — хрупкие, непоследовательные, часто отсутствующие.

4. **Клинические протоколы меняются; выполняющиеся воркфлоу не должны ломаться.**
   Обновление протокола дозирования не должно влиять на текущие заказы. Заказы,
   начатые по протоколу v1, должны завершаться по протоколу v1.

### Соответствие Igniter

| Боль | Примитив Igniter | Механизм |
|------|-----------------|----------|
| Навесная проверка взаимодействий | Контракт как первоклассный этап | Проверка взаимодействий — типизированный узел; её результат — вывод контракта |
| Разрыв руководство-код | `invariant` с `label:` | Имя инварианта = ссылка на руководство; компилятор обеспечивает соответствие |
| Аудит-трейл коррекций | `BiHistory[T]` | Каждая коррекция добавляется; оригинал сохранён; запрос в любой точке valid/transaction |
| Версионирование протоколов | `as_of` на хранилищах правил | Выполняющийся заказ фиксирует версию правил в момент начала |

### Набросок DSL — Контракт Дозирования Лекарств

```ruby
store :patient_vitals,    BiHistory[VitalSigns],     backend: :timeseries,
                          consistency: :strong,       write_conflict: :error
store :active_medications, BiHistory[Medication],    backend: :timeseries,
                           consistency: :strong
store :dosing_protocols,   BiHistory[DosingProtocol], backend: :log

contract :medication_dosing do
  input :patient_id,       String
  input :medication_order, MedicationOrder
  input :as_of,            DateTime, default: -> { Time.now }

  lookup :patient,     from: :ehr,                key: :patient_id
  lookup :vitals,      from: :patient_vitals,     key: :patient_id, as_of: :as_of
  lookup :active_meds, from: :active_medications, key: :patient_id, as_of: :as_of
  lookup :protocol,    from: :dosing_protocols,   key: :medication_order.drug_id,
                       as_of: :as_of   # ← фиксирует версию протокола в момент заказа

  compute :renal_function,  with: [:patient, :vitals],              call: RenalFunctionCalc
  compute :interactions,    with: [:active_meds, :medication_order], call: DrugInteractionChecker
  compute :adjusted_dose,   with: [:medication_order, :renal_function, :protocol], call: DoseAdjuster

  # Каждый инвариант трассируем к ссылке на клиническое руководство
  invariant "interactions.none? { |i| i.severity == :contraindicated }",
    on: :interactions,
    label: "CG-INTERACTION-01",
    message: "Противопоказанная комбинация препаратов — заказ заблокирован"

  invariant "interactions.none? { |i| i.severity == :major && !i.acknowledged? }",
    on: :interactions,
    label: "CG-INTERACTION-02"

  invariant "adjusted_dose.amount.between?(protocol.min_dose, protocol.max_dose)",
    on: :adjusted_dose,
    label: "CG-DOSE-RANGE-01",
    message: "Доза вне терапевтического диапазона протокола"

  invariant "adjusted_dose.amount <= medication_order.drug.absolute_max_dose",
    on: :adjusted_dose,
    label: "CG-DOSE-MAX-01",
    message: "Доза превышает абсолютный максимум — потенциальная передозировка"

  output :approved_dose, from: :adjusted_dose
  output :warnings,      from: :interactions
  output :protocol_used, from: :protocol   # ← аудит-трейл: какая версия протокола
end
```

BiHistory для коррекций медицинских записей:

```ruby
store :lab_results, BiHistory[LabResult], backend: :timeseries, consistency: :strong

# Сценарий: лаборатория сообщает гемоглобин 8,2 г/дл. Позже исправляет на 11,4 (ошибка прибора).
# BiHistory записывает оба значения с полным аудит-трейлом.

# "Какой гемоглобин показывала система, когда мы принимали решение о переливании?"
hgb_at_decision = lab_results[:as_of transfusion_decision_time,
                                :knowledge_as_of transfusion_decision_time]
# → 8.2 (значение, которое было известно в момент принятия решения)

# "Каков корректный гемоглобин для этого забора крови?"
hgb_correct = lab_results[:as_of sample_time, :knowledge_as_of Time.now]
# → 11.4 (значение после коррекции)
```

### Уникальные Преимущества в Медицине

**Что не делает ни один существующий инструмент:**

- **Epic / Cerner**: платформы ЭМК имеют проверки дозирования, но они — чёрные
  ящики. Какое руководство применено, какая версия, каковы промежуточные вычисления
  — ничего из этого не инспектируемо и не тестируемо.
- **Поддержка клинических решений (CDS Hooks)**: событийно-ориентированные хуки,
  нет вычислительной модели, нет инвариантов, нет аудита вычислительного провенанса.
- **Igniter**: инварианты с `label:` — это клинические руководства в исполняемой
  форме. `BiHistory[T]` удовлетворяет требованиям аудита по конструкции. Привязка
  протокола через `as_of` означает, что обновления протоколов никогда не ломают
  текущие заказы. Артефакт компиляции — формальная спецификация для 510(k) FDA.

**Главное преимущество**: инварианты с метками ссылок на клинические руководства
создают живую, тестируемую, аудируемую связь между опубликованным руководством и
работающим кодом. Разрыв между "что говорят руководства" и "что делает ПО" —
причина нежелательных событий — становится ошибкой компилятора.

### Собранные Инсайты для Платформы

- **Уровни серьёзности нарушений инвариантов**: медицинские инварианты имеют
  разные последствия. Противопоказанная комбинация должна `raise` (жёсткая
  блокировка). Серьёзное взаимодействие должно `warn` и требовать подтверждения.
  Незначительное взаимодействие должно `log`. Системе инвариантов нужен параметр
  `severity:`: `invariant "...", severity: :error | :warn | :log`.

- **Рабочий процесс подтверждения инвариантов**: в медицине врач может
  переопределить предупреждение о взаимодействии с документированным обоснованием.
  Платформе нужно: `invariant "...", overridable_with: :documented_justification`.
  Переопределение становится частью аудит-трейла BiHistory.

## §5 Кросс-Доменный Синтез

### Паттерн Универсален

Все четыре домена разделяют один паттерн:

```
Данные физического/цифрового мира
  → Типизированные сенсорные/входные узлы
    → Типизированный граф вычислений (валидирован при компиляции)
      → Инварианты безопасности (проверены до выполнения)
        → Типизированный вывод / воздействие / рекомендация
```

Это именно граф контракта. Домены отличаются только:
- Задействованными типами (последовательности ДНК vs телеметрия vs витальные
  показатели vs сенсорное слияние)
- Требованиями к задержке (часы для науки vs миллисекунды для робототехники)
- Регуляторной базой (FDA vs ESA vs ISO vs рецензирование журналов)
- Последствиями отказа (потраченные вычисления vs смерть пациента vs потеря
  космического аппарата)

**Igniter — это вычислительная модель, не корпоративный инструмент.** Корпоративный
use case — домен, где наиболее серьёзное последствие отказа — финансовые потери.
Остальные домены имеют более серьёзные последствия и поэтому *больше* нуждаются
в гарантиях, которые предоставляет Igniter.

### Общие Требования к Примитивам

| Требование | Нужно всем четырём | Текущий Igniter | Пробел |
|-----------|-------------------|-----------------|--------|
| Типизированный граф зависимостей | Да | ✓ Полностью | Нет |
| Валидация при компиляции | Да | ✓ Полностью | Нет |
| `invariant` как первоклассный | Да | ✓ Полностью | Уровни серьёзности |
| `History[T]` / `BiHistory[T]` | Да | ✓ Специфицирован | Реализация |
| Временны́е правила | Да | ✓ Специфицирован | Реализация |
| Запрос `as_of` | Да | ✓ Специфицирован | Реализация |
| `OLAPPoint` многомерный | Да | ✓ Специфицирован | Реализация |
| Распределённые контракты | Да | ✓ Полностью | Нет |
| Типы физических единиц | Наука, Космос, Робот. | ✗ Отсутствует | Новый примитив |
| Распространение неопределённости | Наука, Робот. | ✗ Отсутствует | Расширить `~T` |
| Контракты с дедлайном | Робот., Космос | ✗ Отсутствует | Новый примитив |
| Сертифицированный экспорт | Космос, Медицина, Робот. | ✗ Отсутствует | Проход компилятора |
| Уровни серьёзности инвариантов | Медицина, Робот. | ✗ Отсутствует | Расширение DSL |

## §6 Инсайты для Развития Платформы

Инсайты, которые четыре домена коллективно вносят в дорожную карту развития
платформы. Упорядочены по стратегической важности.

### Инсайт 1 — Типы Физических Единиц (наивысшая ценность)

**Триггер домена**: Наука (Mars Orbiter), Космос (каждая аномалия из-за несоответствия
единиц), Медицина (ошибки мг vs мкг убивают пациентов), Робот. (углы в градусах vs радианах).

**Предложение**: типы единиц как уточняющие типы в языке.

```
type Kelvin = Float where value >= 0.0
type Meter = Float
type Newton = Float

# Алгебра: Meter * (1 / Second^2) → Acceleration
# Компилятор отклоняет: compute :force, with: [:mass_kg, :accel_m_s2] → Kelvin
```

**Стоимость разработки**: средняя. Требует алгебры типов в компиляторе.
**Ценность**: устраняет целые классы ошибок, ставших причиной реальных смертей и потерь миссий.

### Инсайт 2 — Уровни Серьёзности Инвариантов

**Триггер домена**: Медицина (предупреждение vs блокировка vs лог), Робот.
(жёсткая остановка vs информационный), Космос (критический vs служебный).

**Предложение**: параметр `severity:` для инвариантов.

```
invariant "interactions.none? { |i| i.severity == :contraindicated }",
  severity: :error    # вызывает InvariantViolation, блокирует выполнение

invariant "interactions.none? { |i| i.severity == :major && !i.acknowledged? }",
  severity: :warn,    # не вызывает исключение; логирует + устанавливает предупреждение
  overridable_with: :documented_justification

invariant "response_time < 100.milliseconds",
  severity: :log      # записывает метрику; не влияет на выполнение
```

**Стоимость разработки**: низкая. Существующей системе инвариантов нужны маршрутизация
серьёзности и механизм подтверждения.
**Ценность**: делает систему инвариантов пригодной во всех регулируемых доменах
без жёстких блокировок для предупреждений.

### Инсайт 3 — Контракты с Дедлайном (расширение реального времени)

**Триггер домена**: Робот. (контур управления 10 мс), Космос (окна обработки
телеметрии), Медицина (требования времени реакции на сигнализацию).

**Предложение**: параметр `deadline:` для контрактов с анализом WCET при компиляции.

```
contract :navigation_step, deadline: 10.milliseconds do
  # Компилятор вычисляет критический путь через DAG.
  # Если max(node_wcet для узлов на критическом пути) > deadline → ошибка компиляции.
  ...
end

compute :path_planning, with: [:obstacle_map, :mission_plan],
  call: PathPlanner,
  wcet: 3.milliseconds   # WCET декларирован разработчиком
```

**Стоимость разработки**: высокая. Требует инфраструктуры анализа WCET. Но структура
DAG делает это более решаемым, чем для произвольных программ.
**Ценность**: открывает Igniter для доменов жёсткого реального времени (авиация,
автомобили, промышленное управление).

### Инсайт 4 — Загружаемые Декларации Правил

**Триггер домена**: Космос (обновления правил по радиоканалу без сборки ПО),
Медицина (обновления протоколов без развёртывания ЭМК), Робот. (обновление
поведения без перепрошивки).

**Текущее состояние**: временны́е правила — это Ruby-процы, несериализуемые.

**Предложение**: подмножество деклараций правил, выразимых как данные (JSON/MessagePack):

```json
{
  "rule": "eclipse_heater",
  "applies_to": "heater_power_command",
  "applies": { "op": "and",
               "args": [
                 { "field": "spacecraft_state.in_eclipse", "eq": true },
                 { "field": "temperature.value", "lt": 250.0 }
               ]},
  "compute": { "type": "constant", "value": { "power_watts": 50 } },
  "priority": 20
}
```

**Стоимость разработки**: средняя. Требует языка выражений правил + безопасного
интерпретатора.
**Ценность**: открывает Igniter для операционных сред, где развёртывание кода
невозможно (космические аппараты) или строго регулируется (медицинские устройства).

### Инсайт 5 — Сертифицированный Экспорт из Скомпилированного Графа

**Триггер домена**: Космос (ESA/NASA формальная верификация), Медицина (FDA 510(k)),
Робот. (DO-178C, ISO 26262), Наука (требования воспроизводимости).

Скомпилированный граф контракта уже является формальным DAG с типизированными
рёбрами и верифицированными инвариантами — это формальная спецификация. Проход
компилятора мог бы экспортировать его как:

- **AADL** — авиационно-космический
- **SysML / UML** — общая системная инженерия
- **Coq / TLA+** — формальная верификация
- **SBOM** (Software Bill of Materials) — FDA / цепочка поставок

**Стоимость разработки**: средняя на формат.
**Ценность**: превращает артефакт компиляции Igniter в регуляторный сертификационный
артефакт. Значительный коммерческий дифференциатор в регулируемых отраслях.

### Инсайт 6 — Распространение Неопределённости через Граф Контракта

**Триггер домена**: Наука (неопределённость измерений), Робот. (шум сенсоров),
Космос (неопределённость определения орбиты), Медицина (точность лабораторных тестов).

**Предложение**: `~Float` как встроенный тип неопределённости с правилами распространения.

```
type ~Float = { value: Float, std_dev: Float }

# Арифметическое распространение по стандартным правилам:
# ~a + ~b → ~(a+b) с σ = sqrt(σa² + σb²)

compute :position_estimate, with: [:gps, :imu], call: SensorFusion
# → тип ~Position (неопределённая позиция с ковариацией)

invariant "position_estimate.value.distance_to(target) < 1.0",
  on: :position_estimate
# Компилятор предупреждает, если std_dev достаточно велика, чтобы инвариант
# мог не выполняться с высокой вероятностью
```

**Стоимость разработки**: средняя. Связывается с существующим типом `~T` из спецификации.

## §7 Вердикт

**Результат валидации: подтверждено с дифференциацией.**

Igniter подходит для научных и критических доменов не вопреки своему корпоративному
происхождению, а потому что корпоративная проблема и проблема в науке/робот./космосе/
медицине — это одна и та же проблема при разных уровнях последствий:

> Сложное вычисление со многими входами, типизированными зависимостями, инвариантами,
> которые должны соблюдаться, временны́м состоянием, которое должно быть корректным,
> и необходимостью знать, что именно произошло и когда.

Корпоративная версия этой проблемы приводит к неверным счетам. Медицинская версия
приводит к смерти пациентов. Те же примитивы решают обе; ставки разные.

**Критические домены раскрывают реальное ценностное предложение Igniter острее,
чем корпоративные use cases.** В корпорациях инварианты — приятное дополнение.
В медицине инварианты с метками ссылок на клинические руководства — это разница
между сертификацией и ответственностью. В космосе `BiHistory[T]` с `knowledge_as_of` —
это разница между анализом инцидента и расследованием потери миссии. Критические
домены — не граничные случаи, а доказательство того, что дизайн верен.

**Шесть инсайтов для платформы** (§6), упорядоченных по стратегической важности:
1. Типы физических единиц (устраняет ошибки единиц измерения, хорошо изученная модель)
2. Уровни серьёзности инвариантов (низкая стоимость, высокая ценность для регулируемых доменов)
3. Контракты с дедлайном (высокая стоимость, открывает домены жёсткого реального времени)
4. Загружаемые декларации правил (средняя стоимость, уникальный дифференциатор для космоса/IoT)
5. Сертифицированный экспорт из скомпилированного графа (средняя на формат, коммерческий дифференциатор)
6. Распространение неопределённости (средняя стоимость, обоснованное расширение `~T`)

Из них **типы физических единиц + уровни серьёзности инвариантов** имеют наивысшее
соотношение ценность/стоимость и должны рассматриваться для ближайшей спецификации.
